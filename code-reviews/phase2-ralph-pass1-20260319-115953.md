## CRITICAL

### 1) **Stored ÔÇťcanvas JSONÔÇŁ can silently inject dangerous hooks (command/http) and get persisted on mere selection**
**Where:** `HookCanvas.tsx` load path + debounced `notifyChange` + ReactFlow selection changes  
**Why itÔÇÖs dangerous:** `hooksCanvasJson` is treated as the source of truth (Priority 1) and is loaded *without* validation/sanitization of node types/data. After the first render, **any nodes/edges change** triggers `notifyChange()`, which recompiles hooks and calls `onChange()`ÔÇöpersisting new `hooksJson` and `hooksCanvasJson`.

ReactFlow commonly emits `onNodesChange` for **selection toggles** (changing `node.selected`) and other non-semantic UI state. That means:  
- Attacker supplies a profile/draft containing malicious `hooksCanvasJson` (with `command`/`http` actions).  
- User opens Canvas view and simply **clicks a node** (selection updates) Ôćĺ `nodes` state changes Ôćĺ `notifyChange()` runs Ôćĺ `compileCanvasToHooks()` outputs malicious hooks Ôćĺ `onChange()` persists them.  
- There is **no warning/confirmation** for command/http in this path (unlike Import).

**Impact:** stealthy persistence of command execution / HTTP exfiltration hooks into the agent config with minimal user interaction.

**Fix:**
- Do not compile/persist on purely presentational changes (selection/dragging). Strip ReactFlow UI state before compiling/saving (e.g., omit `selected`, `dragging`, `positionAbsolute`, etc.), or gate compilation on changes to a normalized graph model.
- On load: if `hooksCanvasJson` exists, either (a) validate it strictly and then **immediately** regenerate `hooksJson` and show a ÔÇťthis will change hooksÔÇŁ diff/confirm, or (b) treat `hooksJson` as canonical and only use `hooksCanvasJson` if it matches.
- Add the same command/http ÔÇťdangerous hooksÔÇŁ warning on *any* transition that would persist such hooks (not only file import).

## HIGH

### 2) **Unbounded `edges` in `hooksCanvasJson` Ôćĺ render/CPU DoS on load**
**Where:** `HookCanvas.tsx` saved canvas load  
```ts
if (saved.nodes ... length <= MAX_NODE_COUNT && saved.edges && Array.isArray(saved.edges)) {
  setNodes(saved.nodes);
  setEdges(saved.edges);
  return;
}
```
Only nodes are capped (`MAX_NODE_COUNT`), but **edges are not capped** and not shape-validated. A crafted `hooksCanvasJson` can include tens/hundreds of thousands of edges, causing:
- long JSON.parse time
- huge memory usage
- ReactFlow layout/render slowdown or crash

**Impact:** persistent denial-of-service when opening the Hooks tab (especially if drafts/profiles can be synced/shared).

**Fix:** enforce max edges (e.g. `<= 400` or `<= nodes*4`), validate edge schema (`id/source/target` strings, known node ids), and drop/repair invalid edges.

### 3) **Prototype-pollution gadget via untrusted event keys preserved into output**
**Where:** `canvasCompiler.ts` preserves unknown `event` strings as object keys in `result.hooks`  
Even though `result.hooks` uses `Object.create(null)` (good), the **output JSON** may contain keys like `"__proto__"`, `"constructor"`, `"prototype"` if a trigger event is attacker-controlled (via `hooksJson` Ôćĺ canvas generation, or `hooksCanvasJson`, or drag-drop injection).

Many downstream consumers (SDKs/backends) at some point do things like:
- `Object.assign({}, parsed.hooks)`
- deep-merge into defaults
- `{...parsed.hooks}`

Those patterns can re-activate `__proto__` semantics and create prototype pollution.

**Impact:** potential prototype pollution in the agent runtime / SDK consumer, which can escalate to logic bypasses and, in worst cases, RCE depending on how polluted properties are later used.

**Fix:** enforce an allowlist for events (preferably `HOOK_EVENTS`), or at minimum reject/escape dangerous keys (`__proto__`, `constructor`, `prototype`) and keys containing control characters.

### 4) **Drag-and-drop injection from arbitrary origins can create command/http hooks without any warning**
**Where:** `HookCanvas.tsx` `onDrop` reads `application/reactflow` and trusts it after minimal checks  
Any webpage/app can initiate a drag with `dataTransfer.setData('application/reactflow', '...')`. If the user drops it onto the canvas, attacker-controlled nodes are created.

Because the ÔÇťdangerous hooksÔÇŁ warning exists only in JSON file import, an attacker can social-engineer: ÔÇťdrag this widget into your canvas to installÔÇŽÔÇŁ which installs a `command` action or `http` action.

**Impact:** stealthy creation of dangerous hooks without the explicit import confirmation step.

**Fix:** require an internal origin marker (nonce) for palette drags; reject external drags. Also show the command/http confirmation whenever such nodes appear (regardless of source).

---

### 5) **Regex matcher is unvalidated Ôćĺ potential catastrophic backtracking / runtime DoS**
**Where:** `ConditionNode` / `NodeConfigPanel` stores `matcher` as ÔÇťregexÔÇŁ and compiler emits it as-is  
If the hook engine compiles this as a real regex (likely), attackers can provide pathological patterns (e.g., `(a+)+$`) causing severe CPU spikes on repeated tool invocations.

**Impact:** denial-of-service of the agent runtime (especially if hooks are shared).

**Fix:** length cap + safe-regex checks, or use a non-backtracking engine (RE2), or constrain matcher syntax (glob/contains instead of full regex).

## MEDIUM

### 6) **`generateCanvasFromHooks()` truncates *after* building the full graph Ôćĺ import-time DoS**
**Where:** `canvasGenerator.ts`  
It only truncates at the end:
```ts
if (nodes.length > MAX_GENERATED_NODES) { ... return truncated ... }
```
But it still iterates and allocates nodes/edges for the entire input first. A ~500KB hooks file can encode thousands of hook entries/groups and freeze the UI before truncation.

**Fix:** stop generating once limits are reached (early exit), and cap groups/hooks processed per event.

---

### 7) **No schema validation for `hooksJson` pasted/loaded (only ÔÇťis it JSONÔÇŁ)**
**Where:** `HooksTab.tsx` JSON view; `generateCanvasFromHooks()` loosely checks shape  
Arbitrary JSON can be persisted and later executed/parsed elsewhere. This is a classic ÔÇťconfig injectionÔÇŁ path: the UI provides a convenient persistence mechanism for attacker-crafted structures.

**Fix:** validate against a strict schema (zod/io-ts) before persisting, and provide user-facing errors.

### 8) **Node data sanitization is applied only to drag-drop, not to loaded canvas state or config updates**
**Where:**
- `sanitizeNodeData()` used only in `onDrop`
- `hooksCanvasJson` load uses raw `saved.nodes`
- `onUpdateNode` merges `newData` without caps
This allows oversized strings/values and unexpected nested objects via `hooksCanvasJson`, potentially destabilizing ReactFlow or producing huge compiled JSON.

**Fix:** apply the same sanitization/normalization on **all** ingress paths:
- hooksCanvasJson load
- hooksJson Ôćĺ generateCanvasFromHooks output
- onUpdateNode

### 9) **Timeout accepts non-finite numbers Ôćĺ silent corruption (`Infinity`/`NaN` stringifies to `null`)**
**Where:** `NodeConfigPanel.tsx` number parsing + `canvasCompiler.ts`  
`Number(e.target.value)` can yield `Infinity`/`NaN` (or programmatically). `JSON.stringify` converts `NaN`/`Infinity` to `null`, changing semantics silently.

**Fix:** enforce `Number.isFinite()` and bounds-check (e.g., `0 < timeout <= 600000`).

## LOW

### 10) **Node IDs based on `Date.now()` can collide**
**Where:** `HookCanvas.tsx`  
```ts

```
Fast drops in the same millisecond can collide, causing undefined graph behavior.

**Fix:** use `crypto.randomUUID()` or a monotonic counter + randomness.
