## MEDIUM

1) **“Structural change” detection is likely incomplete vs React Flow change types**
- **Where:** `HookCanvas.tsx` `handleNodesChange` / `handleEdgesChange` only treat `add|remove|replace` as structural.
- **Why it matters:** XYFlow/ReactFlow `NodeChange`/`EdgeChange` unions can include other change types (version-dependent), e.g. `reset`. If a future/alternate change type mutates topology or node `data` and isn’t flagged as structural, `structuralDirtyRef` may remain `false` and `saveCanvas()` will keep emitting `lastCompiledHooksJsonRef` instead of recompiling → hooks JSON can silently drift from the canvas.
- **Fix:** Invert the logic: treat *only* known layout/selection-only changes as non-structural (`position`, `dimensions`, `select`), and mark everything else structural.

2) **Unknown hook events are “preserved” by compiler/generator, but the UI can’t represent them well**
- **Where:** `canvasCompiler.ts` preserves unknown `event` keys; `canvasGenerator.ts` creates trigger nodes for any event string; `NodeConfigPanel.tsx` uses a `<select>` populated only from `HOOK_EVENTS`.
- **Impact:** If hooks JSON contains an event not in `HOOK_EVENTS`, the trigger node will exist and compile back correctly *until the user edits the event*, at which point they’re forced into the supported list. This is a correctness/roundtrip footgun and it is **not flagged** by `hasUnsupportedHooks`.
- **Fix options:**
  - Add a “Custom/unknown” option (or switch to a text input) that preserves arbitrary event strings.
  - Or treat unknown events as “unsupported” in `hasUnsupportedHooks` and block canvas editing without explicit confirmation.

3) **Canvas ↔ hooks JSON mapping is not bijective (roundtrip can change graph structure)**
- **Where:** `compileCanvasToHooks()` merges groups for triggers sharing the same event; `generateCanvasFromHooks()` always generates a single trigger per event.
- **Impact:** Multiple trigger nodes with the same event (or shared subgraphs) cannot be roundtripped faithfully; a compile→generate cycle collapses triggers and loses layout/grouping intent. This is a semantic invariant mismatch between “canvas graph” and “hooks JSON tree”.
- **Fix:** Either enforce invariants in the canvas (e.g., single trigger per event) or document this explicitly and/or normalize in the UI.

4) **Condition nodes can be shared across multiple triggers; generator cannot reproduce this sharing**
- **Where:** `connectionRules.ts` allows `trigger -> condition` with no restriction on inbound edges to a condition; `compileCanvasToHooks()` will effectively duplicate that condition’s group under each connected trigger event; `generateCanvasFromHooks()` produces separate condition nodes per group.
- **Impact:** The canvas permits a DAG-like reuse, but the underlying hooks format is tree-like. Roundtrip will duplicate nodes/edges and alter topology.
- **Fix:** If reuse is not intended, enforce “single inbound edge per condition” similarly to actions (both in `onConnect` and `pruneInvalidEdges`). If reuse *is* intended, consider surfacing it as an explicit feature and accept non-roundtripability.

5) **Generator can create “empty” condition nodes (matcher group) even if all hooks in the group are invalid/unsupported**
- **Where:** `generateCanvasFromHooks()` creates a condition node immediately when `matcher` is present and `hookList.length > 0`, before validating `buildActionData()` for any hook.
- **Impact:** Canvas may show condition nodes with no actions; if the user later performs any structural edit and recompiles, these groups vanish from output. This is confusing and can look like data loss.
- **Fix:** Only create the condition node after collecting at least one valid action hook for the group.

## LOW

1) **`isValidCanvasState()` accepts `data` as any object (arrays included)**
- **Where:** `HookCanvas.tsx` `if (!n.data || typeof n.data !== 'object') return false;`
- **Impact:** Arrays pass validation; later sanitizers will behave oddly (numeric keys). Likely harmless but avoidable.
- **Fix:** Require a plain object: `Object.prototype.toString.call(n.data) === '[object Object]'` (or prototype check).

2) **`sanitizeNodeData()` does not apply tighter per-field limits for some fields (e.g., `method`, `event`, `matcher`)**
- **Impact:** You can store up to `MAX_STRING_LENGTH` (10k) in fields that should be small; compiler later truncates (`method` to 20, matcher to 200). Not unsafe, but inconsistent.
- **Fix:** Add explicit caps for `method`, `event`, `matcher`, `model`, etc., matching compiler/generator limits.

3) **UI bug: palette icons passed as HTML entities won’t render as intended**
- **Where:** `NodePalette.tsx` passes `icon="&#9889;"`/`"&#9881;"` as a string prop; React won’t decode entities in strings.
- **Impact:** Cosmetic, but confusing UX.
- **Fix:** Use the literal unicode characters (`'⚡'`, `'⚙'`) or render entities directly in JSX, not via string props.
