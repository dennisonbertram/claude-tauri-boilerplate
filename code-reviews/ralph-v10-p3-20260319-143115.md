## Audit: Visual Hook Builder (React Flow canvas → Claude Code hooks JSON)

### HIGH

1) **Sanitization/limits are bypassed on node edits (can create unbounded `hooksCanvasJson`)**
- **Where:** `HookCanvas.tsx` → `onUpdateNode` merges `newData` directly into `node.data`:
  ```ts
  n.id === nodeId ? { ...n, data: { ...n.data, ...newData } } : n
  ```
- **Why it matters:** You defined multiple invariants (`MAX_*_LENGTH`, `DANGEROUS_KEYS`) and enforce them on **drop** and **canvasJson load**, but **not** on user edits from the config panel. A user can paste megabytes into `prompt/command/description/url`, and the app will:
  - keep it in React state,
  - serialize it in `saveCanvas()` into `hooksCanvasJson`,
  - potentially freeze the UI / blow storage (DoS-by-input), despite the presence of max constants.
- **Fix:** sanitize/clamp inside `onUpdateNode` (and ideally before saving canvas JSON). Example:
  ```ts
  const sanitizedPatch = sanitizeNodeData(newData);
  // optionally: sanitize full merged data too
  data: sanitizeNodeData({ ...n.data, ...sanitizedPatch })
  ```

2) **`generateCanvasFromHooks` allows unsanitized `headers` objects into canvas state**
- **Where:** `canvasGenerator.ts` → `buildActionData()` for `http`:
  ```ts
  headers: hook.headers && typeof hook.headers === 'object' && !Array.isArray(hook.headers)
    ? hook.headers
    : undefined,
  ```
- **Why it matters:**
  - Unlike the canvasJson-load path, **hooksJson → canvas** does not pass through `sanitizeNodeData`.
  - This can store arbitrarily large / non-string / CRLF-containing header values in node state and then persist them to `hooksCanvasJson` (even though the compiler later drops invalid headers when producing hooks JSON).
  - Creates inconsistent behavior: user sees/edits a canvas containing header data that will never round-trip to hooks JSON; also risks size-based DoS.
- **Fix:** sanitize headers in `buildActionData` similarly to `canvasCompiler.buildHookEntry` and/or run a shared sanitizer over generated nodes before `setNodes`.

### MEDIUM

1) **Invalid edge invariants aren’t enforced when loading saved canvas JSON**
- **Where:** `HookCanvas.tsx` → `isValidCanvasState()` only checks node existence, not semantics:
  - does **not** validate `sourceType → targetType` against `isValidConnection`
  - does **not** enforce “action has max 1 incoming edge” (only enforced in `onConnect`)
- **Impact:** A crafted/old `hooksCanvasJson` can load an impossible graph. Compilation then becomes order-dependent and may silently drop/misplace hooks.
- **Fix:** during validation or post-load sanitization, filter edges by:
  - `isValidConnection(sourceNode.type, targetNode.type)`
  - enforce single-incoming for `action` targets (drop extras deterministically)

2) **`lastCompiledHooksJsonRef` can be inconsistent with the loaded canvas**
- **Where:** `HookCanvas.tsx` initialization when `hooksCanvasJson` is present:
  ```ts
  lastCompiledHooksJsonRef.current = hooksJson ?? compileCanvasToHooks(...)
  ```
- **Impact:** If `hooksJson` and `hooksCanvasJson` ever diverge (migration, corruption, partial saves), then **layout-only edits** will keep emitting the stale `hooksJson` while saving the new canvas layout—persisting inconsistency until a structural change forces recompile.
- **Fix:** when loading from `hooksCanvasJson`, prefer:
  - `compileCanvasToHooks(loadedNodes, loadedEdges)` as the baseline, or
  - compare and reconcile (e.g., if mismatch, pick one as source-of-truth explicitly).

3) **Non-idempotent JSON↔Canvas conversion for “no matcher” groups**
- **Where:** `canvasCompiler.ts`
  - For Trigger → Action directly, compiler creates **one group per action**.
  - But `canvasGenerator.ts` supports a group without matcher containing **multiple hooks**.
- **Impact:** JSON → canvas → JSON can change grouping boundaries. Today that might be semantically equivalent, but it’s a correctness hazard if group-level semantics expand (or if ordering/grouping is meaningful to users).
- **Fix:** either:
  - compile Trigger-direct actions into a single group (preserving ordering), or
  - represent grouping explicitly in the canvas model.

4) **Hook execution ordering is not tied to layout; ordering depends on edge insertion order**
- **Where:** `canvasCompiler.ts` uses adjacency list order from `edges` array; no sorting by node position.
- **Impact:** Users can’t reliably control hook order by arranging nodes (common expectation in visual builders). Reordering edges internally can change output order unexpectedly.
- **Fix:** sort connected action IDs by `node.position.y` (and groups similarly) before emitting hooks arrays.

### LOW

1) **Palette icons render as literal HTML entities**
- **Where:** `NodePalette.tsx` passes `icon="&#9889;"`, rendered as text.
- **Fix:** pass the actual character (`"⚡"`, `"⚙"`) or render entity directly in JSX, not as a string prop.

2) **Potential controlled/uncontrolled React warnings**
- **Where:** `NodeConfigPanel.tsx` uses `value={data.event}`, `value={data.matcher}` etc. If loaded data lacks those keys, values may be `undefined`.
- **Fix:** default to `''`/first option in UI layer (`value={data.event ?? HOOK_EVENTS[0]}` etc).

3) **Debounce wrapper is instantiated on every render**
- **Where:** `HookCanvas.tsx`
  ```ts
  useCallback(debounce(() => { ... }, 500), [onChange])
  ```
- **Impact:** mostly perf/clarity; React returns the memoized prior callback, but the newly created debounced function is still constructed each render.
- **Fix:** use `useMemo` to create the debounced function once per `onChange`.

4) **Duplicate/inconsistent `DANGEROUS_KEYS` definitions**
- **Where:** `HookCanvas.tsx` has a broad `DANGEROUS_KEYS`, but `isValidCanvasState()` redefines a narrower one.
- **Impact:** confusing/fragile; use a single shared constant.
