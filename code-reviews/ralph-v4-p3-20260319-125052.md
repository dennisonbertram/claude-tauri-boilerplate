## Audit: Visual Hook Builder (React Flow canvas → Claude Code hooks JSON)

### What the code gets right (core invariants / transitions)
- **Basic graph model is clear**: `trigger → (optional condition) → action` with `action` as leaf.
- **UI connection enforcement**: `onConnect` checks `connectionRules` and prevents multiple incoming edges to an `action`.
- **Compilation is reachability-based**: only nodes reachable from triggers are emitted; orphaned subgraphs are skipped (good).
- **Debounced propagation**: avoids recompiling hooks on drag (positions excluded from fingerprint) and batches edits (500ms).

---

## Findings

### HIGH — 1) Unbounded canvas-node data can cause storage/perf DoS and “sticky” huge state
**Where**
- `HookCanvas.tsx`
  - Loading saved canvas JSON: only `stripDangerousKeys(n.data)` is applied, **no length caps**.
  - `onUpdateNode`: merges `newData` into node data with **no sanitization**.
  - `notifyChange`: persists `hooksCanvasJson = JSON.stringify({ nodes, edges })` which will include any huge strings verbatim.
  - `getHooksFingerprint` stringifies `n.data` for every structural change; if `data` fields are very large, this becomes expensive.

**Why it matters**
- A user can paste a very large prompt/command/description in the config panel; output hooks JSON is capped in `canvasCompiler`, but **the persisted `hooksCanvasJson` is not**, causing:
  - large profile payloads,
  - slow renders / slow comparisons,
  - potential freezing when fingerprinting/stringifying.

**Fix**
- Apply **the same sanitization/limits on every data mutation and on load**:
  - On load: `data: sanitizeNodeData(stripDangerousKeys(n.data))` (and optionally deep-strip for known nested fields like `headers`).
  - In `onUpdateNode`: sanitize `newData` before merging, and/or sanitize the merged `data`.
- Consider enforcing per-field limits aligned with compiler caps (e.g. prompt 50k, command 10k) rather than a single `MAX_STRING_LENGTH`.

---

### HIGH — 2) `generateCanvasFromHooks` can be forced into heavy work on very large hooks objects
**Where**
- `canvasGenerator.ts`: `for (const [event, groups] of Object.entries(hooks))`

**Why it matters**
- `Object.entries(...)` allocates an array of **all** entries up-front. If `hooksJson` is extremely large (e.g., pasted into JSON textarea), this can spike memory/time even though node/edge generation caps at 200/400.

**Fix**
- Replace `Object.entries` with a `for...in` loop + `hasOwnProperty` and break as soon as limits are reached, to avoid materializing all entries.
- Optionally add an input size cap for pasted JSON similar to the import cap (or warn/confirm).

### MEDIUM — 1) Loaded canvas JSON is not validated against connection invariants
**Where**
- `HookCanvas.tsx`: `isValidCanvasState` checks dangling edges and node primitive types, but **does not check**:
  - `trigger` cannot be an edge target,
  - `action` cannot be an edge source,
  - `condition` must only target `action`,
  - single-incoming-edge constraint for `action`,
  - no duplicate edges.

**Impact**
- Tampered/old `hooksCanvasJson` can create graphs the UI would never allow.
- Compiler behavior then becomes edge-order dependent and may generate unexpected output or silently drop branches.

**Fix**
- Extend `isValidCanvasState` to enforce `connectionRules` + “action has ≤1 incoming edge”.
- Optionally normalize by removing invalid edges on load (instead of rejecting entire state).

---

### MEDIUM — 2) Compiler deduping via `processedActionIds` can drop valid branches in malformed graphs
**Where**
- `canvasCompiler.ts`: `processedActionIds` is shared across all outgoing paths from a trigger (conditions and direct actions).

**Impact**
- If an `action` is (accidentally or via imported canvas) reachable through two conditions under the same trigger, one path will be omitted depending on traversal order.

**Fix**
- If you enforce single-incoming-edge for actions on load, this becomes moot.
- Otherwise, dedupe only per `(parentNodeId, actionId)` edge or only dedupe duplicate edges, not action IDs globally.

---

### MEDIUM — 3) Round-trip structural mismatch for groups without matcher (multi-hook groups split)
**Where**
- Generator supports `group.hooks` containing multiple hooks with no matcher.
- Compiler emits “trigger → action directly” as **one group per action**.

**Impact**
- Round-tripping `hooksJson → canvas → hooksJson` can change shape (though execution order may remain equivalent depending on runtime semantics).

**Fix**
- Either:
  - model “no-matcher group” as an explicit node/group container in canvas, or
  - compile direct trigger actions into a **single** group per trigger (preserving an ordered list), optionally ordered by y-position.

---

### MEDIUM — 4) `crypto.randomUUID()` may not exist in all runtimes
**Where**
- `HookCanvas.tsx` on drop: `crypto.randomUUID()`

**Impact**
- In older browsers / some embedded contexts this throws, breaking node creation.

**Fix**
- Fallback to a small UUID implementation or ReactFlow’s `getId()` style counter.

---

### LOW — 1) `connectionRules.ts` comment is misleading
**Where**
- Says it’s “Used as the isValidConnection prop on ReactFlow”, but current implementation uses it only in `onConnect`.

**Fix**
- Either pass `isValidConnection` prop to ReactFlow or update comment.

---

### LOW — 2) Canvas save doesn’t update on edge-id-only changes
**Where**
- Fingerprint ignores `edge.id`. If edge IDs changed without endpoint changes, `hooksCanvasJson` might not be updated.

**Fix**
- Include `edge.id` in fingerprint or ensure edge IDs don’t mutate post-creation.

---

### LOW — 3) `sanitizeNodeData` forbids objects/arrays but action `headers` is an object (in generator/compiler)
**Impact**
- Inconsistency: DnD-created actions can never have headers, but generated-from-JSON actions can, and loaded canvas state can.

**Fix**
- Decide whether headers are supported in canvas; if yes, explicitly allow/sanitize them.
