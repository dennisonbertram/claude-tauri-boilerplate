## Attack surface / ingress paths considered
- **`draft.hooksCanvasJson`** (persisted canvas state loaded on mount; could come from disk/cloud/IPC)
- **`draft.hooksJson`** (JSON textarea, imported file, persisted profile)
- **Drag-and-drop `DataTransfer`** (`application/reactflow` payload can be supplied by non-palette sources)
- **NodeConfigPanel inputs** (unbounded strings/numbers pushed into `nodes[].data`)
- **Edges/nodes in saved canvas JSON** (can bypass onConnect rules)

---

## HIGH: Persistent crash/DoS via unvalidated `hooksCanvasJson` node `data` types (schema bypass)
### Where
`HookCanvas.tsx`:
- Loads `hooksCanvasJson` → `JSON.parse` → `isValidCanvasState(saved)` → `setNodes(saved.nodes as CanvasNode[])`
- `isValidCanvasState` only checks `n.data` is an object, **not that its fields are the expected primitive types**.

Node renderers assume primitives:
- `TriggerNode`: renders `{data.event || ...}` → if `event` is an object, React throws (“Objects are not valid as a React child”).
- `ActionNode.getPreview`: uses `data.command.slice(...)` / `data.url` formatting → if `command/url/prompt/description` are non-strings, can throw `TypeError: .slice is not a function`.

### Exploit sketch (malicious persisted canvas JSON)
Set profile’s `hooksCanvasJson` to:
```json
{
  "nodes": [
    {
      "id": "trigger-1",
      "type": "trigger",
      "position": { "x": 0, "y": 0 },
      "data": { "event": {}, "label": "boom" }
    }
  ],
  "edges": []
}
```
Opening Canvas view will crash at render time, potentially **bricking the UI for that profile** (persistent client-side DoS).

### Why it matters
If profiles/hooks can be shared/synced/imported from untrusted sources, this becomes a reliable persistent DoS vector.

### Fix
- Replace `isValidCanvasState` with **strict per-node-type schema validation** (allowlist keys + primitive type checks + max lengths).
- On load, **sanitize/coerce** node data (drop malformed nodes/fields rather than trusting).
- Consider storing canvas JSON as a typed, versioned format and migrate defensively.

## MEDIUM: Prototype pollution / object-prototype manipulation via unsanitized keys in drag-drop sanitization (and other merges)
### Where
`HookCanvas.tsx` `sanitizeNodeData`:
```ts
const result: Record<string, unknown> = {};
...
result[key] = val;  // key can be "__proto__", "constructor", etc.
```
`onDrop` accepts `application/reactflow` from any drag source, not just your palette.

### Impact
- Setting `result["__proto__"] = null` will change `result`’s prototype to `null` (prototype mutation). While not always “global” pollution, it can:
  - break assumptions in downstream code (e.g., code expecting `hasOwnProperty`)
  - create hard-to-debug state corruption/DoS conditions
- Similar issues can arise if other code later uses `Object.assign`/merges with attacker-controlled keys.

### Fix
- Build sanitized objects with `Object.create(null)` and **drop dangerous keys**:
  - `__proto__`, `prototype`, `constructor`, `__defineGetter__`, etc.
- Enforce a **key allowlist per node type** (best).
- Cap number of keys processed from drag payload (see next issue).

## MEDIUM: Performance/Memory DoS via large strings & expensive fingerprinting (`JSON.stringify` of full node data)
### Where
- `getHooksFingerprint(nodes, edges)` stringifies `data` for every node on changes.
- `NodeConfigPanel` allows very large `prompt/command/description` to enter `nodes[].data` (no local cap).
- `hooksCanvasJson` load path has **no string length caps at all** (separate from the HIGH crash issue).

### Impact
A user (or imported/synced profile) can introduce multi-megabyte strings into `node.data`, causing:
- UI freezes on each edit/drag due to repeated `JSON.stringify`
- excessive memory use (React state + fingerprint strings)

### Fix
- Enforce max lengths **at input time** (same caps you later apply in `compileCanvasToHooks`).
- Fingerprint only the **minimal structural fields** (type + ids + specific whitelisted fields), not arbitrary `data`.
- Add maximum size checks for `hooksCanvasJson` and `hooksJson` before parsing.

## MEDIUM: Regex ReDoS risk via `matcher` (catastrophic backtracking) carried to runtime
### Where
- `ConditionConfig` collects regex-like matcher; compiler passes it through (trimmed to 200 chars).
- If the runtime evaluates this regex against tool names, crafted patterns can cause catastrophic backtracking.

### Impact
This is typically a **runtime DoS** against the agent execution environment (not the React UI), but the UI is the authoring surface enabling it.

### Fix
- Validate matchers with a safe-regex check, or use a linear-time engine (e.g., RE2) in the runtime.
- Consider restricting syntax or providing a “literal match” mode.

## LOW: Drag-drop payload size/key-count not bounded (CPU spike)
### Where
`onDrop` parses arbitrary JSON; `sanitizeNodeData` iterates all entries.
No limit on:
- raw payload length
- number of keys

### Fix
- Refuse payloads beyond a small size (e.g., 10–50KB).
- Refuse objects with more than N keys.

---

## LOW: Imported/loaded edges can bypass connection rules and create confusing/losing transformations
### Where
- `isValidCanvasState` validates that edge endpoints exist, but does **not** validate:
  - source/target types follow `connectionRules`
  - single-incoming-edge constraint for action nodes
- `compileCanvasToHooks` then effectively ignores some structures (silent data loss), and `processedActionIds` can cause hooks to “disappear” from expected groups.

### Fix
- Validate edge legality on load; drop or repair invalid edges.
- Enforce “one incoming edge to action” invariant at validation time too.
