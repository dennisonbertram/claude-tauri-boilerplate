## HIGH

### 1) Prototype-chain property access → persistent UI crash/DoS via crafted `hookType` (untrusted canvas state)
**Where**
- `src/components/agent-builder/panels/NodeConfigPanel.tsx`
  - `const ConfigComponent = ACTION_CONFIG_COMPONENTS[data.hookType];`
  - `const titleInfo = ACTION_TITLES[data.hookType] ?? ...`
- `src/components/agent-builder/HookCanvas.tsx`
  - Loads `hooksCanvasJson` → `sanitizeNodeData(...)` **does not validate `hookType` values**.

**Why it’s exploitable**
`ACTION_CONFIG_COMPONENTS` / `ACTION_TITLES` are plain objects with normal prototypes. If an attacker gets an action node into `hooksCanvasJson` (e.g., by sharing a profile or tampering with stored state) and sets:
- `hookType: "constructor"` → `ACTION_CONFIG_COMPONENTS["constructor"]` resolves to `Object` (function), which React treats as a component; it returns a plain object → React runtime error and the whole panel/tree can crash.
- `hookType: "__proto__"` → resolves to `Object.prototype` (non-component) → invalid element type error.
- other prototype keys (`toString`, `valueOf`, `__defineGetter__`, …) can crash or behave unpredictably.

**Impact**
- Clicking/selecting the malicious node can crash the Hook editor view.
- Because selection is needed to delete nodes, this can become a *persistent DoS* for that profile’s canvas view (user may be forced to abandon canvas editing entirely).

**Minimal PoC payload (hooksCanvasJson)**
```json
{
  "nodes": [
    {
      "id": "action-1",
      "type": "action",
      "position": { "x": 0, "y": 0 },
      "data": { "hookType": "constructor", "label": "boom", "command": "echo hi" }
    }
  ],
  "edges": []
}
```

**Fix**
- Validate action `hookType` on load/update to a strict allowlist: `command|http|prompt|agent` (drop node or coerce to safe default).
- Make lookup tables null-prototype or Maps:
  - `const ACTION_CONFIG_COMPONENTS = Object.create(null)` or `new Map()`
- Use `Object.prototype.hasOwnProperty.call(ACTION_CONFIG_COMPONENTS, hookType)` before reading.

### 2) Unbounded JSON parsing from persisted profile fields → app freeze/hang DoS
**Where**
- `src/components/agent-builder/tabs/HooksTab.tsx`
  - `hasDangerousHooks` / `hasUnsupportedHooks` do `JSON.parse(draft.hooksJson)` in `useMemo` tied to `draft.hooksJson` changes (can run repeatedly while typing/pasting).
- `src/components/agent-builder/HookCanvas.tsx`
  - `JSON.parse(hooksCanvasJson)` with no length limit (limits are only enforced *after* parse via counts).
  - `generateCanvasFromHooks(hooksJson)` parses the full JSON.

**Why it’s exploitable**
Import has a 500KB limit, but:
- A user can paste arbitrarily large JSON into the textarea (no cap).
- A malicious/remote profile can supply very large `hooksJson` / `hooksCanvasJson`.
`JSON.parse` on multi-megabyte strings can lock the UI thread in a Tauri/React desktop app, effectively DoS’ing the editor.

**Impact**
- Opening the Hooks tab (or switching views) can freeze/hang the UI.
- Potential repeated parsing on each edit can peg CPU.

**Fix**
- Enforce max lengths for `hooksJson` and `hooksCanvasJson` before parsing (both on paste and on load).
- Debounce/idle-time parsing for warnings; consider moving JSON parsing to a Web Worker.
- For `hooksCanvasJson`, reject/clear if over size threshold before `JSON.parse`.

## MEDIUM

### 3) “Event key injection” in compiled hooks JSON can break downstream consumers / cause logic bypasses
**Where**
- `src/lib/canvasCompiler.ts`
  - Preserves unknown trigger events as object keys: `result.hooks[event] = groups;`
  - Only blocks a small dangerous-key set; no charset/length normalization.

**Why it matters**
If an attacker can get a trigger node with `data.event = "hasOwnProperty"` (or similar) into the canvas state, the compiled JSON contains:
```json
{"hooks":{"hasOwnProperty":[...]}}
```
Any downstream consumer that does `hooks.hasOwnProperty(...)` (common bug) will crash or misbehave. This is a classic “unexpected key” injection class.

**Fix**
- Constrain `event` to a known allowlist at compile time (or store unknown events in an array form rather than as object keys).
- At minimum, deny keys matching `Object.prototype` built-ins (`toString`, `hasOwnProperty`, etc.) or output using null-prototype objects everywhere (you already do internally, but consumers won’t).

### 4) HTTP header injection surface (semantic, not CRLF): allows dangerous hop-by-hop headers
**Where**
- `src/lib/canvasCompiler.ts` and `src/lib/canvasGenerator.ts` sanitize CRLF and length, but allow arbitrary header names/values otherwise.

**Why it matters**
Even without CRLF, allowing headers like `Host`, `Connection`, `Transfer-Encoding`, `Proxy-Authorization`, etc. can enable request smuggling-like behavior or SSRF amplification depending on the runtime HTTP client behavior.

**Fix**
- Add an allowlist of safe headers or a denylist of hop-by-hop/sensitive headers.
- Normalize header names and restrict to RFC7230 token charset.

### 5) Potential hooks/canvas divergence & silent data corruption via incomplete “structuralDirty” detection
**Where**
- `src/components/agent-builder/HookCanvas.tsx`
  - `handleNodesChange`/`handleEdgesChange` only treat `add/remove/replace` as “structural”.
  - Other change types (library-dependent) that can affect semantics (e.g., if XYFlow introduces `reset`/`update` variants impacting IDs/types) may not trigger recompilation, causing `hooksJson` to remain stale while canvas changes.

**Fix**
- Treat any change that can affect `id/type/data` as structural.
- Alternatively, recompute hooks JSON whenever node `data` changes (you do on `onUpdateNode`, but not necessarily for all change paths).
