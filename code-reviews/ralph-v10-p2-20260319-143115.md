## CRITICAL

### 1) Debounced save is canceled on unmount ‚Üí easy loss of user edits
**Where:** `HookCanvas.tsx` debounced `saveCanvas` + cleanup `saveCanvas.cancel()`  
**Impact:** If the user edits anything and switches JSON/Canvas (or navigates away) within 500ms, changes are silently dropped because you cancel instead of flushing. This is a classic ‚ÄúI edited, switched tabs, and it reverted‚Äù data-loss bug.

**Repro:**
1. Open Canvas view.
2. Select an action node, paste/edit text in prompt/command.
3. Immediately click ‚ÄúJSON‚Äù view (or close the panel) before 500ms.
4. Return to Canvas ‚Üí edits may be gone; parent state never received the change.

**Fix:**
- On unmount (or before view switch), **flush pending saves** instead of canceling:
  - Keep last ‚Äúdirty‚Äù state refs and call a non-debounced `commitNow()` in cleanup.
  - Or implement debounce with `{ flush() }` semantics and call `flush()` on unmount.
- Also consider `beforeunload`/route-change guard if you rely on delayed persistence.

---

### 2) Node updates are not sanitized/limited ‚Üí `hooksCanvasJson` can explode and freeze/crash
**Where:** `HookCanvas.tsx` `onUpdateNode` merges raw `newData`; `saveCanvas` JSON.stringify persists full node data  
**Impact:** You enforce length limits only on drag/drop and load (`sanitizeNodeData`), but **not** when editing via the config panel. A user can paste multi-megabyte text into Prompt/Command/Description. Even though `compileCanvasToHooks()` slices output, the **canvas state** persisted to `hooksCanvasJson` will contain the full huge string, causing:
- UI lag/freeze (large React state + serialization every change)
- storage bloat / persistence failures
- import/export/performance issues

**Repro:**
1. Add a Prompt action.
2. Paste ~5‚Äì50MB into the prompt textarea.
3. Observe lag; `hooksCanvasJson` becomes massive; subsequent renders/saves choke.

**Fix:**
- Apply `sanitizeNodeData()` (and max lengths) inside `onUpdateNode` before merging.
- Add global size cap for `hooksCanvasJson` (and refuse/save-truncate with UX message).
- Consider throttling config-panel edits separately from canvas layout updates.

## HIGH

### 3) Round-trip data loss beyond ‚Äúunsupported types‚Äù (unsupported fields + extra top-level keys)
**Where:** `canvasGenerator.ts` + `canvasCompiler.ts`  
**Impact:** You warn only for *unsupported hook types*, but Canvas mode will drop:
- unknown hook fields (e.g., `env`, `cwd`, `body`, custom fields, advanced HTTP options)
- any top-level JSON keys besides `{ hooks: ... }` (compiler always emits only that shape)

This is silent corruption of existing configs even when all hook `type`s are ‚Äúsupported‚Äù.

**Fix:**
- Either preserve unknown fields in node `data` and re-emit them, or
- Store original JSON and apply minimal patches, or
- Add a strong warning: ‚ÄúCanvas will drop unsupported fields and metadata‚Äù + diff preview.

---

### 4) Hook ordering/grouping is not stable or faithfully preserved
**Where:** `compileCanvasToHooks()`  
**Impact:**
- Trigger‚ÜíAction direct connections compile into **one group per action**, even if the original JSON had a single group containing multiple hooks (structure changes).
- Hook order depends on `edges` insertion order (and adjacency list), not on visual layout. If order matters (often does), users can get nondeterministic or unintuitive execution order.

**Fix:**
- Preserve original grouping when generating canvas (store `groupId`), and compile back using it.
- Define deterministic ordering (e.g., sort actions by `position.y`, then by id).
- If you keep ‚Äúone group per action‚Äù, explicitly document it and provide ordering controls.

---

### 5) Loaded canvas JSON can violate connection constraints ‚Üí compile output can be wrong / hooks dropped
**Where:** `isValidCanvasState()` + load path in `HookCanvas.tsx`  
**Impact:** `isValidCanvasState` checks basic shape and dangling edges, but does **not** enforce your own semantic rules:
- multiple incoming edges to an `action` (UI prevents it, but loaded JSON can contain it)
- invalid type combinations (e.g., condition‚Üícondition), trigger as target, etc.

Compiler behavior in these cases is inconsistent (e.g., `processedActionIds` can cause actions to be skipped), creating hard-to-debug missing hooks.

**Fix:**
- Validate edges against `connectionRules` during load.
- Normalize/repair invalid edges (drop extras) and warn the user.

---

### 6) `onUpdateNode` allows unsafe shapes ‚Üí potential stringify failure / proto keys reintroduced
**Where:** `HookCanvas.tsx` `onUpdateNode`  
**Impact:** While the UI only sends primitives, nothing prevents other callers/devtools from injecting:
- `__proto__` keys back into `data`
- cyclic objects (will throw in `JSON.stringify` in `saveCanvas`, breaking persistence)
- large nested objects you intended to skip in `sanitizeNodeData`

**Fix:**
- Sanitize/validate in `onUpdateNode` (same path as drag/drop/load).
- Wrap `JSON.stringify` in `saveCanvas` with try/catch and fail closed (don‚Äôt overwrite parent state with partials).

## MEDIUM

### 7) Canvas does not react to prop changes while mounted (stale editor risk)
**Where:** `HookCanvasInner` initialization effect `useEffect(..., [])`  
**Impact:** If `hooksJson` / `hooksCanvasJson` changes externally while Canvas is open (profile refresh, background load, parent state restore), the canvas won‚Äôt reload and can overwrite newer data on the next save.

**Fix:** Add a controlled ‚Äúsource-of-truth‚Äù strategy:
- Depend on `profileId`/revision and reload when it changes, or
- Detect prop changes and prompt user to reload/merge.

---

### 8) Controlled/uncontrolled input hazards when loaded data has missing fields
**Where:** `NodeConfigPanel` inputs/selects use `value={data.event}`, `value={data.matcher}`, etc.  
**Impact:** If sanitized/loaded data yields `undefined`, React will warn and UI behavior becomes inconsistent.

**Fix:** Always coerce:
- `value={data.event ?? ''}` with a placeholder `<option value="">Select‚Ä¶</option>`
- `value={data.matcher ?? ''}`, etc.

---

### 9) MAX_NODE_COUNT enforcement can be bypassed with rapid drops (stale closure)
**Where:** `onDrop` checks `if (nodes.length >= MAX_NODE_COUNT)` using captured `nodes`  
**Impact:** Multiple drops before React updates state can exceed the cap.

**Fix:** Enforce in the functional updater:
```ts
setNodes(prev => prev.length >= MAX_NODE_COUNT ? prev : [...prev, newNode]);
```

---

### 10) Dangerous-hook detection in HookCanvas load uses substring matching (false negatives/positives)
**Where:** `hooksAlreadyDangerous = hooksJson.includes('"type":"command"')...`  
**Impact:** Pretty-printed JSON (`"type": "command"`) won‚Äôt match, causing unnecessary confirm prompts or inconsistent safety gating.

**Fix:** Parse JSON (like `HooksTab` already does) instead of substring searching.

---

### 11) Headers from JSON ‚Üí canvas are not sanitized on generation
**Where:** `canvasGenerator.ts` passes `hook.headers` through as-is  
**Impact:** Even if runtime output sanitizes later, `hooksCanvasJson` can persist CRLF-containing header keys/values or proto-ish keys, and future features/UI might accidentally use them unsafely.

**Fix:** Sanitize headers in `buildActionData` similarly to compiler/load sanitizer.
