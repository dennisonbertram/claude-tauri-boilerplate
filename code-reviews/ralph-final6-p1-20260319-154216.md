## HIGH — Canvas layout changes can silently change *visual* execution order without recompiling hooks JSON (integrity/safety bug)
**Where:** `src/components/agent-builder/HookCanvas.tsx` (`structuralDirtyRef`, `handleNodesChange`, `saveCanvas`) + `compileCanvasToHooks` ordering logic  
**What happens:** `compileCanvasToHooks()` sorts connected nodes by `position.y` to generate deterministic, top-to-bottom hook ordering. But `HookCanvas` only sets `structuralDirtyRef.current = true` for node changes of type `add/remove/replace`. Pure position changes (dragging to reorder) are treated as “layout-only”, so `saveCanvas()` reuses `lastCompiledHooksJsonRef` and does **not** recompile hooks JSON.  
**Impact:** The canvas can show a reordered sequence of actions, while the persisted `hooksJson` keeps the old order. For command/http hooks, this can be safety-critical: users may think they moved a dangerous action later (or behind a “safer” step), but the actual runtime order remains unchanged. This is a real “what you see is not what executes” failure.

**Repro:**
1. Create Trigger → Condition → two Actions.
2. Drag the 2nd action above the 1st (swap Y positions).
3. Observe canvas order changed, but exported/saved JSON action order does not.

**Fix:**
- Treat position changes as “structural” for compilation purposes *if ordering depends on Y*. E.g. in `handleNodesChange`, mark dirty when `c.type === 'position'` (or when `position.y` changes).
- Alternatively: remove Y-based ordering from compilation and store explicit order metadata in node data.

## HIGH — Action `hookType` is not validated; attacker-controlled values can hit Object prototype keys and crash the config panel (persistent-ish UI DoS)
**Where:**  
- `HookCanvas.tsx` loads `hooksCanvasJson` → `sanitizeNodeData(stripDangerousKeys(n.data))` but does **not** validate `data.hookType`  
- `NodeConfigPanel.tsx` → `const ConfigComponent = ACTION_CONFIG_COMPONENTS[data.hookType];` then renders `<ConfigComponent ... />`

**What happens:** `ACTION_CONFIG_COMPONENTS` / `ACTION_TITLES` are plain objects with normal prototypes. If an action node’s `data.hookType` is set to a prototype key like `"toString"`, lookup returns `Object.prototype.toString` (a function). React then treats it as a function component and calls it, which throws (`TypeError: Cannot convert undefined or null to object`), crashing rendering when the node is selected.

**Impact:**
- A malicious/tainted `hooksCanvasJson` (or drag payload via `dataTransfer`) can create a node that **crashes the editor when selected**, preventing deletion/editing via the UI and effectively bricking canvas editing for that profile until the user resets to JSON view (if they realize that clears `hooksCanvasJson`).

**PoC `hooksCanvasJson`:**
```json
{
  "nodes": [
    {
      "id": "action-1",
      "type": "action",
      "position": { "x": 0, "y": 0 },
      "data": { "hookType": "toString", "label": "boom", "prompt": "x" }
    }
  ],
  "edges": []
}
```

**Fix:**
- Validate `hookType` with an allowlist (`command|http|prompt|agent`) in **all** ingestion paths:
  - `isValidCanvasState()` should reject action nodes whose `data.hookType` is not in the allowlist.
  - `sanitizeNodeData()` should clamp/strip invalid `hookType`.
- Make these maps prototype-safe:
  - Use `Object.create(null)` for `ACTION_CONFIG_COMPONENTS`, `ACTION_TITLES`, etc., or use `Map`.
  - Use `Object.hasOwn()` checks before consuming lookups.

## MEDIUM — `hasUnsupportedHooks` can fail open (returns false) on malformed `group.hooks`, suppressing the warning and enabling silent data loss
**Where:** `src/components/agent-builder/tabs/HooksTab.tsx` (`hasUnsupportedHooks`)

**What happens:** The code does:
```ts
if (!group?.hooks) continue;
for (const hook of group.hooks) { ... }
```
If `group.hooks` is a truthy non-array object (common in malicious or corrupted JSON), `for..of` throws. The surrounding `try/catch` returns `false` on any exception → **unsupported warning is suppressed**.

**Impact:** User may switch to Canvas view without the “unsupported hooks will be dropped” interstitial, and on first edit the canvas compilation will overwrite `hooksJson`, dropping the malformed/unsupported content.

**PoC hooksJson (malformed hooks list):**
```json
{
  "hooks": {
    "PreToolUse": [
      { "matcher": "Bash", "hooks": { "type": "command", "command": "echo hi" } }
    ]
  }
}
```

**Fix:**
- Fail closed: on parse/walk errors, return `true` (treat as unsupported).
- Require `Array.isArray(group.hooks)` before iterating; otherwise flag unsupported.

## MEDIUM — Unbounded JSON size via paste can cause repeated `JSON.parse` UI hangs (client-side DoS)
**Where:** `HooksTab.tsx` (`hasDangerousHooks`, `hasUnsupportedHooks`, `jsonValid`) all `JSON.parse(draft.hooksJson)` inside render-driven `useMemo`

**What happens:** Import has a 500KB limit, but the textarea allows arbitrarily large pastes. Multiple `useMemo` blocks parse the entire string. With multi-megabyte JSON, typing/pasting can freeze the UI.

**Impact:** Local DoS (hang) and potential loss of unsaved work.

**Fix:**
- Enforce a max size for `hooksJson` in the textarea path too (not just import).
- Debounce parsing and/or move parsing to a Web Worker.
- Avoid parsing 2–3 times independently; parse once and reuse results.

## MEDIUM — Generator truncation (200 nodes / 400 edges) is not surfaced as “will drop hooks”, causing accidental destructive edits
**Where:** `src/lib/canvasGenerator.ts` caps generation; `HooksTab.tsx` warnings do not cover truncation

**What happens:** `generateCanvasFromHooks()` silently truncates large hook sets to `MAX_GENERATED_NODES/MAX_EDGES` (it logs `console.warn`, but user doesn’t see it). If the user then edits in canvas, `compileCanvasToHooks()` writes back only what was generated → permanent loss of omitted hooks.

**Impact:** Silent destructive data loss on large profiles; can remove safety-related hooks/conditions.

**Fix:**
- Detect truncation and show a blocking warning similar to “unsupported hooks”.
- Store a “truncated” flag in `hooksCanvasJson` and refuse to overwrite `hooksJson` unless user explicitly confirms.

## LOW — Header sanitization doesn’t enforce RFC-valid header name characters (can break downstream / request smuggling edge cases in other layers)
**Where:** `sanitizeNodeData()` (HookCanvas) and `buildHookEntry()` (compiler) sanitize CRLF and length, but allow arbitrary characters (spaces, colons) in header names.

**Impact:** Mostly reliability, but in some HTTP stacks odd header names can trigger undefined behavior.

**Fix:** Enforce a conservative header-name regex, e.g. `^[!#$%&'*+.^_`|~0-9A-Za-z-]+$`.

## LOW — “Dangerous hook” confirmations are inconsistent across sources (social-engineering footgun)
**Where:**  
- Import JSON: confirm on command/http present  
- Load canvas JSON: confirm on command/http nodes  
- But switching to canvas with dangerous `hooksJson` only shows a passive warning (no confirm)

**Impact:** If profiles are shared, a user can be nudged into canvas editing and saving dangerous hooks with minimal friction. (Not a technical sandbox escape, but a practical safety issue.)

**Fix:** Consider requiring an explicit acknowledgement when *entering canvas edit mode* if dangerous hooks exist (similar to the unsupported interstitial).
