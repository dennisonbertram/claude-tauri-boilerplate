## CRITICAL

### 1) Hooks JSON can be reverted to an older version (race between `notifyChange` and `saveLayout`)
**Where:** `HookCanvas.tsx` — `notifyChange` (500ms) vs `saveLayout` (2000ms) + `lastCompiledHooksJsonRef`

**What happens:**
- Structural edits (add node/edge, edit node data) schedule `notifyChange()` which **recompiles hooks** after 500ms and updates `lastCompiledHooksJsonRef`.
- Pure position edits schedule `saveLayout()` which **does not compile hooks**; it re-saves the canvas using `lastCompiledHooksJsonRef.current`.

**Break scenario (data loss):**
1. Make a structural change (e.g., connect Trigger → Action). `notifyChange` is pending.
2. Immediately drag a node (position-only change). This schedules `saveLayout`.
3. `saveLayout` fires later and calls `onChange(currentHooksJson, canvasJson)` with **stale `currentHooksJson`**, overwriting the parent’s `hooksJson` with the old version while the canvas JSON contains new structure.

**Impact:** silent corruption / rollback of `hooksJson`, leading to lost hooks when switching views, exporting, or reloading.

**Fix ideas:**
- Don’t let `saveLayout` write `hooksJson` at all; create a separate `onLayoutChange(canvasJson)` channel.
- Or make `saveLayout` compute hooks if a structural change is pending (e.g., track a `dirtyStructural` flag).
- Or update `lastCompiledHooksJsonRef` *immediately* on structural edits (even if debounced notify is delayed).

---

### 2) `lastCompiledHooksJsonRef` gets clobbered on first render, breaking “layout-only” saves and initial consistency
**Where:** `HookCanvas.tsx` — structural watcher effect:

```ts
if (isFirstRender.current) {
  ...
  lastCompiledHooksJsonRef.current = hooksJson ?? '';
  return;
}
```

**Why it’s bad:**
- The mount-loader effect may set `lastCompiledHooksJsonRef.current` to a compiled value (or should), but the first-run watcher overwrites it with `hooksJson ?? ''`.
- If `hooksJson` is `''` (very plausible in drafts) but `hooksCanvasJson` contains a real graph, then `lastCompiledHooksJsonRef` becomes `''`.
- Then `saveLayout()` early-returns (`if (!currentHooksJson) return`) so **positions won’t persist** until a structural change happens.
- More broadly: the “source of truth” for hooks can become empty even while the canvas has a valid graph.

**Impact:** inconsistent state; layout changes not saved; potential for saving/exporting an empty hooks JSON while canvas appears populated.

**Fix ideas:**
- On first render, **do not overwrite** `lastCompiledHooksJsonRef` if it already has a value.
- After loading nodes/edges (from canvas JSON or generated), trigger a one-time sync to parent (`onChange(compiledHooks, canvasJson)`) so parent state is coherent.

## HIGH

### 3) Unsupported hooks can be dropped without the “unsupported types” confirmation (missing/invalid `type`)
**Where:** `HooksTab.tsx` — `hasUnsupportedHooks`

Current logic only flags unsupported when `hook.type` exists:
```ts
if (hook?.type && !SUPPORTED_TYPES.has(hook.type)) return true;
```

**Problem:**
- Hooks with **no `type`** but with meaningful fields (`command`, `url`, etc.) are not flagged as unsupported.
- Canvas generator (`buildActionData`) requires a valid `type`; those hooks will be **silently dropped** on canvas edit/roundtrip.

**Impact:** irreversible data loss when switching to Canvas (even after user confirmation intended for unsupported types).

**Fix:** Treat missing/invalid `type` as unsupported when the hook object contains recognizable hook payload fields.

---

### 4) Node edits bypass all sanitization/limits → easy UI freeze + huge JSON.stringify cost
**Where:** `HookCanvas.tsx` — `onUpdateNode` merges raw `newData` into node `data` with no `sanitizeNodeData`.

**Why it breaks:**
- Users can paste multi-megabyte `prompt`/`command` in `NodeConfigPanel`.
- Every nodes/edges change recomputes `getHooksFingerprint()` which does `JSON.stringify` over `n.data` for every node → can become catastrophically slow.
- Compiler truncation happens later, but the UI state already contains the huge strings.

**Impact:** performance death spiral / memory blowup; can hang the app.

**Fix:** Apply length/field sanitization inside `onUpdateNode` (same limits as drag/drop/load). Consider debouncing config-panel updates.

---

### 5) `generateCanvasFromHooks` outputs unsanitized `headers` (and other fields) directly into node data
**Where:** `canvasGenerator.ts` → `buildActionData()` for `http`:

```ts
headers: hook.headers && typeof hook.headers === 'object' ... ? hook.headers : undefined
```

**Issues:**
- No key/value length caps, no CRLF stripping, no filtering of dangerous keys like `__proto__`, etc.
- HookCanvas sanitizes headers *only* when loading `hooksCanvasJson` or on drag/drop; generation path sets nodes directly without sanitization.

**Impact:** large/malicious headers can bloat state and cause performance issues; plus downstream risk if headers are later used unsafely.

**Fix:** sanitize generator output (shared sanitizer), and also filter dangerous header keys in `canvasCompiler` as defense-in-depth.

---

### 6) Graph invariants not enforced: multiple incoming edges to a Condition can produce nonsense semantics
**Where:** `HookCanvas.tsx` `onConnect` only enforces single-incoming for `action`, not for `condition`.

**Impact:**
- A single Condition node shared by multiple triggers doesn’t map cleanly to hooks JSON; compile will effectively duplicate action groups across events.
- Roundtripping JSON → canvas will not preserve that shared structure, surprising the user.

**Fix:** enforce “Condition has exactly one incoming edge (from a Trigger)” in `onConnect` and in canvas-state validation.

---

## MEDIUM

### 7) Canvas-state validation doesn’t enforce connection rules (invalid edges survive and confuse compile/save)
**Where:** `HookCanvas.tsx` `isValidCanvasState()`

It validates dangling edges but not whether `trigger→trigger`, `action→condition`, etc. exist, nor single-incoming constraints.

**Impact:** users can load a “valid” canvas JSON that compiles unpredictably (edges ignored) and later may not be re-loadable after edits.

**Fix:** validate edges against `connectionRules` + enforce “action has ≤1 incoming”, “condition has ≤1 incoming”, etc.

---

### 8) Dangerous hook detection during canvas load relies on brittle substring matching
**Where:** `HookCanvas.tsx`:

```ts
hooksJson.includes('"type":"command"')
```

**Impact:** misses equivalent JSON with spacing/order differences; may skip warning/confirmation when it should appear.

**Fix:** parse JSON and inspect structurally (reuse the safer logic from `HooksTab`).

---

### 9) `URL.revokeObjectURL(url)` immediately after `a.click()` can break downloads in some environments
**Where:** `HooksTab.tsx` export

**Fix:** revoke in a `setTimeout`, or after the browser has started the download.

---

### 10) `crypto.randomUUID()` may not exist in some runtimes
**Where:** `HookCanvas.tsx` `onDrop`

**Impact:** drag/drop node creation fails hard on older WebViews.

**Fix:** fallback to a UUID library / simple counter.

---

## LOW

### 11) Palette icons passed as HTML entity strings render literally
**Where:** `NodePalette.tsx` passes `icon="&#9889;"` etc; React will render the literal text.

**Fix:** use actual unicode chars (`'⚡'`, `'⚙'`) or render entities in JSX, not as strings.

---

### 12) Import invalid JSON is silently ignored
**Where:** `HooksTab.tsx` import `catch { /* silently ignore */ }`

**Impact:** confusing UX; user thinks import did nothing.

**Fix:** show an error toast/alert.

---

### 13) Trigger `<select value={data.event}>` can become `undefined`
If a trigger node is malformed/generated oddly, React may warn about uncontrolled/controlled mismatch.

**Fix:** default to `''` and include a placeholder option.
