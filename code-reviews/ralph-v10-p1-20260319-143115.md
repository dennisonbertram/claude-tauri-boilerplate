## HIGH

### 1) Dangerous-hook confirmation bypass when loading `hooksCanvasJson` (leads to silent command/HTTP hook injection)
**Where:** `src/components/agent-builder/HookCanvas.tsx` (mount effect loading `hooksCanvasJson`)

**Bug:** The “already dangerous” check uses raw substring matching on `hooksJson`:

```ts
const hooksAlreadyDangerous =
  hooksJson &&
  (hooksJson.includes('"type":"command"') || hooksJson.includes('"type":"http"'));
```

If this evaluates true, **the app will load a dangerous canvas without confirmation**, even if `hooksJson` contains *no actual command/http hooks*. An attacker can embed that substring inside any string field (e.g., a prompt/description) to bypass the confirmation.

**Impact chain (practical exploit):**
1. Victim receives/opens a profile that contains:
   - A *benign* `hooksJson` (e.g., only prompt hooks), but with a prompt string containing the literal text `"type":"command"` somewhere.
   - A `hooksCanvasJson` that contains action nodes with `hookType: "command"` / `"http"` and payloads.
2. User switches to Canvas view:
   - No confirmation prompt (bypassed).
   - `HooksTab`’s `hasDangerousHooks` banner won’t show (because parsed hooks are benign).
3. User makes any edit in canvas (especially via config panel, which marks `structuralDirtyRef.current = true`):
   - `compileCanvasToHooks()` runs.
   - Malicious command/http hooks are **written into `hooksJson`** and saved.
4. When the agent runs, those hooks can execute commands / make HTTP requests (RCE/SSRF via configuration).

**Minimal PoC `hooksJson` (benign but bypasses confirm):**
```json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          { "type": "prompt", "prompt": "note: \"type\":\"command\" is just text" }
        ]
      }
    ]
  }
}
```
Pair with a `hooksCanvasJson` that includes a command action node. Canvas loads without confirmation; no banner; a subsequent edit persists the dangerous hook into `hooksJson`.

**Fix recommendations:**
- Replace substring checks with structured parsing (same style as `hasDangerousHooks`), and base the “already dangerous” decision on actual hook entries, not raw text.
- Independently warn if the **canvas** contains dangerous actions, regardless of `hooksJson` content.
- Extend `canvasHasDangerousActions()` to also flag actions with dangerous fields (e.g., `command`/`url`) even if `hookType` is missing/malformed.

## MEDIUM

### 2) Unbounded node-data updates allow persistent UI/IPC DoS and storage bloat
**Where:** `HookCanvas.tsx` `onUpdateNode` and debounced `saveCanvas()`

**Bug:** `sanitizeNodeData()` (with length caps like `MAX_PROMPT_LENGTH`, etc.) is applied only:
- when loading `hooksCanvasJson`
- on palette drag/drop

…but **not** when editing via the config panel:

```ts
n.id === nodeId ? { ...n, data: { ...n.data, ...newData } } : n
```

So a user (or malicious automation) can paste extremely large strings into `prompt/command/description/url/event/matcher`, and:
- `saveCanvas()` will `JSON.stringify()` huge data every 500ms,
- `onChange()` will push massive payloads through state/IPC/persistence,
- the app can freeze/crash, and the oversized `hooksCanvasJson` becomes persistent.

**Fix recommendations:**
- Apply `sanitizeNodeData()` (and key stripping) inside `onUpdateNode` before merging.
- Add hard limits in the config UI (max length, early truncation, or refusal) consistent with compiler limits.

### 3) Hooks → Canvas generation lacks sanitization for several fields (oversized strings + unsafe object keys carried into runtime state)
**Where:** `src/lib/canvasGenerator.ts`

Issues:
- Trigger event keys are used verbatim as `TriggerNodeData.event`/`label` with **no length cap**.
- HTTP `headers` are copied as-is into node data:
  ```ts
  headers: hook.headers && typeof hook.headers === 'object' ... ? hook.headers : undefined
  ```
  This can carry dangerous keys (e.g., `__proto__`) and very large objects into React state. Even if later compilation sanitizes some parts, the **in-memory canvas state** can be weaponized for performance degradation.

**Fix recommendations:**
- Sanitize/normalize generated node `data` through the same `sanitizeNodeData()` used elsewhere (or a shared sanitizer in `lib/`).
- Cap event string length (and ideally validate against known events unless explicitly allowing custom ones).

## LOW

### 4) Prototype manipulation risk in header sanitization during compile (`__proto__` not filtered)
**Where:** `src/lib/canvasCompiler.ts` `buildHookEntry()` (http headers)

It filters CRLF/length but does **not** filter `__proto__` / `constructor` / `prototype` keys:
```ts
sanitizedHeaders[k] = v; // if k="__proto__" this can mutate prototype of sanitizedHeaders
```

Because `v` must be a string, this is not classic global prototype pollution, but it can still create weird objects and future footguns if that object is later merged/assigned elsewhere.

**Fix recommendations:**
- Reject dangerous keys (reuse `DANGEROUS_KEYS`).
- Use `Object.create(null)` for `sanitizedHeaders` to avoid magic `__proto__` behavior.

---

### 5) Inconsistent dangerous-hook detection methods (UX/security logic gaps)
**Where:**
- `HooksTab.tsx` uses structured parsing for `hasDangerousHooks` (good).
- `HookCanvas.tsx` uses substring checks for `hooksAlreadyDangerous` (bad).

This inconsistency creates both false positives and false negatives, and (combined with finding #1) becomes security-relevant.

**Fix recommendations:**
- Centralize hook parsing + dangerous-type detection in one shared function and use it everywhere.
