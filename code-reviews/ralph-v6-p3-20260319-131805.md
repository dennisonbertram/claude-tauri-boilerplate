## CRITICAL

### 1) Silent data loss on reload: `sanitizeNodeData` truncates all strings to 10k
**Where:** `HookCanvas.tsx` (`MAX_STRING_LENGTH = 10_000`, used when loading `hooksCanvasJson`)

**Why it’s critical:** `canvasCompiler`/`canvasGenerator` explicitly support much larger strings (notably `prompt` up to **50,000**), but when a saved canvas is loaded, **all** string fields are truncated to 10k. This becomes irreversible because after loading, the “structural change” effect recompiles and calls `onChange`, overwriting the original `hooksJson` with the truncated content—potentially **without any user edit**.

**Repro (data loss):**
1. Have a prompt hook with >10k prompt in `hooksJson` (supported by generator/compiler).
2. Enter canvas view → it saves `hooksCanvasJson`.
3. Later reopen canvas view → load from `hooksCanvasJson` → prompt truncated to 10k.
4. The nodes/edges watcher recompiles and calls `onChange` → profile `hooksJson` is now truncated permanently.

**Fix direction:**
- Either raise `MAX_STRING_LENGTH` to at least the largest supported field (50k), **or** make sanitization field-aware (per node type / per key limits).
- Additionally, prevent emitting `onChange` during hydration (see HIGH #1), so load-time sanitization can’t silently rewrite `hooksJson`.

---

### 2) Silent data loss: headers (and any object-valued fields) are dropped on canvas reload
**Where:** `HookCanvas.tsx` `sanitizeNodeData` (“Skip objects/arrays — keep data flat”)

**Why it’s critical:** `canvasCompiler.buildHookEntry()` and `canvasGenerator.buildActionData()` both support `http.headers: Record<string,string>`. But:
- Canvas state save (`notifyChange`) serializes nodes with `data.headers`.
- Canvas state load sanitization **drops** objects → `headers` are lost.
- The watcher recompiles and calls `onChange` → headers are removed from `hooksJson` permanently, potentially without edits.

**Fix direction:**
- Either explicitly support `headers` in canvas state with deep sanitization (key/value length + CRLF checks + dangerous keys), **or** remove `headers` support from compiler/generator to avoid misleading partial support.
- Same as above: avoid load-time recompilation rewriting JSON.

---

## HIGH

### 1) Unintended state transition: hydration triggers `onChange()` (rewrites hooks) even with no user edits
**Where:** `HookCanvas.tsx` nodes/edges watcher effect + `notifyChange`

**Why it matters:** After initial `setNodes/setEdges` (from either generator or saved canvas), the watcher treats it as a structural change and schedules `notifyChange()`, which:
- recompiles hooks JSON (normalizes formatting/order),
- can drop data that canvas can’t represent,
- amplifies CRITICAL issues (string truncation / headers loss) into permanent overwrites.

**Fix direction:**
- Add an explicit “hydrating” gate:
  - set a ref `isHydrating=true` before loading,
  - after nodes/edges have been set and `prevStructuralRef` initialized to the loaded fingerprint, set `isHydrating=false`,
  - only call `notifyChange` when `!isHydrating`.
- Alternatively, keep `hooksJson` as source-of-truth and treat `hooksCanvasJson` as **layout-only**, overlaying positions onto nodes generated from `hooksJson`.

---

### 2) Performance/UX risk: `getHooksFingerprint()` JSON-stringifies full node data on every node move
**Where:** `HookCanvas.tsx` `getHooksFingerprint(nodes, edges)` in an effect that runs on every `nodes` change (including dragging)

**Why it matters:** On drag, React Flow updates node positions frequently; this effect still recomputes a full JSON stringify of `data` for every node. With large `prompt` strings (up to 50k supported), this can cause significant UI jank.

**Fix direction:**
- Compute a structural fingerprint without serializing large fields:
  - Use `{id,type}` plus a stable “data version” counter per node updated only when data changes,
  - or hash only small whitelisted keys,
  - or track position-only changes via `onNodesChange` change types instead of stringifying the whole graph.

---

### 3) Key sanitization/type validation gap on updates (potential security + invariants drift)
**Where:** `HookCanvas.tsx` `onUpdateNode` merges `newData` into `n.data` without sanitizing keys or enforcing per-node schemas.

**Why it matters:**
- If the build target/transpilation turns object spread into `Object.assign`, `__proto__`-style prototype pollution can reappear in some toolchains.
- Even aside from security, invalid fields/types can enter node data and later be silently skipped by compilation.

**Fix direction:**
- Sanitize `newData` with the same dangerous-key filter and per-field length/type constraints as load-time sanitization (but field-aware).
- Consider per-node-type validators so canvas state remains compile-stable.

---

## MEDIUM

### 1) Connection-rule invariants not enforced for loaded canvas JSON
**Where:** `isValidCanvasState()` checks dangling edges but not “valid type-to-type” rules

**Impact:** A saved canvas JSON can include edges that violate `connectionRules` (e.g., `condition -> trigger`), creating surprising compilation results (compiler ignores non-trigger reachable paths) and confusing UI states.

**Fix:** Validate edges against `isValidConnection(sourceType,targetType)` during `isValidCanvasState()` (and/or drop invalid edges during sanitization).

---

### 2) Empty matcher condition compiles as “no matcher” (semantic mismatch risk)
**Where:** `compileCanvasToHooks()` only sets `group.matcher` if `condData.matcher` is truthy

**Impact:** A condition node present with an empty matcher becomes a group with **no matcher** (often interpreted as “match all”), which may not match user intent (“no matcher” vs “match nothing” vs invalid).

**Fix:** Decide explicit semantics:
- treat empty matcher as invalid and skip group,
- or treat it as `.*`,
- or disallow saving/connecting until matcher is non-empty.

---

### 3) Ordering is implicit and potentially unstable
**Where:** `compileCanvasToHooks()` derives hook order from adjacency list insertion order (edge array order), not from layout or an explicit order field.

**Impact:** Users may expect vertical order (y-position) to define execution order; reloads / edge recreation may change order.

**Fix:** Sort outgoing action targets by `node.position.y` (or persist an explicit `order` on edges).

---

### 4) Canvas representation changes grouping for “no matcher” groups
**Where:** `compileCanvasToHooks()` emits **one group per action** for `trigger -> action` (no condition), even if the original JSON had multiple hooks in a single group.

**Impact:** Often equivalent today, but it is a semantic transformation and could diverge if group-level options appear or if execution semantics differ by grouping.

**Fix:** Represent “no matcher” groups as a dedicated “group” node or compile multiple trigger-connected actions into a single group (optionally ordered by y-position).

---

## LOW

### 1) NodePalette icons passed as HTML entity strings won’t render as intended
**Where:** `NodePalette.tsx` uses `icon="&#9889;"` etc, but renders `{icon}` as text.

**Fix:** Pass actual characters (`"⚡"`, `"⚙"`) or render via `dangerouslySetInnerHTML` (not recommended) or map to components.

---

### 2) Duplicated/fragmented dangerous-key logic
**Where:** multiple `DANGEROUS_KEYS` sets (global + nested inside `isValidCanvasState`)

**Fix:** Centralize and consistently apply (load, update, compile).

---

### 3) Minor API robustness
**Where:** `crypto.randomUUID()` may be unavailable in some environments.

**Fix:** Fallback to a small UUID polyfill or React Flow’s ID helpers.
