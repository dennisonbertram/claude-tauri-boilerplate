## Audit: Visual Hook Builder (React Flow canvas → Claude Code hooks JSON)

### Invariants & state model (expected)
- **Graph shape**: `trigger → (condition → action*) | action*`, with **action as leaf**.
- **Single-parent action**: each action should have **≤ 1 inbound edge** (prevents duplicate execution).
- **Serialization**:
  - `hooksCanvasJson` should round-trip nodes/edges safely.
  - `hooksJson` should be derivable deterministically from the graph.
- **Robustness**: malformed `hooksJson` or `hooksCanvasJson` should not crash the UI; should degrade gracefully.

---

## CRITICAL

### 1) TypeScript compilation failure: `React` namespace used without import
**Files:**
- `src/components/agent-builder/panels/NodeConfigPanel.tsx`
  - Uses `React.FC` but does **not** import `React` (type-only is fine).
- `src/components/agent-builder/panels/NodePalette.tsx`
  - Uses `React.ReactNode` but does **not** import `React`.

With the modern JSX runtime (`"jsx": "react-jsx"`), you may omit runtime `import React`, but **the `React` namespace is not globally available for types**. This will typically fail `tsc` with `Cannot find namespace 'React'`.

**Fix** (either approach):
- Add `import type React from 'react';` (or `import type { FC, ReactNode } from 'react';`) and remove `React.` namespace usage, e.g.
  - `Record<string, React.FC<...>>` → `Record<string, FC<...>>`
  - `children: React.ReactNode` → `children: ReactNode`

## HIGH

### 2) `canvasGenerator.buildActionData` is not type-safe for user JSON; can throw and abort generation
`generateCanvasFromHooks` claims: “Gracefully skips malformed groups.”  
In reality, `buildActionData` calls `.slice()` on fields that are **not validated as strings**:

Examples:
- `hook.command.slice(...)` (command label)
- `hook.url.slice(...)` (http label)
- `hook.prompt.slice(...)` (prompt label)
- `hook.description.slice(...)` (agent label)

If a hooks JSON contains `command: 123` or `url: {}`, `buildActionData` will throw; the outer `try/catch` in `generateCanvasFromHooks` returns **null for the entire canvas**, not “skip malformed hook”.

**Impact**
- A single malformed hook entry can prevent canvas view from working at all (hard fail), even if most of the JSON is valid.

**Fix**
- Add runtime guards in `buildActionData` before slicing/using fields:
  - `const command = typeof hook.command === 'string' ? hook.command : ''`
  - same for `url/prompt/description/method/model`
- Prefer “skip just that hook” instead of aborting the full parse.

---

### 3) Loaded `hooksCanvasJson` is not validated against connection rules or the single-parent-action invariant
`isValidCanvasState` checks:
- nodes/edges array shape, node ids/types, positions finite, edges not dangling

But it does **not** enforce:
- connectionRules (`trigger→condition|action`, `condition→action`, etc.)
- **single inbound edge per action**, which you enforce during interactive `onConnect`

**Impact**
- A saved/edited canvas JSON can violate invariants and produce:
  - unexpected compilation output (duplicate hook execution risk if same action reachable multiple ways across events; or unpredictable dropping due to `processedActionIds`)
  - confusing UI state (graph shows edges that would be impossible to create via UI)

**Fix**
- Extend `isValidCanvasState` (or add a normalization step on load) to:
  - reject or drop invalid edges by `checkConnection(source.type, target.type)`
  - enforce `inDegree(action) ≤ 1` (drop extras deterministically)

## MEDIUM

### 4) `HookCanvasInner` ignores prop changes after first mount (stale view risk)
The initialization effect is `useEffect(..., [])` plus `initialized.current` guard. If `hooksJson`/`hooksCanvasJson` change while the canvas is mounted (e.g., profile switched externally), the canvas will **not** reload.

This may be intended to protect local edits, but it creates a correctness hazard unless the component is guaranteed to unmount/remount on profile change.

**Mitigation**
- Key the `<HookCanvas />` by profile id, or
- Add explicit “Reload from JSON” action, or
- React to prop changes when they come from outside (needs source-of-truth design).

---

### 5) NodePalette icons render HTML entities as literal text
In `NodePalette`, `icon="&#9889;"` is passed as a string and rendered as text, not parsed HTML—users will see the entity text, not ⚡/⚙.

**Fix**
- Use the actual characters (`'⚡'`, `'⚙'`) or render via `dangerouslySetInnerHTML` (not recommended) or use an icon component.

---

### 6) `compileCanvasToHooks` is not fully defensive if used outside `HookCanvas`
It assumes `node.data` conforms to `TriggerNodeData/ConditionNodeData/ActionNodeData`. If called with arbitrary `Node[]` (it’s exported library code), `buildHookEntry(node.data as ActionNodeData)` can misbehave if `data` is null/non-object.

**Fix**
- Add guards before reading `data.hookType`, `data.command`, etc.

## LOW

### 7) Type soundness weakened by broad casts
Examples:
- `const newNode: CanvasNode = { ... } as CanvasNode;`
- `... ) as CanvasNode[],`

This can mask real shape issues during refactors (e.g., missing required `data` fields).

**Fix**
- Construct strongly-typed node variants per `nodeType`, or validate `data` into typed structures.

### 8) Minor dead/duplicate code
- `reactFlowWrapper` is declared but not used (no measurements).
- `VALID_NODE_TYPES` duplicated logically with local set inside `isValidCanvasState`.

## Compilation logic correctness (core transformation)
### `compileCanvasToHooks`
- Correctly builds adjacency and walks from **trigger nodes only**, skipping orphaned nodes (good).
- Implements:
  - Trigger→Condition→Actions ⇒ group with optional matcher
  - Trigger→Action ⇒ group without matcher (one action per group)
- Deduping via `processedActionIds` is reasonable given the “single inbound action” invariant, but becomes nondeterministic if malformed graphs are loaded (see HIGH-3).

### `generateCanvasFromHooks`
- Layout strategy is deterministic and edge/node caps are in place.
- Biggest issue is robustness/type checking (HIGH-2).
