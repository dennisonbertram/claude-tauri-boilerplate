## Findings (frontend / canvas → hooks pipeline)

### HIGH: Unbounded JSON size parsing ⇒ app-level DoS / crash on opening a profile
**Where**
- `HooksTab`: `hasDangerousHooks`, `hasUnsupportedHooks`, `jsonValid` all `JSON.parse(draft.hooksJson)` with **no size cap**.
- `HookCanvas`: mount effect does `JSON.parse(hooksCanvasJson)` with **no size cap**.
- `isValidCanvasState()` validates node/edge counts, types, etc., but **does not bound string lengths** inside `node.data` before the parse is already done.

**Exploit**
An attacker who can get a victim to open a profile (or any persisted draft) containing a very large `hooksJson` or `hooksCanvasJson` (e.g., tens/hundreds of MB, or a single node with a massive `prompt` string) can freeze the UI or crash the renderer process during `JSON.parse`. The later `sanitizeNodeData()` slicing is “too late” because allocation already occurred.

**Impact**
- One-click denial of service: opening the Hooks tab (or switching to Canvas view) can hang/crash the app.
- If this is a Tauri desktop app, this can be a full application crash loop if the profile auto-loads.

**Fix**
- Enforce hard maximum lengths for `draft.hooksJson` and `draft.hooksCanvasJson` **before** parsing (e.g., 500KB/1MB).
- Consider parsing in a worker, or incremental validation, and fail closed with a user-visible error.


### HIGH: Per-field limits are bypassed on node edits ⇒ persistent oversized payloads + data bloat in `hooksCanvasJson`
**Where**
- `HookCanvas.tsx`: `onUpdateNode()` merges `newData` into `node.data` without calling `sanitizeNodeData()`.
  ```ts
  n.id === nodeId ? { ...n, data: { ...n.data, ...newData } } : n
  ```
- `notifyChange()` persists `hooksCanvasJson` as:
  ```ts
  JSON.stringify({ nodes: ... data })
  ```
  meaning the **raw, unbounded** `node.data` is stored even if `compileCanvasToHooks()` truncates when generating `hooksJson`.

**Exploit**
A user (or any script running in the renderer context) can paste extremely large strings into:
- command textarea
- prompt textarea
- description textarea

These values can be megabytes+ and will be persisted into `hooksCanvasJson` (and included in `getHooksFingerprint()` comparisons), causing:
- runaway memory usage
- slow renders / lockups
- huge profile updates (storage exhaustion / sync issues)

**Impact**
- Persistent DoS: once saved, simply re-opening canvas will parse the huge canvas JSON again.
- Data integrity mismatch: `hooksJson` is truncated but `hooksCanvasJson` is not, creating a “hidden payload” channel inside the canvas state.

**Fix**
- Apply `sanitizeNodeData()` (and dangerous-key stripping) inside `onUpdateNode()` before merging.
- Add UI-level `maxLength` / enforced truncation at input time.
- Consider capping total serialized canvas size before saving.

### MEDIUM: Dangerous-hook warning/confirmation bypass via substring heuristic
**Where**
- `HookCanvas.tsx` dangerous-load gate:
  ```ts
  const hooksAlreadyDangerous =
    hooksJson && (hooksJson.includes('"type":"command"') || hooksJson.includes('"type":"http"'));
  if (hasDangerous && !hooksAlreadyDangerous) { confirm(...) }
  ```

**Exploit**
An attacker can make `hooksAlreadyDangerous === true` without actually having dangerous hooks in `hooksJson` by embedding the substring `"type":"command"` inside an unrelated string field (e.g., a prompt/description). That suppresses the confirmation even when `hooksCanvasJson` contains real command/http action nodes.

**Impact**
- Safety control bypass: dangerous canvas state can be loaded without the intended “load anyway?” friction.
- Increases chance a user unknowingly edits/saves and ends up compiling/saving real command/http hooks.

**Fix**
- Replace substring checks with semantic detection (parse and inspect) like the safer logic already present in `HooksTab.hasDangerousHooks`.
- Or: always confirm when `hooksCanvasJson` contains dangerous actions, regardless of `hooksJson`.


### MEDIUM: Prototype-pollution hardening gaps in header handling (poisoning risk + future-footgun)
**Where**
- `canvasGenerator.buildActionData()` accepts `hook.headers` as-is (no dangerous key stripping / no CRLF enforcement / no size caps):
  ```ts
  headers: hook.headers && typeof hook.headers === 'object' ? hook.headers : undefined
  ```
- `canvasCompiler.buildHookEntry()` sanitizes CRLF/length but does **not** exclude `DANGEROUS_KEYS` for header keys and uses `{}` (not `Object.create(null)`):
  ```ts
  const sanitizedHeaders: Record<string, string> = {};
  sanitizedHeaders[k] = v;
  ```

**Exploit**
Today, values must be strings to round-trip into output headers, which reduces classic `__proto__` exploitation. But the code still permits “poison” keys like `constructor`, `__defineGetter__`, etc. to exist on a plain object and can cause odd behavior in downstream merges/usages (especially if future code spreads/assigns headers).

**Impact**
- Not a clear RCE today in this snippet, but it’s a fragile edge that commonly becomes a real prototype pollution issue after refactors.

**Fix**
- Strip `DANGEROUS_KEYS` for headers in **both** generator and compiler.
- Use `Object.create(null)` for header maps you construct.


### LOW: Drag-and-drop `dataTransfer` JSON has no size limit (minor DoS vector)
**Where**
- `HookCanvas.onDrop`: `JSON.parse(raw)` with no length bound.

**Impact**
Usually limited by browser drag data mechanics, but still a small “hang the UI” vector.

**Fix**
Reject `raw.length` above a small threshold before parsing.


### LOW: Import error handling is silent (weak feedback, can mask malicious/invalid inputs)
**Where**
- `HooksTab.handleImport` catches parse errors and “silently ignore”.

**Impact**
Not a direct exploit, but reduces user ability to notice unexpected behavior when importing.

**Fix**
Show a non-blocking error toast/message for invalid JSON.
