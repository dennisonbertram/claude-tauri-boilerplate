## Findings (adversarial security review)

### 1) HIGH ŌĆö Cross-origin drag-and-drop payload injection + `__proto__` prototype mutation in `sanitizeNodeData`
**Where**
- `HookCanvas.tsx`: `onDrop()` accepts `application/reactflow` from *any* drag source.
- `sanitizeNodeData()` writes attacker-controlled keys into a plain object `{}` without filtering `__proto__` / `constructor` / `prototype`.

**Why it matters**
- A malicious webpage can convince a user to drag an element into the app. That page can set `dataTransfer.setData('application/reactflow', ...)` with arbitrary JSON.
- If the JSON includes a `data.__proto__` key with a primitive value, this line in `sanitizeNodeData`:
  ```ts
  result[key] = val.slice(...)
  ```
  will invoke the `__proto__` setter on `result` (because `result` is `{}`), mutating the prototype of `result`. This is a classic ŌĆ£prototype mutationŌĆØ primitive that can destabilize downstream code expecting a normal plain object, and can create hard-to-debug state corruption / crashes in ReactFlow or other utilities.

**Practical PoC payload**
Drag from a malicious site into the canvas:
```json
{
  "nodeType": "action",
  "data": {
    "__proto__": null,
    "hookType": "prompt",
    "prompt": "hi"
  }
}
```
This can create a node whose `data` object has a `null` prototype (or other unexpected prototype), which many libraries do not expect.

**Recommended fix**
- In `onDrop`, reject non-palette drags by using a per-app secret marker (e.g., also require a second mime type like `application/x-yourapp-signed`), or check `event.dataTransfer.types` more strictly.
- Apply `stripDangerousKeys()` **before** `sanitizeNodeData()`, and/or make `sanitizeNodeData()` write into `Object.create(null)` and explicitly skip dangerous keys:
  ```ts
  const result = Object.create(null);
  if (DANGEROUS.has(key)) continue;
  result[key] = ...
  ```

### 2) HIGH ŌĆö Untrusted `hooksCanvasJson` deserialization gaps (edge id keys, nested objects, coordinate ranges) ŌåÆ potential ReactFlow internal pollution + persistent canvas DoS
**Where**
- `HookCanvas.tsx`:
  - `hooksCanvasJson` is parsed and then (if `isValidCanvasState`) loaded into ReactFlow state.
  - `isValidCanvasState()` validates some shape but misses important constraints.

**Issues**
1) **Edge IDs are not checked for dangerous keys**
   - `isValidCanvasState()` blocks dangerous **node** IDs (`__proto__`, `constructor`, `prototype`) but does **not** block dangerous **edge** IDs.
   - Many graph libs index edges by `id` using plain objects internally (pattern: `map[id] = edge`). If `id="__proto__"`, this can prototype-mutate that map object inside the library.

2) **Node `data` is only shallow-ŌĆ£dangerous keyŌĆØ stripped on load**
   - `stripDangerousKeys(n.data)` is shallow and does not sanitize nested objects (e.g., `headers` objects).
   - `isValidCanvasState()` also does **not** enforce ŌĆ£flat primitives onlyŌĆØ, despite comments elsewhere implying you want flat data.

3) **Position range is unconstrained**
   - You check `isFinite(pos.x/y)` but allow extreme magnitudes (e.g., `1e308`). `fitView` and minimap calculations in graph libs can overflow or become extremely slow, causing a persistent hang when switching to Canvas view.

**Practical PoC `hooksCanvasJson`**
```json
{
  "nodes": [
    { "id": "trigger-1", "type": "trigger", "position": { "x": 1e308, "y": 1e308 }, "data": { "event": "PreToolUse" } },
    { "id": "action-1", "type": "action", "position": { "x": 0, "y": 0 }, "data": { "hookType": "command", "command": "echo pwn" } }
  ],
  "edges": [
    { "id": "__proto__", "source": "trigger-1", "target": "action-1" }
  ]
}
```

**Recommended fix**
- Extend `isValidCanvasState()` to:
  - Reject dangerous **edge IDs** (`__proto__`, `constructor`, `prototype`, plus getter/setter keys).
  - Enforce coordinate bounds, e.g. `Math.abs(x) <= 100000` (or whatever makes sense).
  - Enforce data ŌĆ£flat primitives onlyŌĆØ on load (reuse `sanitizeNodeData()` + `stripDangerousKeys()`), not only on drag/drop.
- Consider validating loaded edges against `connectionRules` as well (reject triggerŌåÆtrigger, actionŌåÆ*, etc.) to reduce weird graph states fed into ReactFlow.

### 3) MEDIUM ŌĆö Nested `headers` object is unsanitized in generator and can trigger prototype mutation in compiler; also enables memory bloat
**Where**
- `canvasGenerator.ts`:
  - For HTTP hooks, `headers` is passed through directly:
    ```ts
    headers: hook.headers && typeof hook.headers === 'object' ... ? hook.headers : undefined
    ```
- `canvasCompiler.ts`:
  - Headers are copied into `sanitizedHeaders: Record<string, string> = {};`
  - Dangerous keys are **not** excluded (`__proto__`, `constructor`, `prototype`).

**Impact**
- An attacker-controlled hooks JSON can introduce a `headers` object with dangerous keys and/or very large key/value strings.
- In the compiler, assigning `sanitizedHeaders["__proto__"] = "x"` prototype-mutates the headers object (local prototype mutation). Even if it doesnŌĆÖt immediately become ŌĆ£global Object.prototype pollutionŌĆØ, itŌĆÖs still a destabilizing behavior and an unnecessary risk.
- Generator stores headers without any size/entry limits; large headers can bloat `hooksCanvasJson` and freeze UI when serializing fingerprints (`JSON.stringify`) or rendering config panels.

**Recommended fix**
- In generator: deep-sanitize and bound headers (count, key/value length), and store them in `Object.create(null)` to remove magic keys.
- In compiler: build `sanitizedHeaders` with `Object.create(null)` and reject dangerous header names explicitly.

### 4) MEDIUM ŌĆö No size/length enforcement for node data on edit or canvas-json load ŌåÆ stored UI freeze / data corruption
**Where**
- `HookCanvas.tsx`:
  - On load from `hooksCanvasJson`, strings are not truncated.
  - `onUpdateNode()` merges `newData` directly (no `sanitizeNodeData()`).
- `NodeConfigPanel.tsx`: user can paste extremely large strings into textareas.

**Impact**
- A crafted profile (or user paste) can store multi-megabyte strings into `hooksCanvasJson`. This can:
  - Freeze the renderer (textarea value binding, ReactFlow updates).
  - Inflate stored profile size significantly.
  - Cause sluggishness in `getHooksFingerprint()` (stringifies full node data every structural change).

**Recommended fix**
- Apply `sanitizeNodeData()` (with max lengths) inside `onUpdateNode()` and during canvas-json load, not just DnD.
- Consider a global cap for `hooksCanvasJson` length before parsing.

### 5) MEDIUM ŌĆö Dangerous hook confirmation can be bypassed via pre-seeded canvas state (no confirmation on compile)
**Where**
- `HookCanvas.tsx` warns/confirm only during **drag-drop** creation of command/http nodes.
- If a profile arrives with dangerous nodes already present in `hooksCanvasJson` (or in `hooksJson`), editing/connecting nodes compiles to command/http hooks without an explicit confirm at that moment.

**Impact**
- Social-engineering risk: a shared profile can include dangerous action nodes; user enters Canvas view and makes a small edit; compilation produces command/http hooks without the ŌĆ£Add anyway?ŌĆØ confirm (only the passive banner in `HooksTab`, and that banner can be absent if `hooksJson` is empty but canvas contains nodes).

**Recommended fix**
- On first compile that introduces `command`/`http`, require a confirmation (stateful ŌĆ£acknowledged for this profileŌĆØ flag), not only on DnD.
