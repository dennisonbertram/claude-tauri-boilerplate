## CRITICAL

### 1) Silent data loss for hooks entries missing `type` (no warning, dropped on canvas edit)
**Where:** `HooksTab.hasUnsupportedHooks`, `canvasGenerator.buildActionData`  
**Problem:** Canvas only generates action nodes when `hook.type` is one of `command|http|prompt|agent`. Hooks in JSON that omit `type` (but include `command`, `url`, etc.) are:
- flagged as “dangerous” (sometimes), **but not flagged as “unsupported”**
- **not representable on canvas**
- **silently skipped** by `generateCanvasFromHooks`
- then **permanently removed** from output JSON the moment the user makes any structural canvas change (add/delete/connect/update), because `compileCanvasToHooks` re-emits only what exists on canvas.

**Repro (minimal):**
1. In JSON view paste:
   ```json
   {
     "hooks": {
       "PreToolUse": [{ "hooks": [{ "command": "echo hi" }] }]
     }
   }
   ```
2. Switch to Canvas (no unsupported warning shown because `hook.type` is absent).
3. Add any node or connection.
4. Switch back to JSON: the untyped command hook is gone.

**Impact:** irreversible hook removal without explicit user confirmation; very likely to hit older/handwritten hook JSON.  
**Fix:**
- Treat missing `type` as **unsupported** (block canvas edit until confirmed), OR
- Infer `type` when unambiguous (e.g. `command` if `command` present, `http` if `url` present), and surface a warning.
- Update `hasUnsupportedHooks` to return true for any hook with missing/invalid `type` **or** for “legacy shape” hooks.

## HIGH

### 2) Hook execution order + grouping are not faithfully round-tripped (behavior can change)
**Where:** `canvasGenerator.generateCanvasFromHooks`, `canvasCompiler.compileCanvasToHooks`  
**Problems:**
- **Ordering is implicit and unstable:** compiler uses edge insertion order (`adjacency` built from `edges` array order). Users can change edge creation order unintentionally; node Y-position is ignored.
- **Grouping semantics change:** for Trigger → Action (no condition), compiler emits **one group per action**:
  ```ts
  // "each gets its own group"
  groups.push({ hooks: [hookEntry] });
  ```
  If the source JSON had a single group with multiple hooks (no matcher), it becomes N groups. If the runtime semantics differ (even subtly) between “one group with many hooks” vs “many groups with one hook”, behavior changes.

**Repro:**
1. JSON with one group containing multiple hooks under same event (no matcher).
2. Open canvas, make a change, return to JSON: now multiple groups.
3. Create actions in some order, reconnect edges: output order can flip.

**Impact:** output JSON may execute hooks in different order / different grouping than original; can break workflows.  
**Fix:**
- Persist explicit ordering in canvas state (e.g. `data.order` or edge metadata) and sort deterministically when compiling.
- Prefer ordering by node Y-position if you don’t want explicit ordering UI.
- Preserve “group with multiple hooks and no matcher” as a first-class canvas construct (e.g. a “Group” node) or compile Trigger → multiple Actions into a single group in a deterministic order.

### 3) MAX_NODE_COUNT can be exceeded via rapid drops (non-atomic check → invalid saved state/data loss)
**Where:** `HookCanvas.onDrop`  
**Problem:** Node limit check uses stale closure state:
```ts
if (nodes.length >= MAX_NODE_COUNT) return;
setNodes((nds) => [...nds, newNode]);
```
Dropping multiple nodes quickly can pass the `nodes.length` check multiple times before React commits the first update.

**Impact:**
- Canvas can exceed internal limits (200 nodes).
- You can save a canvas JSON that later fails `isValidCanvasState` and forces regeneration, risking unexpected loss of layout/edges.

**Fix:** enforce limits inside the functional updater:
```ts
setNodes(nds => nds.length >= MAX_NODE_COUNT ? nds : [...nds, newNode]);
```
Also consider hard-trimming/alerting before saving.

---

### 4) HookCanvas ignores prop changes after mount → can overwrite external/parent updates
**Where:** `HookCanvasInner` initialization effect has `[]` deps + `initialized.current` gate  
**Problem:** Once mounted, `hooksJson` / `hooksCanvasJson` prop changes are ignored. If parent updates the draft while canvas is open (undo/redo, remote load, switching profiles without changing key, programmatic edits), the next debounced save can push stale canvas-derived JSON back up via `onChange`.

**Impact:** last-writer-wins overwrites parent state; confusing “my edits reverted” or “canvas clobbered JSON”.  
**Fix:**
- Add a controlled sync strategy: watch `hooksJson`/`hooksCanvasJson` changes and either reinitialize (with confirmation if dirty) or block external updates while editing.
- Track a `dirty` flag and prompt on incoming prop changes.

## MEDIUM

### 5) Unsupported-hooks confirmation doesn’t reset when JSON changes
**Where:** `HooksTab` (`unsupportedConfirmed` reset only on `[profileId]`)  
**Problem:** User can confirm once, then edit JSON to add unsupported hooks; canvas view won’t re-prompt.

**Impact:** easier to accidentally drop unsupported hooks later.  
**Fix:** reset `unsupportedConfirmed` when `draft.hooksJson` changes from “supported” → “unsupported”, or always reset on any `hooksJson` change.

---

### 6) Dangerous/unsupported detection can be bypassed by malformed `group.hooks`
**Where:** `hasDangerousHooks`, `hasUnsupportedHooks`, `hasExecutionHooks`  
**Problem:** Some loops do:
```ts
if (!group?.hooks) continue;
for (const hook of group.hooks) { ... }
```
If `group.hooks` is a truthy non-iterable object, the `for..of` throws, is caught by the outer try/catch, and the function returns `false` (no warning).

**Impact:** warnings can silently disappear on malformed JSON; increases risk of unintentional data loss on canvas edit.  
**Fix:** always guard with `Array.isArray(group.hooks)` before iterating (as you already do in other places).

---

### 7) Trigger event dropdown can’t represent unknown events → editing clobbers them
**Where:** `NodeConfigPanel.TriggerConfig`  
**Problem:** generator preserves unknown event keys as trigger node `data.event = event`, but the `<select>` only contains `HOOK_EVENTS`. Unknown values won’t be selectable/visible; any change overwrites with a supported event.

**Impact:** unknown/custom events are effectively uneditable without loss.  
**Fix:** include current value as an extra `<option>` when it’s not in `HOOK_EVENTS`, or switch to a combobox/free-text input with validation.

---

### 8) Canvas can save inconsistent `hooksJson` vs `hooksCanvasJson` (layout-only saves keep old hooksJson)
**Where:** `saveCanvas` uses `structuralDirtyRef` to decide whether to compile  
**Problem:** If canvas is generated from `hooksJson` containing constructs the canvas cannot represent (unsupported hooks), layout-only changes save a new `hooksCanvasJson` (missing those hooks) while keeping old `hooksJson`. Future loads prioritize `hooksCanvasJson`, showing a canvas that doesn’t match the JSON (until a structural edit compiles and drops them).

**Impact:** confusing divergence; “canvas doesn’t match JSON”, hard to reason about what will be preserved.  
**Fix:** when representation is lossy, either:
- avoid writing `hooksCanvasJson` until user confirms lossy conversion, or
- store metadata in `hooksCanvasJson` that it’s derived/partial and shouldn’t be prioritized over `hooksJson` unless confirmed.

---

### 9) Debounced save can call stale `onChange` if parent recreates callback
**Where:** `saveCanvas` is recreated when `onChange` identity changes; old timer isn’t canceled  
**Impact:** out-of-order updates, especially during fast view toggles/state changes.  
**Fix:** keep a `onChangeRef` and use a stable debounced function, cancel old debounced on replacement.

**Impact:** minor UI regression; confusing affordances.  
**Fix:** use actual unicode characters (`'⚡'`, `'⚙'`) or render entities via `dangerouslySetInnerHTML` (prefer unicode).
