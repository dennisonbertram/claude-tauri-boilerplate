## CRITICAL

### 1) Hooks JSON can revert / data loss due to debounced `layoutSave` writing stale `hooksJson`
**Where:** `HookCanvas.tsx` (`notifyChange` 500ms + `layoutSave` 2000ms)

**What happens:**
- Any structural change (add node, connect edge, edit node config) schedules `notifyChange` to compile new hooks JSON.
- Independently, the “layout-only save” effect schedules `layoutSave(nodes, edges, hooksJson)` and passes the **current prop** `hooksJson` at scheduling time.
- If `hooksJson` prop is still the *old* value when scheduled (very common), when `layoutSave` fires later it calls:
  - `onChange(oldHooksJson, newCanvasJson)`
- This can overwrite the freshly compiled hooks JSON and effectively “undo” your edits after ~2 seconds.

**Why it’s especially bad:**
- The “position-only” detection is flawed: `layoutSave` compares `fingerprint === prevStructuralRef.current`, but `prevStructuralRef.current` is updated in the other effect *during the same commit*, so many structural changes are misclassified as “position-only” and still schedule `layoutSave`.

**Repro (likely):**
1. Start with empty hooks.
2. Switch to Canvas.
3. Drag in Trigger + Action, connect them, fill required Action field.
4. Wait >2s without touching anything.
5. Observe hooks JSON can revert to previous/empty value (depending on parent state timing).

**Fix ideas:**
- Never call `onChange` with a potentially stale `hooksJson` prop.
  - Either: `layoutSave` should only update `hooksCanvasJson` (separate callback / separate state update path).
  - Or: compute hooks JSON from current nodes/edges at fire time (or store “latest compiled hooks JSON” in a ref updated by `notifyChange` and use that).
- Correct “position-only” detection by tracking previous fingerprint separately (don’t update the comparator before the layout effect runs).
- Cancel pending `layoutSave` whenever a structural change is detected / whenever `notifyChange` fires.

### 2) Prototype pollution risk via untrusted `hooksCanvasJson` (and object spread merges)
**Where:** `HookCanvas.tsx` (`isValidCanvasState` + `setNodes(saved.nodes)` + later merges in `onUpdateNode`)

**What happens:**
- `isValidCanvasState` checks basic shape but allows arbitrary keys inside `node.data`.
- A crafted `hooksCanvasJson` can include `node.data` like `{ "__proto__": { polluted: true } }`.
- Later code does `{ ...n.data, ...newData }` which in JS can mutate the created object’s prototype (classic prototype pollution vector).

**Impact:**
- Potentially breaks app logic in hard-to-debug ways, or worse if any sensitive logic depends on object properties.
- This is especially relevant if profiles/canvas JSON can be imported/shared.

**Fix ideas:**
- Deep-sanitize loaded canvas state (strip `__proto__`, `constructor`, `prototype`, etc.) from **all** objects, especially `node.data`.
- Consider creating `data` objects with `Object.create(null)` when normalizing.
- Avoid spreading untrusted objects directly.

## HIGH

### 3) “Unsupported hook types will be dropped” warning is not a safe guard; canvas edits can destroy data
**Where:** `HooksTab.tsx` warnings + `generateCanvasFromHooks` dropping unsupported hook types + `compileCanvasToHooks` only emits supported types

**What happens:**
- The warning exists, but switching to Canvas and making *any* change will regenerate/save hooks JSON **without** those unsupported hooks.
- There is no “read-only” canvas mode, no confirmation gate on first canvas edit, and no merge strategy to preserve unknown hooks.

**Fix ideas:**
- Block Canvas editing when unsupported hooks exist unless user explicitly confirms “Canvas edits will remove unsupported hooks”.
- Preserve unsupported hooks in a passthrough blob and merge them back on compile (best), or keep hooks JSON authoritative and only store layout separately.

---

### 4) Unknown hook events are not editable safely in Canvas (can cause accidental event change / loss)
**Where:** `NodeConfigPanel.tsx` Trigger select uses `HOOK_EVENTS` only

**What happens:**
- `compileCanvasToHooks` claims it “preserves unknown events”.
- But if you load JSON containing an unknown event key, the Trigger config `<select value={data.event}>` has no matching option.
- UX outcome varies by browser/React behavior (blank selection / forced first option), and any user interaction can silently switch the event to a supported one, losing the original unknown event.

**Fix ideas:**
- Allow custom event string entry (combo box) when event isn’t in `HOOK_EVENTS`.
- At least render an `<option value={data.event}>Custom: {data.event}</option>` when unknown.

---

### 5) Invalid connection attempts fail silently (confusing UX; users think edges “should work”)
**Where:** `HookCanvas.tsx` `onConnect` returns without user feedback

**What happens:**
- Disallowed connections (e.g., action → anything, or invalid combinations) just do nothing.
- There’s a `getInvalidConnectionReason()` helper in `connectionRules.ts` but it’s unused.

**Fix ideas:**
- Use ReactFlow `isValidConnection` prop for real-time affordance + show toast/tool-tip with `getInvalidConnectionReason`.
- Provide visual “shake”/error when connection rejected.

---

### 6) Hooks can be silently skipped in compile if required fields empty (no indication of “this won’t run”)
**Where:** `canvasCompiler.ts` `buildHookEntry()` returns null unless command/url/prompt/description is non-empty

**What happens:**
- Canvas allows creating action nodes with empty command/url/etc.
- Compiler drops them silently, so user can believe the hook exists but it won’t be in output.

**Fix ideas:**
- Mark invalid/incomplete nodes in the canvas UI (badge/error state).
- Prevent connecting trigger/condition to invalid action nodes, or show compile-time warnings in UI.

## MEDIUM

### 7) JSON import failures are silently ignored
**Where:** `HooksTab.tsx` `handleImport` catch block

**What happens:**
- If JSON is invalid, nothing happens and user gets no feedback.

**Fix ideas:**
- Show an error message with parsing failure.
- Also consider validating shape (`hooks` object) and warning if missing.

---

### 8) Performance risks: frequent `JSON.stringify` on potentially large graphs during drag/edit
**Where:** `HookCanvas.tsx`
- `getHooksFingerprint()` stringifies nodes+edges (data-heavy)
- layout effect stringifies full `{nodes, edges}`

**What happens:**
- With large graphs and large prompts, dragging/typing can cause noticeable lag and delayed input.

**Fix ideas:**
- Use cheaper hashing (incrementing version counters, structural diffing, or only stringify minimal fields).
- Avoid stringifying full node data on every minor change.

---

### 9) Canvas does not respond to external `hooksJson` changes while mounted
**Where:** `HookCanvas.tsx` load effect has empty deps; uses `initialized` ref

**What happens:**
- If parent updates `draft.hooksJson` while user is in Canvas view (autosave, remote update, switching profiles without changing `profileId`, etc.), canvas doesn’t reload or warn about conflicts.

**Fix ideas:**
- Add a “source-of-truth changed” detection and prompt to reload/merge.
- Or remount canvas when hooksJson changes meaningfully (careful with local edits).

---

### 10) Saved canvas state validation does not enforce edge type rules
**Where:** `HookCanvas.tsx` `isValidCanvasState`

**What happens:**
- A saved canvas JSON can contain edges like action→trigger, condition→condition, etc.
- UI/compile behavior becomes unpredictable (skips, partial compile).

**Fix ideas:**
- Validate edges against `connectionRules` during load and drop invalid edges (with warning).

## LOW

### 11) Palette icons for Trigger/Condition likely render as literal `&#9889;` / `&#9881;`
**Where:** `NodePalette.tsx` passes `icon="&#9889;"` as a string and renders `{icon}`

**What happens:**
- React will not interpret that as an HTML entity; user may see the literal text instead of ⚡ / ⚙.

**Fix ideas:**
- Use actual unicode characters (`"⚡"`, `"⚙"`) or render via `dangerouslySetInnerHTML` (not recommended) or an icon component.
