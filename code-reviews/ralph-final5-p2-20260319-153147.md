## CRITICAL

### 1) Silent, *unwarned* data loss for “extra” fields on otherwise supported hooks/groups
**Where:** `HooksTab.hasUnsupportedHooks` vs `compileCanvasToHooks` / `generateCanvasFromHooks`

- `hasUnsupportedHooks` only warns if:
  - root has keys besides `hooks`, or
  - a hook has an unsupported `type`, or missing `type`.
- But the compiler/generator only preserves a **small allowlist** of fields:
  - `command`: `command`, `timeout`
  - `http`: `url`, `method`, `headers`, `timeout`
  - `prompt`: `prompt`, `model`
  - `agent`: `description`
- Any additional fields inside hook entries or groups (e.g. `body`, `query`, `retries`, `env`, `cwd`, `onError`, `name`, custom metadata) will be **dropped** after any structural canvas save, **with no warning** because `hasUnsupportedHooks` won’t flag them.

**Repro:** Start with JSON:
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "http", "url": "https://x", "method": "POST", "body": {"a":1} }
        ]
      }
    ]
  }
}
```
No unsupported warning (type is supported, no extra root keys). Switch to canvas, make any structural edit (add/remove node/edge). Return to JSON: `body` is gone.

**Impact:** irreversible config loss + behavioral change while user was not warned.
**Fix ideas:** detect unknown keys at (a) group level and (b) hook-entry level and force the same “will drop data” confirmation; or round-trip by storing opaque `extra` blobs per hook and merging back on compile.

---

### 2) Hook execution order is not reliably represented; compilation order depends on edge array/insertion order (not visual order)
**Where:** `compileCanvasToHooks` adjacency iteration order; no ordering model in canvas

- In Claude hooks JSON, order inside `hooks: []` and order of groups is meaningful.
- In canvas, there’s no explicit ordering UI/metadata.
- `compileCanvasToHooks` emits hooks in the order of:
  - `adjacency.get(trigger.id)` targets (depends on `edges` array order),
  - then `adjacency.get(condition.id)` targets (again `edges` array order).
- That order is not guaranteed to match:
  - vertical layout (node `position.y`),
  - the original JSON order after load,
  - or user expectation (“top-to-bottom executes top-to-bottom”).
- Minor graph edits (delete + reconnect) can silently reshuffle the serialized hook order and therefore runtime behavior.

**Impact:** behavioral changes without user intent; can break multi-step command sequences.
**Fix ideas:** define deterministic ordering (e.g., sort outgoing action targets by `position.y`, tie-break by id) and/or add an explicit “order” field stored in node data; show order numbers in UI.

## HIGH

### 1) Unsupported-hooks warning can be bypassed (false-negative) due to unsafe iteration over `group.hooks`
**Where:** `HooksTab.hasUnsupportedHooks`

```ts
if (!group?.hooks) continue;
for (const hook of group.hooks) { ... }
```
If `group.hooks` is truthy but not iterable (e.g., object), `for..of` throws, the outer `try/catch` returns `false`, and the user sees **no warning**. Then canvas generation treats `hooks` as empty (it uses `Array.isArray`), and a subsequent save can overwrite the JSON with `{ "hooks": {} }` or otherwise drop content.

**Fix:** mirror `hasDangerousHooks` logic: `const hookList = Array.isArray(group.hooks) ? group.hooks : [];` and continue safely; treat malformed structures as “unsupported” (warn).

---

### 2) HookCanvas ignores subsequent `hooksJson` / `hooksCanvasJson` prop changes (stale state → overwrites newer edits)
**Where:** `HookCanvasInner` load effect has `[]` deps and `initialized` ref

If the parent updates `draft.hooksJson` or `draft.hooksCanvasJson` while the canvas is mounted (import, remote refresh, switching drafts without changing `profileId`, etc.), HookCanvas will **not** reinitialize. A later debounced save from any canvas interaction will emit the stale internal state back via `onChange`, overwriting newer parent state.

**Fix:** rehydrate when inputs change (with safeguards), or at least detect mismatch and prompt (“external changes detected; reload canvas?”).

---

### 3) Auto-save-on-mount / auto-save-on-load can mutate draft without an explicit “Apply” action
**Where:** nodes/edges change effect + debounced `saveCanvas`

On initial load (from `hooksCanvasJson` or generated from hooks), `setNodes/setEdges` triggers the save effect after first render and calls `onChange`, which can:
- populate/overwrite `hooksCanvasJson` even if user “just opened canvas”
- and (after any structural change) rewrite `hooksJson` via compiler (dropping unsupported data as above)

This is especially risky combined with the incomplete unsupported-data detection: user can lose fields immediately after a trivial structural tweak, with no deliberate “export/apply”.

**Fix:** add a “dirty since user action” gate; don’t call `onChange` until the user actually edits (or add an explicit Apply/Save inside canvas mode).

---

### 4) Canvas compile/generate is not structurally faithful: groups-without-matcher get split into separate groups
**Where:** `canvasGenerator` vs `canvasCompiler`

- JSON can contain a group without matcher that has multiple hooks: `{ hooks: [a,b,c] }`.
- Generator creates multiple action nodes connected directly to trigger.
- Compiler emits **one group per direct action** (each with a single hook), not one group containing multiple hooks.

Even if runtime semantics are similar, this creates big diffs and can change ordering/grouping assumptions downstream.

**Fix:** when compiling direct trigger→actions, aggregate into a single group per trigger (or preserve original grouping info as metadata).

## MEDIUM

### 1) Timeout parsing can “stick” old values and be hard to clear due to NaN handling
**Where:** `NodeConfigPanel` + `sanitizeNodeData`

`onChange` does `Number(e.target.value)` when value is truthy; certain invalid inputs can produce `NaN`. `sanitizeNodeData` drops non-finite numbers, so the update silently does nothing and the previous timeout remains.

**Fix:** explicitly treat `NaN` as `undefined` in the panel before calling `onUpdate`.

---

### 2) Unknown trigger events can’t be represented in the Trigger config UI (select has only `HOOK_EVENTS`)
**Where:** `TriggerConfig` `<select>`

If hooks JSON contains a nonstandard event key (compiler even says “preserving in output”), the canvas will create a trigger node with that event, but the config select cannot display/select it properly (no option). This invites accidental overwrites.

**Fix:** include current value as a fallback option (e.g., “Custom: X”), or render as editable text when not in `HOOK_EVENTS`.

---

### 3) Edge pruning on load can silently drop edges (including extra inbound edges to actions) with no user feedback
**Where:** `pruneInvalidEdges`

If a saved canvas state has multiple inbound edges to an action, pruning keeps the “first” one based on edge array order. Users will see connections disappear without explanation.

**Fix:** surface a warning/toast listing how many edges were dropped and why; choose a deterministic edge to keep (e.g., closest source by y).

---

### 4) Silent truncation of long fields (prompt/command/url/etc.) with no UI indication
**Where:** `sanitizeNodeData`

Edits exceeding limits are silently sliced. This can cause confusing “why did my prompt get cut” behavior.

**Fix:** show remaining character count and/or block additional input instead of truncating invisibly.

---

### 5) Import invalid JSON fails silently
**Where:** `HooksTab.handleImport` catch block

Invalid JSON file is “silently ignore” → user thinks import worked.

**Fix:** show an error message/toast.

---

## LOW

### 1) NodePalette icons for triggers/condition likely render as literal entity text
**Where:** `NodePalette` passes `icon="&#9889;"` and renders `{icon}`

React won’t decode that string as an HTML entity; you’ll see `&#9889;` instead of ⚡ (same for `&#9881;`).

**Fix:** pass actual characters (`'⚡'`, `'⚙'`) or render via `dangerouslySetInnerHTML` (not recommended).

---

### 2) Unused `reactFlowWrapper` ref
**Where:** `HookCanvasInner`

It’s created but never used.

---

### 3) `debounce.flush()` implementation is unsafe if used later
**Where:** local `debounce` helper

If `lastArgs` is null, it calls `fn()` even if `fn` expects args (typed cast). Not used today, but a footgun.

---

### 4) Comment drift / missing invalid-connection UX
**Where:** `connectionRules.ts` comment mentions `isValidConnection prop`, but canvas uses `onConnect` validation only

This means users get no reason message (even though `getInvalidConnectionReason` exists), just “nothing happens”.

**Fix:** wire `isValidConnection` + show reason tooltip/toast.
