## CRITICAL

### 1) **Edge ÔÇťselect/deselectÔÇŁ triggers structural recompilation Ôćĺ unexpected hooks JSON rewrite / data loss**
**Where:** `HookCanvas.tsx` Ôćĺ `handleEdgesChange`  
```ts

const handleEdgesChange = useCallback((changes) => {
  if (changes.length > 0) structuralDirtyRef.current = true;
  onEdgesChange(changes);
}, ...)
```
**Why it breaks:** In XYFlow, `onEdgesChange` fires for non-structural changes like `select`. Your code marks *any* edge change as structural, so merely clicking an edge can trigger `compileCanvasToHooks()` on the next debounced save, overwriting `hooksJson`.

**Impact:** Users can lose information even when they didnÔÇÖt ÔÇťeditÔÇŁ anything (see CRITICAL #2). Also creates unexpected diffs/reformatting.

**Repro:** Open Canvas view Ôćĺ click an edge (select) Ôćĺ wait 500ms Ôćĺ parent `hooksJson` may get replaced by compiled output.

**Fix:** Only set `structuralDirtyRef` for `add/remove/replace` (and maybe `reset`), like you already do for nodes:
```ts
const hasStructural = changes.some(c => c.type === 'add' || c.type === 'remove' || c.type === 'replace');
if (hasStructural) structuralDirtyRef.current = true;
```
Also consider: only compile when you truly need to (graph connectivity or action/condition/trigger semantic fields changed), not on UI state changes.

### 2) **Canvas compile output drops non-canvas-preserved JSON fields with no explicit ÔÇťwill be removedÔÇŁ confirmation**
**Where:** `canvasCompiler.ts` always emits:
```ts
return JSON.stringify({ hooks: ... }, null, 2);
```
**Why it breaks:** Any time `structuralDirtyRef` is true (and currently it can become true from trivial actions), you replace the entire user-provided JSON with `{ hooks: ... }`. This destroys:
- root-level fields beyond `hooks`
- any per-group/per-hook fields not modeled in the canvas (even for supported `type`)
- unknown-but-important metadata users might rely on (e.g. `enabled`, `name`, `id`, `retries`, `when`, `env`, etc.)

You *do* warn about unsupported hook **types**, but you do **not** warn about **unsupported fields** that will be dropped.

**Impact:** Irreversible data loss on first compile-triggering interaction.

**Fix options:**
- Add a one-time explicit confirmation: ÔÇťCanvas edits will rewrite hooks JSON and drop unknown fields (not just unsupported types). Continue?ÔÇŁ
- Preserve ÔÇťextraÔÇŁ JSON by parsing the existing JSON, updating only `.hooks` and re-stringifying with the original top-level structure (and ideally preserving unknown per-hook fields via round-trip mapping IDsÔÇöhard but doable).
- At minimum: detect presence of extra fields and show the same gating UX you have for unsupported hook types.

## HIGH

### 3) **Potentially huge `hooksCanvasJson` payload (prompts up to 50k * nodes up to 200) Ôćĺ storage/IPC/perf failures**
**Where:** `HookCanvas.tsx` Ôćĺ `saveCanvas` JSON.stringify of full node data including prompts  
You allow:
- `MAX_PROMPT_LENGTH = 50_000`
- `MAX_NODE_COUNT = 200`

Worst case canvas JSON size can reach many MB. ThereÔÇÖs no cap/check on the serialized canvas payload or on the compiled hooks JSON size. This can:
- freeze the UI during stringify (every save)
- exceed storage limits / DB column limits / Tauri IPC message limits
- cause silent failures or partial writes

**Fix:** Add a maximum serialized size and either:
- refuse to save with a clear error
- compress prompts (or store prompts outside canvasJson)
- lower limits, or make limits storage-aware
