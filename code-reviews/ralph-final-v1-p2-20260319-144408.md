## CRITICAL
None found in this slice (no obvious RCE/XSS or unavoidable data wipe on mere view-switch).

## HIGH

1) **Canvas cannot preserve “no-matcher” hook groups (group boundaries are lost)**
- **Where:** `generateCanvasFromHooks()` (no-matcher branch) + `compileCanvasToHooks()` (Trigger → Action direct).
- **What happens:** A JSON group like:
  ```json
  { "hooks": { "PreToolUse": [ { "hooks": [A, B] } ] } }
  ```
  becomes two separate groups on compile:
  ```json
  { "hooks": { "PreToolUse": [ { "hooks": [A] }, { "hooks": [B] } ] } }
  ```
- **Impact:** Structural drift + potential semantics change if Claude Code treats group boundaries differently now or in future (group-level options, error handling, ordering guarantees, etc.). Users are not warned about this class of loss.
- **Fix:** Introduce an explicit “group/sequence” node for no-matcher groups, or encode group identity in canvas JSON and have compiler reconstitute it.

2) **Hook execution order is effectively nondeterministic**
- **Where:** `compileCanvasToHooks()` uses edge insertion order (`edges` array → adjacency list) and iterates `connectedIds` / `actionIds` without sorting.
- **What happens:** Reordering can occur due to edge creation timing, internal ReactFlow edge ordering, or edge duplication. The canvas provides a strong visual “top-to-bottom” ordering expectation, but compilation ignores node `position.y`.
- **Impact:** Hooks may run in an unexpected order after edits (order is often semantically important for hooks).
- **Fix:** Sort:
  - trigger-connected condition/action nodes by `position.y`
  - action nodes under a condition by `position.y`
  - and/or persist an explicit order field in node data.

3) **Duplicate edges / edge ID collisions are possible (can break graph + duplicate compiled groups)**
- **Where:** `onConnect` uses `addEdge(params, eds)` without:
  - preventing duplicate source→target edges, and
  - ensuring unique edge IDs in all cases.
- **What happens:** Users can connect Trigger→Condition multiple times; depending on ReactFlow’s edge-id generation, this can create duplicate edges or key collisions. Compiler then may process the same condition multiple times, duplicating groups in output.
- **Impact:** UI weirdness (edges flicker/overwrite), duplicated hook groups, confusing “why did my hooks duplicate?” behavior.
- **Fix:** Before adding an edge, reject if an existing edge has same `{source,target,sourceHandle,targetHandle}`; generate edge IDs with `crypto.randomUUID()` (with fallback) instead of relying on library defaults.

4) **Loaded `hooksCanvasJson` is not validated against connection rules / single-inbound constraints**
- **Where:** `isValidCanvasState()` only checks node existence and primitive types; it does **not** enforce `trigger→condition/action`, `condition→action`, “trigger cannot be a target”, or “action has max 1 inbound edge”.
- **Impact:** A persisted/hand-edited canvas JSON can load an invalid graph that:
  - renders misleadingly,
  - compiles to missing/duplicated hooks,
  - bypasses the UI’s connect-time constraints (e.g., multiple inbound edges to an action).
- **Fix:** On load, filter/normalize edges using `checkConnection(sourceType,targetType)` and enforce “max 1 inbound to action (and probably condition too, if intended)”.

5) **Unsupported-hook warning misses typeless hooks that will still be dropped by Canvas**
- **Where:** `HooksTab.hasUnsupportedHooks` only flags `hook?.type && !SUPPORTED_TYPES.has(hook.type)`.
- **What happens:** Hooks that omit `type` (but contain fields) are not considered “unsupported”, yet `generateCanvasFromHooks()` drops them because `buildActionData()` requires a valid `type`.
- **Impact:** User enters Canvas view with **no warning**, makes any structural edit, and those hooks can be permanently lost.
- **Fix:** Treat missing/invalid `type` as unsupported for the warning gate (and ideally display which entries will be dropped).

6) **Condition nodes can be shared across multiple triggers (compiles duplicated groups across events)**
- **Where:** UI allows multiple inbound edges to `condition` nodes; compiler will compile that condition’s actions once per trigger that points at it.
- **Impact:** Unintended duplication across events; hard to notice until runtime.
- **Fix:** Decide intent:
  - If forbidden: block multiple inbound edges to `condition` (like you do for `action`), and/or enforce at load.
  - If allowed: make it explicit in UI (badge/warning), and ensure ordering/grouping is stable.

## MEDIUM

1) **Node cap can be exceeded via rapid drops (stale closure on `nodes.length`)**
- **Where:** `onDrop` checks `if (nodes.length >= MAX_NODE_COUNT)` using captured `nodes`.
- **Impact:** Multiple quick drops before state updates can push beyond 200 nodes → performance + huge saves.
- **Fix:** Use `latestNodesRef.current.length` or enforce inside functional `setNodes(nds => ...)` with a hard cap.

2) **`generateCanvasFromHooks()` injects headers into canvas state without sanitization**
- **Where:** `buildActionData()` passes `hook.headers` through as-is.
- **Impact:** Even though `compileCanvasToHooks()` sanitizes headers later, the **saved canvas JSON** (`hooksCanvasJson`) can store very large/unexpected header shapes (DoS/bloat) and CRLF-containing strings until a compile occurs.
- **Fix:** Sanitize headers in `buildActionData()` (or run `sanitizeNodeData()` over all generated node data before `setNodes`).

3) **Dangerous-hook confirmation logic is inconsistent (string includes)**
- **Where:** `HookCanvasInner` uses:
  ```ts
  hooksJson.includes('"type":"command"')
  ```
- **Impact:** False negatives/positives depending on formatting/spacing; confirmation may appear unexpectedly or be skipped.
- **Fix:** Parse JSON (like `HooksTab.hasDangerousHooks` does) or reuse the same parsed detection helper.

4) **Malformed-but-loadable nodes can create controlled/uncontrolled form warnings**
- **Where:** `TriggerConfig` uses `value={data.event}` without `?? ''`. If a loaded trigger node has missing `event`, the `<select>` becomes uncontrolled.
- **Impact:** React warnings + potentially sticky UI state for corrupted/legacy canvasJson.
- **Fix:** Coerce safe defaults in config UI (`value={data.event ?? ''}`) and/or enforce required fields on load.

## LOW

1) **Palette icons likely render as literal `&#9889;` text**
- **Where:** `NodePalette` passes icon strings like `"&#9889;"` into `<span>{icon}</span>`.
- **Impact:** Broken visual affordances.
- **Fix:** Pass actual characters (`'⚡'`, `'⚙'`) or render entities directly in JSX (not as strings).

2) **`isValidConnection` helper not wired to ReactFlow’s `isValidConnection` prop**
- **Where:** Only enforced inside `onConnect`; users get no “invalid connection” feedback.
- **Impact:** UX confusion (“why doesn’t it connect?”).
- **Fix:** Add `isValidConnection={(c)=>...}` and optionally show `getInvalidConnectionReason`.

3) **`crypto.randomUUID()` may not exist in all runtimes**
- **Where:** `onDrop` node id generation.
- **Impact:** Drag/drop can hard-fail in older environments.
- **Fix:** Fallback to a small UUID polyfill or timestamp+counter.

4) **No edge ID uniqueness validation on load**
- **Where:** `isValidCanvasState` doesn’t check duplicate `edge.id`.
- **Impact:** ReactFlow rendering/key collisions if persisted canvas has duplicates.
- **Fix:** Validate uniqueness and regenerate IDs when needed.
