## HIGH — Persistent DoS / storage bloat via **unsized `hooksCanvasJson`** load + stringify
**Where:** `src/components/agent-builder/HookCanvas.tsx` (mount load effect; `saveLayout` / `notifyChange` JSON.stringify)

**Issue:** `hooksCanvasJson` is parsed with `JSON.parse(hooksCanvasJson)` without any size limit, and later the app repeatedly `JSON.stringify`s the full `nodes[].data` back into `hooksCanvasJson` (layout save every ~2s, structural save every ~500ms). A malicious profile (or corrupted backend record) can supply a huge `hooksCanvasJson` that:
- freezes the UI during `JSON.parse` (main thread),
- triggers memory spikes / OOM,
- then persists the huge payload back on any change, “cementing” the bloat.

**Why it’s exploitable:** This is a classic persistent front-end DoS vector in desktop apps (Tauri) and SPAs: untrusted saved state is treated as trusted and is both parsed and re-saved.

**POC idea:** store a profile where `hooksCanvasJson` contains 200 nodes, each with `data.prompt` sized tens of MB (or huge `data.headers` object). Opening the Hooks tab in Canvas view will parse, render, then stringify the enormous state on any position change or autosave.

**Fix:**
- Enforce a hard cap before parsing, e.g. `if (hooksCanvasJson.length > MAX_CANVAS_JSON_CHARS) { reject }`.
- When saving layout/canvas, do **not** persist arbitrary `data` keys; persist only a whitelisted schema per node type (and cap lengths).
- Consider compressing canvas state or storing it separately with quotas.

## HIGH — Sanitization bypass on the **hooksJson → canvas** path enables large-object injection (especially `headers`)
**Where:** `src/lib/canvasGenerator.ts` + `HookCanvas.tsx` “Priority 2: generate from hooks JSON”

**Issue:** When generating the canvas from `hooksJson`, the produced node `data` is not run through `sanitizeNodeData(...)`. In particular:
- `buildActionData()` for `http` passes through `hook.headers` **verbatim** if it’s an object:
  ```ts
  headers: hook.headers && typeof hook.headers === 'object' ... ? hook.headers : undefined
  ```
- There is no cap on header key count, key/value sizes, or dangerous keys at this stage.

**Impact:**
- Attacker-controlled `hooksJson` can inject a massive `headers` object (or very large strings inside it). Even if the compiler later drops these headers when producing `hooksJson`, the canvas state still holds them and `saveLayout()` will persist them into `hooksCanvasJson`, causing persistent bloat/DoS.
- This bypasses the careful limits you added in `sanitizeNodeData` (which only runs for drag/drop and saved canvas loads).

**POC idea:** Paste into JSON view:
```json
{
  "hooks": {
    "PreToolUse": [{
      "hooks": [{
        "type": "http",
        "url": "https://example.com",
        "headers": { "X": "AAAA....(very large)....AAAA" }
      }]
    }]
  }
}
```
Switch to Canvas view → generator imports the huge headers → autosave persists huge `hooksCanvasJson`.

**Fix:**
- Run generated nodes through the same `sanitizeNodeData` used elsewhere (and also cap event/label length).
- In `buildActionData()`, sanitize and cap headers (count + sizes) and drop keys in `DANGEROUS_KEYS`.

## MEDIUM — Prototype-pollution hardening gap in compiler header reconstruction
**Where:** `src/lib/canvasCompiler.ts` in `buildHookEntry()` (`http` case)

**Issue:** You rebuild `sanitizedHeaders` using a normal object `{}` and assign attacker-controlled keys:
```ts
const sanitizedHeaders: Record<string, string> = {};
sanitizedHeaders[k] = v;
```
You do **not** filter `k` against `DANGEROUS_KEYS` here (unlike `HookCanvas.sanitizeNodeData` which does). Today, values must be strings so classic `__proto__` object-gadget pollution is mitigated, but:
- this is fragile defense-in-depth (future changes to allow non-string header values or different merge patterns would re-open prototype pollution),
- and even string assignment to special keys can cause odd behavior in some code that later iterates/merges.

**Fix:**
- Use `Object.create(null)` for header maps and skip `__proto__/constructor/prototype/...` keys:
  ```ts
  const sanitizedHeaders = Object.create(null) as Record<string,string>;
  if (DANGEROUS_KEYS.has(k)) continue;
  ```

## MEDIUM — Unbounded `onUpdateNode` allows state bloat and bypasses all per-field limits
**Where:** `src/components/agent-builder/HookCanvas.tsx` `onUpdateNode`

**Issue:** Node edits from the config panel are merged directly:
```ts
{ ...n, data: { ...n.data, ...newData } }
```
No call to `sanitizeNodeData`, no length caps, no stripping of unexpected keys. While `compileCanvasToHooks()` slices some fields when producing `hooksJson`, **`saveLayout()` persists the raw `data`** into `hooksCanvasJson`. So a user (or any code path that can call updates) can paste extremely large content (e.g., prompt/command/description), producing:
- UI lag (ReactFlow state + minimap + JSON.stringify),
- persistent storage bloat (huge `hooksCanvasJson`).

**Fix:**
- Sanitize and cap on update:
  ```ts
  const merged = sanitizeNodeData({ ...n.data, ...newData });
  ```
- Optionally whitelist keys by node type.

## MEDIUM — Imported canvas JSON can violate graph invariants → silent miscompilation / confusion
**Where:** `HookCanvas.tsx` `isValidCanvasState(...)` + load path; compile logic in `canvasCompiler.ts`

**Issue:** `isValidCanvasState` validates IDs/types exist but does **not** validate:
- connection rules (trigger→condition/action only, condition→action only),
- “only one incoming edge to action” rule (enforced only in `onConnect`),
- unique edge IDs, etc.

A malicious `hooksCanvasJson` can create states not possible via UI interactions, which can:
- duplicate hooks in unexpected ways,
- make the canvas visually misleading relative to compiled output,
- create “action sharing” across triggers that gets duplicated in compiled hooks.

**Fix:** Add structural validation in `isValidCanvasState` using `connectionRules` and enforce target fan-in constraints.

## MEDIUM — Drag/drop accepts unbounded `dataTransfer` payload → UI hang
**Where:** `HookCanvas.tsx` `onDrop`

**Issue:** `raw = event.dataTransfer.getData('application/reactflow')` is parsed with `JSON.parse(raw)` with no size limit. Another app/tab can provide a massive string payload under that MIME type; even though node count is capped, the parse itself can hang.

**Fix:** Reject large `raw` strings before parsing (and consider try/catch with timing guards).

## LOW — Import dangerous-hook detection is inconsistent (typed vs untyped)
**Where:** `HooksTab.tsx` `handleImport` vs `hasDangerousHooks`

**Issue:** `handleImport` only checks for `"type":"command"` / `"type":"http"` substrings, while `hasDangerousHooks` also flags hooks missing `type` but containing `command`/`url`. This is a minor warning-bypass inconsistency.

**Fix:** Reuse the same structured detection used in `hasDangerousHooks` for the import confirmation.
