## Security review (Visual Hook Builder / ReactFlow canvas)

### HIGH — `__proto__`/special-key injection in HTTP headers can crash compiler (DoS) and risks proto-side effects
**Where**
- `src/lib/canvasCompiler.ts` → `buildHookEntry()`:
  ```ts
  const sanitizedHeaders: Record<string, string> = {};
  ...
  sanitizedHeaders[k] = v;   // k can be "__proto__"
  ```
- `src/lib/canvasGenerator.ts` → `buildActionData()` passes through untrusted headers:
  ```ts
  headers: hook.headers && typeof hook.headers === 'object' ... ? hook.headers : undefined
  ```

**Impact**
- If a user imports/pastes hooks JSON containing an HTTP hook with `headers: {"__proto__": "x"}` (or other magic keys depending on engine behavior), then switching to Canvas and making any structural edit that triggers `compileCanvasToHooks()` can throw (common outcome: `TypeError: Object prototype may only be an Object or null`), breaking autosave/updates and potentially crashing the UI flow.
- Even if it doesn’t throw in some environments, writing to `__proto__` is a well-known footgun that can mutate an object’s prototype (behavior varies), which can cause unpredictable downstream behavior.

**Minimal PoC hooks.json**
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "hooks": [
          {
            "type": "http",
            "url": "https://example.com",
            "method": "GET",
            "headers": { "__proto__": "boom" }
          }
        ]
      }
    ]
  }
}
```
Steps: Import → switch to Canvas → edit any node/edge (or anything that triggers recompilation). The compile path can throw.

**Fix**
- Treat header maps as *untrusted maps*, not normal objects:
  - Build `sanitizedHeaders` as `Object.create(null)` (no prototype), and skip dangerous keys:
    - skip: `__proto__`, `prototype`, `constructor` (and ideally the full set you already maintain elsewhere).
  - Also sanitize *on generation*: in `canvasGenerator.ts`, copy/sanitize headers into a safe structure (string keys/values only, dangerous keys removed, cap count).
- Wrap compilation in a try/catch at the debounce boundary so one bad node can’t brick the whole editor.

### HIGH — Untrusted `headers` objects are stored in React state unsanitized (perf/DoS + persistence of hazardous keys)
**Where**
- `src/lib/canvasGenerator.ts` returns `ActionNodeData.headers` as the original parsed object (no key/value validation, no size cap).
- `src/components/agent-builder/HookCanvas.tsx` fingerprints nodes with:
  ```ts
  JSON.stringify(nodes.map(n => ({ id: n.id, type: n.type, data: n.data })))
  ```
  so large nested objects in `data` amplify CPU/memory costs on every nodes/edges update.

**Impact**
- A crafted hooks JSON (within your 500KB import limit) can embed a very large `headers` object and force repeated expensive `JSON.stringify` work in `getHooksFingerprint()`, causing UI lag/freezes.
- This also preserves hazardous keys (including `__proto__`) inside state until compile time (ties directly into the crash in issue #1).

**Fix**
- Enforce “flat, primitive-only node.data” consistently:
  - In `buildActionData()` either drop headers entirely from canvas nodes, or store a sanitized+bounded representation (e.g., array of `{k,v}` pairs with max N).
  - Apply a hard cap: header count, total serialized size, and key/value lengths.
- Consider hashing fingerprint inputs (or using stable structural hashes) rather than stringifying full `data` objects.

### MEDIUM — Drag-and-drop accepts arbitrary JSON payloads; no input size / key-count limits (CPU DoS)
**Where**
- `src/components/agent-builder/HookCanvas.tsx` → `onDrop()` parses:
  ```ts
  const { nodeType, data } = JSON.parse(raw);
  const sanitizedData = sanitizeNodeData(data); // iterates Object.entries(data)
  ```
**Impact**
- Another app/window can drag data with MIME `application/reactflow` containing a massive object (huge number of keys). Even though values are truncated/skipped, `Object.entries(data)` is still O(n) and can freeze the UI thread.

**Fix**
- Enforce limits before parsing/sanitizing:
  - Max `raw.length` (bytes/chars)
  - Max number of keys in `data`
  - Reject if `Object.keys(data).length > N`
- Consider using a safe schema validator (zod/valibot) with explicit bounds.

### MEDIUM — Loaded canvas JSON does not validate connection rules; inconsistent graphs can be persisted (data corruption / weird compile)
**Where**
- `isValidCanvasState()` checks that edges reference existing nodes, but **does not** enforce `connectionRules.ts` constraints for loaded state.

**Impact**
- A crafted `hooksCanvasJson` can include edges like `action -> trigger`, `condition -> condition`, multiple incoming edges to an action, etc.
- UI may display an invalid graph; compile behavior becomes surprising (partial drops, missing hooks), enabling intentional corruption of saved hooks by supplying a malicious canvas JSON.

**Fix**
- During canvas-state load, validate every edge against `isValidConnection(source.type, target.type)` and also enforce single-incoming-edge constraints for action targets (you already enforce this in `onConnect`, but not on load).

### MEDIUM — “Dangerous hooks” import confirmation is brittle (easy to bypass); can lead to unprompted executable content import
**Where**
- `src/components/agent-builder/tabs/HooksTab.tsx` → `handleImport()`:
  ```ts
  const stringified = JSON.stringify(parsed);
  const hasCommandHooks = stringified.includes('"type":"command"');
  ```
**Impact**
- This warning can be bypassed by trivial JSON variations (`"type": "command"` with whitespace) because the check is substring-based.
- Result: user may import command/http hooks (which can execute commands / make requests when the agent runs) without the intended confirmation gate.

**Fix**
- Parse and structurally traverse hooks (like you already do in `hasDangerousHooks`) instead of substring matching.

---

### LOW — Inconsistent “dangerous key” handling (partial lists) increases future risk surface
**Where**
- Multiple `DANGEROUS_KEYS` sets differ:
  - `HookCanvas.tsx` has a broader set
  - `isValidCanvasState()` redefines a smaller set for IDs only

**Impact**
- Inconsistency makes it easier to miss a future injection path (especially if later you add editable fields like headers/env/template variables).
- Not immediately exploitable beyond the higher issues above, but it’s a maintainability/security footgun.

**Fix**
- Centralize dangerous-key policy in one utility and apply consistently to:
  - node IDs
  - node.data (including nested maps like headers)
  - event keys and any other user-controlled “keyed objects”.

---

### LOW — Invalid/unknown `hookType` can be introduced (via DnD payload) and silently dropped by compiler (integrity issue)
**Where**
- `HookCanvas.tsx` `onDrop()` only validates `nodeType`, not `data.hookType`.
- `canvasCompiler.ts` drops unknown action types (`buildHookEntry` default returns `null`).

**Impact**
- A malicious drop payload can create “action” nodes that look valid in the UI but always compile to nothing, causing silent loss of intent and confusing state.

**Fix**
- Validate `hookType` against an allowlist at creation/update time and surface an error state in the node UI instead of silently compiling it away.
