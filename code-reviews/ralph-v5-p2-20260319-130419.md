## CRITICAL

1) **Canvas-only load can wipe hooksJson on first “position-only” change (silent data loss)**
- **Where:** `HookCanvas.tsx` (`lastCompiledHooksJsonRef`, `saveLayout`, structural fingerprint excludes position)
- **How to trigger (realistic):**
  1. Open a profile/state where `hooksCanvasJson` exists but `hooksJson` is `null`/empty (migration, partial saves, backend bug, or user state corruption).
  2. Canvas loads from `hooksCanvasJson` (no compilation runs).
  3. Drag any node (position-only change) or even do a selection change that causes a nodes update.
  4. After 2s debounce, `saveLayout()` calls `onChange(lastCompiledHooksJsonRef.current, canvasJson)` where `lastCompiledHooksJsonRef.current === ''`.
  5. Parent receives `hooksJson = ''` and persists it.
- **Impact:** Hooks are effectively deleted without any user intent; could break agents in production.
- **Fix:**
  - On successful load from `hooksCanvasJson`, initialize `lastCompiledHooksJsonRef.current` to:
    - existing `hooksJson` if present, else
    - `compileCanvasToHooks(loadedNodes, loadedEdges)` (or delay any `saveLayout` until first compilation completes).
  - Consider blocking `saveLayout` until `lastCompiledHooksJsonRef.current` is non-empty *or* known-valid.

## HIGH

1) **hooksCanvasJson persists transient ReactFlow node state (selection, internal flags) + triggers writes on mere selection**
- **Where:** `HookCanvas.tsx`
  - Fingerprint ignores everything except `{id,type,data}` and edge endpoints.
  - Any nodes change that doesn’t affect fingerprint (e.g., `selected` toggles) is treated as “position-only” and triggers `saveLayout()`.
  - `saveLayout()` serializes full `currentNodes/currentEdges` objects (not a minimal schema).
- **Impact:**
  - Clicking nodes/pane can mark the profile “dirty” and overwrite stored canvas JSON even though nothing meaningful changed.
  - Saved canvas JSON can bloat with transient fields and become version-fragile (future ReactFlow updates may add fields that break assumptions).
  - Reload may restore stale `selected` state (confusing UI) or other internal flags.
- **Fix:**
  - Persist a **minimal** canvas schema: `nodes: {id,type,position,data}`, `edges: {id,source,target}` only.
  - Detect actual position changes (compare prior positions) instead of “fingerprint equals => layout change”.
  - Alternatively, strip transient keys before saving.

2) **HookCanvas ignores prop changes after mount (stale canvas, overwrites external updates)**
- **Where:** `HookCanvas.tsx` “Load saved state on mount” effect has `[]` deps and `initialized` ref.
- **Impact:** If `draft.hooksJson` / `draft.hooksCanvasJson` changes while Canvas view stays mounted (undo/redo, remote load, profile switch without remount), canvas won’t update. Next debounced `onChange` can overwrite the new parent state with stale canvas state.
- **Fix:** Add a controlled “reset/reload” path:
  - watch `hooksJson/hooksCanvasJson/profileId` changes and reconcile (with a “you have unsaved canvas changes” prompt if needed), or
  - ensure remount key always changes when underlying data changes (not just `profileId`).

3) **State leakage when `profileId` is undefined or reused**
- **Where:** `HooksTab.tsx` uses `key={profileId}` for `<HookCanvas />`.
- **Impact:** If multiple drafts can exist with `profileId === undefined` (new profiles, templates, etc.), the canvas can reuse the same component instance/state across different drafts, showing and saving the wrong hooks.
- **Fix:** Use a stronger key, e.g. `key={profileId ?? draftId}` or `key={profileId ?? hash(draft.hooksJson ?? '')}`.

4) **Unbounded string sizes accepted when loading hooksCanvasJson (UI DoS / memory blowup)**
- **Where:** `HookCanvas.tsx` `isValidCanvasState()` validates types but **does not enforce length limits**; then loads into React state.
- **Impact:** A large `hooksCanvasJson` (or very large `data.*` strings) can freeze the UI/tab, crash the app, or make the profile impossible to open.
- **Fix:** Enforce length caps on load (similar to `sanitizeNodeData`), not only on drag/drop.

## MEDIUM

1) **Controlled/uncontrolled input hazards for imported/invalid node data**
- **Where:** `NodeConfigPanel.tsx`
  - `TriggerConfig`: `<select value={data.event}>` (can be `undefined`)
  - `ConditionConfig`: `<input value={data.matcher}>` (can be `undefined`)
- **Impact:** React warnings; in some cases the controls can behave oddly if a malformed canvas state is loaded (which `isValidCanvasState` currently allows by permitting `event/matcher` to be `undefined`).
- **Fix:** Use null-safe values: `value={data.event ?? HOOK_EVENTS[0]}` and `value={data.matcher ?? ''}`.

2) **Palette icons rendered as literal HTML entities**
- **Where:** `NodePalette.tsx` passes `"&#9889;"`, `"&#9881;"` as strings into `{icon}`.
- **Impact:** UI shows `&#9889;` text instead of the icon.
- **Fix:** Pass actual characters (`"⚡"`, `"⚙"`) or render entities directly in JSX.

3) **Export may fail in some environments due to immediate URL revocation**
- **Where:** `HooksTab.tsx` `handleExport()`: `a.click(); URL.revokeObjectURL(url);`
- **Impact:** In some browsers/webviews the download can be interrupted.
- **Fix:** Revoke in a `setTimeout(..., 0/1000)` or on `a` `onload` pattern.

4) **Import invalid JSON is silently ignored**
- **Where:** `HooksTab.tsx` `handleImport()` catch block is empty.
- **Impact:** Users get no feedback (“nothing happened”), leading to repeated attempts and confusion.
- **Fix:** Show an inline error/toast.

5) **Unsupported/dangerous hook detection is best-effort and can miss malformed shapes**
- **Where:** `HooksTab.tsx` `hasUnsupportedHooks` / `hasDangerousHooks`
- **Impact:** If `group.hooks` isn’t an array but is truthy, current loops can throw and fall into `catch { return false; }`, suppressing warnings. User may switch to canvas and lose hooks without being warned.
- **Fix:** Validate `Array.isArray(group.hooks)` before iterating in both detectors; avoid “exception => no warning”.

## LOW

1) **Debounce functions are created every render (wasteful)**
- **Where:** `HookCanvas.tsx` `useCallback(debounce(...), [onChange])` still evaluates `debounce(...)` each render.
- **Impact:** Minor perf/memory churn.
- **Fix:** Wrap debounce creation in `useMemo` or create inside `useEffect` with refs.

2) **Round-trip fidelity loss: multiple triggers with same event collapse into one event in JSON**
- **Where:** `canvasCompiler.ts` merges groups under the same `event` key; `canvasGenerator.ts` creates only one trigger node per event.
- **Impact:** Layout/structure not preserved if user intentionally separated triggers of same event.
- **Fix:** If you want fidelity, store trigger-grouping metadata in `hooksCanvasJson` only (and avoid regenerating solely from hooksJson unless needed).

3) **Duplicate edges/groups possible for non-action targets**
- **Where:** `HookCanvas.tsx` `onConnect` only prevents multiple incoming edges to `action`; duplicates to `condition` are possible (esp. via loaded canvas JSON).
- **Impact:** Compiler may emit duplicated groups or confusing graph states.
- **Fix:** Add dedupe checks for identical `source->target` edges universally.

4) **`crypto.randomUUID()` may not exist in some webviews**
- **Where:** `HookCanvas.tsx` `id: \`${nodeType}-${crypto.randomUUID()}\``
- **Impact:** Drag/drop can hard-crash in older environments.
- **Fix:** Fallback to a small UUID polyfill or `Date.now()+Math.random()`.
