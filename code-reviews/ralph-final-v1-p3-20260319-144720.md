## HIGH

### HIGH-1: Incomplete prototype-pollution / dangerous-key hardening for `headers`
**Where**
- `src/lib/canvasCompiler.ts` → `buildHookEntry()` (HTTP case)
- `src/lib/canvasGenerator.ts` → `buildActionData()` (HTTP case)

**Problem**
You have a `DANGEROUS_KEYS` set in multiple places, and `HookCanvas` sanitizes headers carefully. But the compiler’s header sanitization **does not** reject dangerous keys (e.g. `__proto__`, `constructor`, `prototype`, `__defineGetter__`, etc.), and it builds `sanitizedHeaders` as a normal object (`{}`).

This creates a correctness + security inconsistency:
- If a canvas node’s `data.headers` contains `__proto__`, then `sanitizedHeaders[k] = v` can invoke the `__proto__` setter and mutate the prototype of the headers object (at minimum causing header loss / weird serialization; in some contexts it can be leveraged further).
- Generator currently passes `hook.headers` through largely “as-is” (only checks it’s a non-array object). That means hostile JSON pasted into the JSON view can flow into canvas state without being normalized to a safe “plain dictionary” representation.

**Why it matters**
- Violates an expected invariant: “we strip dangerous keys everywhere”.
- Can lead to surprising dropped headers, corrupted objects, or downstream prototype issues when other code merges/iterates.

**Fix**
- In **both** generator and compiler, build headers as **null-prototype** objects and skip dangerous keys:
  - `const sanitizedHeaders = Object.create(null) as Record<string,string>;`
  - `if (DANGEROUS_KEYS.has(k)) continue;`
- Also consider enforcing `Object.prototype.hasOwnProperty.call(headers, k)` when iterating untrusted objects.

### HIGH-2: Loaded canvas graphs can violate connection invariants; compiler does not validate/prune invalid edges
**Where**
- `src/components/agent-builder/HookCanvas.tsx` → `isValidCanvasState()` and load path
- `src/lib/canvasCompiler.ts` → compilation logic assumes valid graph shape

**Problem**
UI creation (`onConnect`) enforces some invariants (e.g. action has single incoming edge; valid type-to-type connections). But `isValidCanvasState()` only checks:
- nodes/edges arrays
- ids exist and aren’t dangling
- some basic data-type checks

It does **not** enforce key graph invariants, so persisted/hand-edited `hooksCanvasJson` can load edges like:
- `action → condition` (invalid, action should be leaf)
- `anything → trigger` (trigger should never be target)
- multiple edges targeting the same `action` (violates “actions have one incoming”)
- `condition → condition` (invalid)
- `trigger → trigger` (invalid)

The compiler then:
- builds adjacency for *all* edges
- only “walks” 1–2 hops from triggers, effectively **silently ignoring** parts of invalid graphs
- can produce nondeterministic “which group did this action end up in?” behavior when an action is reachable via multiple paths (your `processedActionIds` is per-trigger and order-dependent)

**Why it matters**
- Breaks correctness invariants: same canvas state can compile unexpectedly, or omit hooks without obvious UI explanation.
- Makes persistence brittle: a slightly corrupted canvas JSON can load “fine” but compile wrong.

**Fix**
- Strengthen `isValidCanvasState()` (or a post-load normalization step) to:
  - validate each edge’s `(source.type,target.type)` with `connectionRules.isValidConnection`
  - enforce “no edges into trigger”
  - enforce “no outgoing edges from action”
  - enforce single incoming edge for action targets
  - optionally drop/prune invalid edges on load with a warning
- Consider validating uniqueness/length of edge IDs too (ReactFlow can behave badly with duplicate IDs).

## MEDIUM

### MEDIUM-1: `structuralDirtyRef` is set for *any* edge change, including selection; can rewrite hooks JSON without real edits
**Where**
- `src/components/agent-builder/HookCanvas.tsx` → `handleEdgesChange`

**Problem**
```ts
if (changes.length > 0) structuralDirtyRef.current = true;
```
Edge “change” events include selection/interaction changes, not just add/remove. That means merely selecting an edge can trigger:
- recompilation
- `onChange(hooksJson, canvasJson)` emission
- potential JSON normalization (pretty-print) or reordering effects

**Why it matters**
- Unintended state transitions: “view interaction” becomes “data edit”.
- Can mark profiles dirty / overwrite formatting unexpectedly.

**Fix**
- Only set `structuralDirtyRef` for structural edge change types (`add`, `remove`, `replace`, possibly `reset`), not `select`.

### MEDIUM-2: `generateCanvasFromHooks()` does not cap/sanitize event key length (UI/perf risk)
**Where**
- `src/lib/canvasGenerator.ts`

**Problem**
Trigger nodes take `event` directly from JSON keys:
```ts
const triggerData: TriggerNodeData = { event, label: event };
```
No length cap. Very large keys can produce huge labels, expensive renders, large saved canvas JSON, etc.

**Fix**
- Enforce a maximum event/label length (similar to matcher caps), e.g. 200–500 chars, and truncate label separately from stored event if needed.

### MEDIUM-3: Canvas ↔ hooks JSON round-trip changes grouping semantics for “no matcher” groups
**Where**
- `src/lib/canvasCompiler.ts` (Trigger → Action directly)
- `src/lib/canvasGenerator.ts` (group-without-matcher can contain multiple hooks)

**Problem**
Generator supports a hook group with no matcher and multiple hooks (common JSON shape). Compiler emits **one group per direct action**:
```ts
groups.push({ hooks: [hookEntry] });
```
So JSON → canvas → JSON will often restructure:
- from one group with N hooks
- into N groups with 1 hook each

This is *likely* semantically equivalent today (given only `matcher` exists at group level), but it’s a correctness hazard if group-level fields expand later or if ordering/grouping is expected to be preserved.

**Fix**
- When compiling direct Trigger→Action edges, consider emitting a **single** group containing hooks in edge order (or preserve grouping explicitly in the canvas model).

## LOW

### LOW-1: NodePalette icons are HTML entities stored as strings; render shows literal text
**Where**
- `src/components/agent-builder/panels/NodePalette.tsx`

**Problem**
`icon="&#9889;"` is rendered via `{icon}`, so React escapes it and displays the literal `&#9889;` text.

**Fix**
Use actual characters (`'⚡'`, `'⚙'`) or render entities as JSX text, not as a string prop.

### LOW-2: Declining “dangerous canvas load” logs misleading “Invalid canvas state” warning
**Where**
- `src/components/agent-builder/HookCanvas.tsx` load effect

If the canvas JSON is valid but the user declines the dangerous-hook confirmation, the code falls through and then logs:
```ts
console.warn('[HookCanvas] Invalid canvas state, regenerating from hooks JSON');
```
This is not an invalid-state situation and can confuse debugging.

**Fix**
Add a separate branch/return to avoid the “invalid canvas state” warning on user-decline.
