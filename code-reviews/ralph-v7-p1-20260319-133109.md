## HIGH — Dangerous hook confirmation bypass via brittle substring check (silent load of command/HTTP canvas)
**Where:** `HookCanvas.tsx` load-path for `hooksCanvasJson`  
**Code:**  
```ts
const hooksAlreadyDangerous =
  hooksJson &&
  (hooksJson.includes('"type":"command"') || hooksJson.includes('"type":"http"'));
...
if (hasDangerous && !hooksAlreadyDangerous) { window.confirm(...) }
else { /* load directly */ }
```

**Impact:** If an attacker can provide a profile containing both `hooksJson` and `hooksCanvasJson`, they can **bypass the “load dangerous canvas?” confirmation** and cause dangerous command/http action nodes to load silently. Any subsequent canvas edit will compile and persist real dangerous hooks into `hooksJson`, potentially leading to command execution / HTTP exfiltration when the agent runs.

**Exploit sketch (persistent “smuggling”):**
1. Set `hooksJson` to contain a *prompt* hook whose prompt text includes the literal substring `"type":"command"` (not an actual command hook).
2. Set `hooksCanvasJson` to a valid canvas state containing an action node with `data.hookType === "command"` and a real `command`.
3. On load:
   - `hasDangerousHooks` (in `HooksTab`) stays **false** (because JSON has no real command/http hook types).
   - `hooksAlreadyDangerous` becomes **true** due to the prompt string containing that substring.
   - Canvas loads dangerous nodes **without confirmation**.

**Fix:**
- Replace substring heuristics with structured parsing (reuse `hasDangerousHooks` logic or parse+walk hooksJson).
- Consider always confirming when `hooksCanvasJson` contains dangerous nodes, regardless of `hooksJson`.

## MEDIUM — UI Denial-of-Service via expensive fingerprinting on every node/edge change (dragging triggers huge stringify)
**Where:** `HookCanvas.tsx`  
**Code:**  
```ts
function getHooksFingerprint(nodes, edges) {
  return JSON.stringify(nodes.map(n => ({ id:n.id, type:n.type, data:n.data })))
    + '|' + JSON.stringify(edges.map(...));
}
...
useEffect(() => {
  const fingerprint = getHooksFingerprint(nodes, edges);
  if (fingerprint === prevStructuralRef.current) { saveLayout(...); return; }
  notifyChange(...);
}, [nodes, edges]);
```

**Impact:** `getHooksFingerprint` stringifies **all node `data`**, including up to:
- `prompt` length 50,000
- `MAX_NODE_COUNT` 200

Dragging a node causes frequent `nodes` updates (position changes), which repeatedly recomputes this fingerprint even when the structure didn’t change. With large prompts (or large `headers` objects), this becomes a **CPU/memory spike** and can freeze the UI.

**Fix:**
- Fingerprint only structural fields needed for compilation (e.g., ids/types + a small subset of data fields relevant to compilation, or per-node revision counters).
- Cache per-node hashes; don’t repeatedly stringify huge prompt strings on position-only updates.

## MEDIUM — Drag-and-drop parsing accepts unbounded payload (cross-origin/social DoS)
**Where:** `HookCanvas.tsx` `onDrop`  
**Code:**  
```ts
const raw = event.dataTransfer.getData('application/reactflow');
const { nodeType, data } = JSON.parse(raw);
```

**Impact:** An attacker can craft a drag source (e.g., a web page) that sets a **very large** `application/reactflow` payload. If the user drags into the app, `JSON.parse(raw)` can block the UI thread (no size cap, no early rejection).

**Fix:**
- Enforce a maximum `raw.length` before parsing (e.g., 10–50KB).
- Validate schema with a strict parser and reject extra/unexpected fields early.

## MEDIUM — `canvasGenerator` imports HTTP headers unsanitized (proto keys / bloat / downstream surprises)
**Where:** `src/lib/canvasGenerator.ts`  
**Code:**  
```ts
headers: hook.headers && typeof hook.headers === 'object' && !Array.isArray(hook.headers)
  ? hook.headers
  : undefined,
```

**Impact:**
- Allows huge header maps (no count/size cap) → amplifies the fingerprinting DoS above.
- Allows dangerous keys like `__proto__` / `constructor` to persist into canvas state (even if later compilation partially filters by type checks). This is the kind of “latent” object-shape risk that often becomes exploitable later when code changes (e.g., if someone later `Object.assign`s headers).

**Fix:**
- Sanitize headers at import time the same way as in `sanitizeNodeData` / `canvasCompiler`:
  - Drop dangerous keys.
  - Enforce key/value string-only, CRLF-free.
  - Cap header count and total size.

## LOW — `onUpdateNode` merges arbitrary keys without sanitization (future proto-pollution footgun)
**Where:** `HookCanvas.tsx`  
**Code:**  
```ts
n.id === nodeId ? { ...n, data: { ...n.data, ...newData } } : n
```

**Impact:** Today `newData` comes from controlled inputs, but this is an attractive future injection point (e.g., if later you allow editing headers or importing arbitrary node fields). Keeping a generic merge without filtering dangerous keys is a common way prototype pollution and data-shape corruption gets introduced.

**Fix:** Run `sanitizeNodeData(stripDangerousKeys(newData))` (or a per-node-type schema validator) before merging.

## LOW — HTTP method not validated / CRLF not filtered
**Where:** `canvasCompiler.ts`  
**Code:** `entry.method = data.method.slice(0, 20);`

**Impact:** Method is user-controlled and not constrained to standard tokens; if downstream code builds raw HTTP requests (not `fetch`), this can become request smuggling / header injection via control characters.

**Fix:** Restrict to `GET|POST|PUT|PATCH|DELETE` (or RFC token regex) and reject `\r\n`.

## LOW — Regex matcher can be weaponized for ReDoS in downstream runtime
**Where:** `ConditionConfig` + compiler truncation only (`MAX_MATCHER_LENGTH`)  
**Impact:** Users can input catastrophic backtracking regex that may freeze the agent runtime when evaluated. This is a config-level DoS rather than a front-end exploit, but the UI currently provides no guardrails.

**Fix:** Prefer safe-regex validation, timeouts, or a limited glob-like matcher.
