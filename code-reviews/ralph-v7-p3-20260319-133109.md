## Audit: Visual Hook Builder (React Flow canvas → Claude Code hooks JSON)

### CRITICAL
None found in this subset.

---

### HIGH

1) **Structural-change detection is computationally expensive and runs on every drag**
- `HookCanvas.getHooksFingerprint()` does `JSON.stringify` over *all* node `data` (including prompts up to 50k, and up to 200 nodes). Even though positions are excluded from the fingerprint, the fingerprint is still recomputed on every `nodes` update (including position changes while dragging), which can cause severe UI jank/freezes.
- **Invariant impacted:** “Dragging should only save layout, not do heavy work.” The code avoids *compiling* on drags, but still does heavy fingerprinting work.
- **Fix:** compute a lightweight structural hash (exclude large fields like `prompt/command/description`, exclude `label`), or track “structural vs position change” via `onNodesChange` diffs rather than re-stringifying whole state. At minimum, fingerprint only stable identifiers + the minimal fields that affect compilation.

2) **Sanitization/limits are not applied consistently across all ingress/update paths**
- You sanitize when loading `hooksCanvasJson` and on drag-drop (`sanitizeNodeData`), but **not** when:
  - generating from `hooksJson` (`generateCanvasFromHooks` output is used directly), and
  - updating via config panel (`onUpdateNode` merges `newData` without sanitizing or enforcing max lengths).
- This breaks the intended invariants implied by the constants (`MAX_PROMPT_LENGTH`, `MAX_COMMAND_LENGTH`, etc.) and can lead to state bloat and performance degradation.
- Additionally, `canvasCompiler.buildHookEntry()` sanitizes HTTP headers for CRLF/length, but **does not exclude dangerous keys** (`__proto__`, etc.) and uses `{}` rather than a null-prototype object. While the current value-type checks reduce typical prototype-pollution impact, it’s still an avoidable hazard and inconsistent with the rest of your defensive coding.
- **Fix:** centralize sanitization (including dangerous-key stripping) and apply it:
  - to `generateCanvasFromHooks` outputs before `setNodes/setEdges`,
  - inside `onUpdateNode` (sanitize merged data), and
  - in `buildHookEntry` for headers (filter `DANGEROUS_KEYS`, use `Object.create(null)` for `sanitizedHeaders`).

### MEDIUM

1) **Initial load triggers a compile/onChange even with no user edits**
- On mount, `nodes/edges` go from `[]` → loaded/generated. The watcher effect then treats this as a structural change and calls `notifyChange()`, rewriting `hooksJson` and `hooksCanvasJson`.
- **Impact:** unnecessary “dirtying”/reformatting, and if the canvas representation cannot round-trip some constructs, this can cause unexpected changes immediately upon entering canvas.
- **Fix:** gate notifications until after initialization completes (e.g., a `didHydrateRef`, set baseline fingerprint after hydration, and skip the first non-user-driven change).

2) **Imported/loaded canvas state validation misses key graph invariants**
- `isValidCanvasState` checks IDs/positions/dangling edges, but does **not** enforce:
  - edge type validity (must respect `trigger→condition/action`, `condition→action`),
  - max-one-incoming edge for `action` (your UI enforces on new connects, but loaded JSON can violate),
  - `data` being a plain object (arrays currently pass).
- **Impact:** compiler may silently drop/alter behavior (e.g., duplicated/ambiguous groups) depending on edge ordering.
- **Fix:** extend validation (or normalize after load by removing invalid edges and enforcing one inbound edge to action).

3) **Unknown trigger events are preserved by the compiler but not representable in the config UI**
- `compileCanvasToHooks` explicitly preserves unknown events, but `TriggerConfig` uses a `<select>` limited to `HOOK_EVENTS`. If a trigger has an unknown event, React will have a value not in options (bad UX and potential accidental coercion).
- **Fix:** allow free-form event input, or inject the current value as an extra `<option>` when not in `HOOK_EVENTS`.

4) **Hook ordering is not deterministic/visual-order aware**
- Compilation order is based on adjacency/edge iteration order, not node position. Users will expect top-to-bottom action order to match output order.
- **Fix:** within each trigger/condition, sort connected actions by `node.position.y` before emitting hooks; similarly sort groups by the condition’s `position.y` (or stable edge creation index).

5) **`NodePalette` icons for triggers/conditions likely render as literal entity text**
- Passing `"&#9889;"`/`"&#9881;"` as strings renders the literal characters `& # ... ;` in React, not the symbol.
- **Fix:** pass the actual Unicode character (e.g. `"⚡"`, `"⚙"`) or render as JSX entity directly.

---

### LOW

1) **Connection validation is only enforced in `onConnect`, not via ReactFlow’s `isValidConnection`**
- Comment in `connectionRules.ts` suggests `isValidConnection` prop usage, but `HookCanvas` doesn’t pass it, reducing immediate user feedback during connection attempts.
- **Fix:** provide `isValidConnection` to `<ReactFlow>`.

2) **Debounced functions are created during render (avoidable allocations)**
- `useCallback(debounce(...))` still evaluates `debounce(...)` every render (React will ignore the new function if deps unchanged, but the allocation still happens).
- **Fix:** create debounced functions with `useMemo`/`useRef` and update internals carefully.

3) **Duplicate/inconsistent dangerous-key sets across modules**
- `DANGEROUS_KEYS` differs between files and is redefined inside `isValidCanvasState`.
- **Fix:** centralize constants to prevent drift and ensure consistent blocking.
