## HIGH: Prototype pollution / state corruption via unvalidated **edge IDs** (and weak ID key blacklist)
**Where:** `HookCanvas.tsx` → `isValidCanvasState()` + load of `hooksCanvasJson`  
**Issue:** `isValidCanvasState()` validates node IDs against only `['__proto__','constructor','prototype']`, but **does not validate edge IDs at all**. A malicious saved canvas can contain `edges[].id="__proto__"` (or similar), pass validation, and be fed into `@xyflow/react`.

Many graph/state libs build internal lookup tables using plain objects keyed by IDs (e.g., `acc[id] = edge`). If an ID is `__proto__`, this can mutate the prototype of that accumulator (“prototype pollution”) or crash/derange logic. Even if xyflow uses `Map` in some places, you’re still depending on internals not to ever use `{}` as a keyed map.

**Exploit path (malicious profile / canvas JSON):**
```json
{
  "nodes":[
    {"id":"trigger-1","type":"trigger","position":{"x":0,"y":0},"data":{"event":"SessionStart"}},
    {"id":"action-1","type":"action","position":{"x":0,"y":150},"data":{"hookType":"prompt","prompt":"hi"}}
  ],
  "edges":[{"id":"__proto__","source":"trigger-1","target":"action-1"}]
}
```
Saved as `hooksCanvasJson`, it passes `isValidCanvasState` and loads.

**Fix:**
- Reject dangerous IDs for **edges** too (`e.id`, and ideally also `source`/`target` though those are node IDs).
- Use the **same** comprehensive `DANGEROUS_KEYS` set everywhere (not a smaller per-function set).
- Consider enforcing an ID regex (e.g., `^[a-zA-Z0-9._:-]{1,100}$`) for both nodes and edges.

## HIGH: Import confirmation bypass for “dangerous” hooks when `type` is omitted
**Where:** `HooksTab.tsx` → `handleImport()`  
**Issue:** Import-time warning checks only:
```ts
stringified.includes('"type":"command"') || stringified.includes('"type":"http"')
```
But elsewhere (`hasDangerousHooks`) you *explicitly* treat hooks as dangerous even **without `type`** if they include `command` or `url` fields:
```ts
if (!hook?.type && (typeof hook?.command === 'string' || typeof hook?.url === 'string')) return true;
```
So a JSON file can include executable-looking fields and bypass the **explicit import confirmation**. After import, the UI shows a yellow warning banner, but it does **not** require confirmation and is easy to ignore—meaning you’ve effectively downgraded a “confirm before importing potentially executable content” to “show a passive warning after the fact”.

**Exploit example:** user imports hooks JSON containing:
```json
{"hooks":{"SessionStart":[{"hooks":[{"command":"rm -rf ~"}]}]}}
```
No import confirm dialog appears.

**Fix:**
- Use the same structured detection logic in `handleImport` as `hasDangerousHooks` (parse and walk hooks).
- Consider requiring an explicit confirmation gate before allowing save/run when dangerous hooks exist (not just a banner).

## MEDIUM: Canvas mode can silently drop hooks that are missing `type` without warning (data corruption)
**Where:** `HooksTab.tsx` (`hasUnsupportedHooks`) + `canvasGenerator.ts`  
**Issue:** `hasUnsupportedHooks` only flags unsupported hooks when `hook.type` exists and is not in the supported set. Hooks **without** `type` are not flagged as unsupported, but `generateCanvasFromHooks()` will drop them because `buildActionData()` requires a valid `type`.

Net effect: switching to canvas can silently delete hooks while showing **no “unsupported hooks” warning/confirmation**, because the detection misses the “missing type” case.

**Fix:**
- Treat hooks with missing/invalid `type` as unsupported for the canvas path and warn accordingly.

## MEDIUM: Unbounded `headers` object accepted from hooks JSON → UI/memory DoS + persistence into canvas JSON
**Where:** `canvasGenerator.ts` → `buildActionData()`  
**Issue:** For HTTP hooks, generator does:
```ts
headers: hook.headers && typeof hook.headers === 'object' && !Array.isArray(hook.headers) ? hook.headers : undefined
```
No size/key/value limits, no dangerous key stripping, no CRLF stripping at this stage.

An attacker can paste a very large `hooksJson` (textarea has no max) containing a huge headers object (many keys, very large values). Even with node-count limits (200), a single node can carry megabytes of header data. This can:
- freeze the app when rendering/updating,
- freeze on `saveCanvas()` due to `JSON.stringify` of `canvasJson`,
- bloat stored profile state (`hooksCanvasJson`) dramatically.

**Fix:**
- Sanitize/limit headers in `buildActionData()` similarly to `sanitizeNodeData()` / `canvasCompiler`:
  - strip dangerous keys
  - enforce key/value length caps
  - enforce max header count (e.g., 50–200)
  - strip CRLF
- Consider a global max size for `hooksJson` when switching to canvas view.

## MEDIUM: `canvasCompiler` does not block dangerous header keys (`__proto__`, etc.) in output hooks JSON
**Where:** `canvasCompiler.ts` → `buildHookEntry()` for `http` headers  
**Issue:** Header sanitization checks CRLF and lengths, but does **not** exclude dangerous keys:
```ts
for (const [k, v] of Object.entries(data.headers)) {
  if (typeof k === 'string' && typeof v === 'string' && ...) sanitizedHeaders[k] = v;
}
```
So `__proto__` / `constructor` / `prototype` keys can be emitted into the resulting hooks JSON (even if they’re later “just data”). This is a known footgun for downstream code that merges objects or treats these keys specially.

**Fix:**
- Apply `DANGEROUS_KEYS` filtering to header keys in `canvasCompiler` (as you already do in `HookCanvas.sanitizeNodeData`).

## LOW: Connection validity not enforced when loading saved canvas JSON
**Where:** `HookCanvas.tsx` load path for `hooksCanvasJson`  
**Issue:** `isValidCanvasState` ensures edges reference existing nodes but doesn’t enforce `connectionRules` (trigger→condition/action, condition→action only). This can create confusing canvases and unexpected compilation drops/skips. Not a direct exploit, but increases attack surface for weird states.

**Fix:** validate edges against `checkConnection(source.type, target.type)` during `isValidCanvasState`.
