This is a highly ambitious and well-structured plan that will significantly expand the capabilities of the Claude Tauri app. The architecture cleanly separates key management from the Node/Bun environment, which is excellent for security. 

However, as a senior architect and security engineer, I have identified **one critical security vulnerability** in the proposed signing bridge, several race conditions, and a few architectural edge cases that need to be addressed before implementation.

Here is the detailed review across the 7 requested areas:

### 1. SECURITY (Critical Vulnerability Found)
**CRITICAL: The `localhost:3132/sign` unauthenticated HTTP endpoint.**
*   **The Threat:** If the Rust shell exposes a local HTTP server on port 3132 that blindly signs payloads, *any* process running on the user's machine (or a malicious website via DNS rebinding / lack of strict CORS) can issue a request to `http://localhost:3132/sign` and drain the user's wallet.
*   **The Fix:** Do not use an unauthenticated hardcoded port. If the Bun sidecar needs to request signatures from Rust without routing through the frontend:
    1. Rust should bind to port `0` (dynamic port).
    2. Rust generates a cryptographically secure random token (e.g., 32-byte hex).
    3. Rust passes both the dynamic port and the token to the Bun sidecar as environment variables when spawning it (e.g., `RUST_SIGNING_PORT=45123 RUST_SIGNING_TOKEN=abc...`).
    4. The sidecar must include `Authorization: Bearer <token>` in all requests.
*   **Blind Signing:** The Rust `wallet_sign` command must *not* accept a raw string and sign it blindly. It must parse the 402 challenge, verify the structure (amount, recipient, network), and format it according to protocol specs before signing. This prevents an attacker from tricking the wallet into signing a malicious transaction payload.

### 2. ARCHITECTURE FEASIBILITY
*   **The Keychain Bridge Pattern:** The secure HTTP bridge (described above) is much better than the "Frontend Bridge" flow. Routing sidecar backend requests up to the React frontend via WebSocket/SSE just to call a Tauri `invoke()` command, and then routing the response back down, introduces massive latency and failure points, especially during AI tool execution.
*   **State Desync:** The plan stores wallet keys in the macOS Keychain and metadata in SQLite. If a user uninstalls the app or deletes `~/.claude-tauri/data.db`, the SQLite metadata is lost, but the Keychain retains the private key. On reinstall, `wallet_create` might fail or create a duplicate. 
    * *Recommendation:* Add a `syncWalletsFromKeychain()` boot step in Phase 1.4 that queries the Keychain for `"claude-tauri-wallet"` and reconstructs the SQLite row if missing.

### 3. MISSING EDGE CASES
*   **TOCTOU (Time-of-Check to Time-of-Use) on Spend Limits:** If an agent spawns multiple concurrent subagents, or makes rapid parallel tool calls, two requests might check the spend limit simultaneously. Both see `$0.09` spent (limit `$0.10`), both authorize a `$0.05` payment, resulting in `$0.14` spent.
    * *Fix:* Enforce limits at the database level using an atomic update in `apps/server/src/db/index.ts`. 
      ```sql
      UPDATE spend_limits 
      SET current_spend_usd = current_spend_usd + ? 
      WHERE current_spend_usd + ? <= max_amount_usd AND id = ?
      ```
      If `changes === 0`, the limit is hit, and the payment is rejected.
*   **LLM Timeout during User Approval:** If a payment is over the `$0.10` threshold, it requires user approval. If the user takes 2 minutes to approve it, the AI SDK's HTTP tool call (or the Claude API itself) will time out. 
    * *Fix:* The interceptor should immediately return a simulated tool error to Claude: `"Payment of $0.50 required. Awaiting user approval. Do not retry yet."` Emit the `StreamPaymentApprovalRequired` event. When the user approves via the UI, inject a system prompt or synthetic user message: `"Payment of $0.50 approved. You may now retry the tool call."`

### 4. PHASING & DEPENDENCIES
*   **Dependency Adjustments:** 
    * Phase 4 (Payment Integration) heavily depends on Phase 2 (UI) being complete so that the user *can* approve payments over the limit.
    * Phase 5 (Spend Tracking) should not run in parallel with Phase 4. It relies on the database being populated by real intercepted transactions. It should strictly follow Phase 4.
*   **Phase 6 (x402) is a massive architectural leap.** Tempo and EVM (Base) use completely different cryptography. EVM uses `secp256k1` and requires EIP-1559 transaction formatting. Tempo uses different signatures. 
    * *Recommendation:* In Phase 1.3, explicitly define the key type (e.g., `Ed25519`) in the Keychain metadata so you don't paint yourself into a corner when you need `secp256k1` for Phase 6.

### 5. PROTOCOL ISSUES
*   **Payment Failure / 500s:** What happens if the wallet signs the challenge, the money is deducted from the Tempo/Base channel, but the external API returns a 500 Internal Server Error? The user paid for nothing. 
    * *Recommendation:* Add a `refund_status` to the `transactions` table. If the `paidFetch` receives a 5xx after payment, it should automatically record the transaction as requiring a dispute/refund.
*   **Validation:** You must strictly validate the 402 challenge using Zod schemas (similar to `chatRequestSchema` in `routes/chat.ts`). Do not trust the `WWW-Authenticate: x402 ...` header blindly.

### 6. TDD STRATEGY
*   **Mocking the Keychain:** Testing the sidecar requires a mock signing server. Create a `MockSigningServer` in Bun that spins up on a random port, generates fake signatures, and sets the `RUST_SIGNING_PORT` env var for the sidecar tests.
*   **Interceptor Tests:** The `payment-interceptor.ts` is the most critical logic. Write tests that mock `global.fetch` to return a `402 Payment Required`, verify the interceptor catches it, calls the signing bridge, and retries `fetch` with the `Authorization: MPP ...` header.
*   **Concurrency Tests:** Write a specific Bun test using `Promise.all()` to fire 10 simultaneous `$0.05` payment requests to ensure the atomic SQL spend limit prevents exceeding the `$0.10` limit.

### 7. CODEBASE FIT
The plan aligns beautifully with the existing architecture, but here are specific integration points based on the provided codebase:

*   **Database (`apps/server/src/db/schema.ts`)**: The plan to add `migrateMppTables(db)` fits perfectly alongside `migrateLinearIssueColumns(db)`. Remember that SQLite doesn't support complex `ALTER TABLE`, so design the new tables right the first time. Add indexes on `wallet_id` and `created_at` in the initial `CREATE TABLE` strings.
*   **Settings UI (`apps/desktop/src/components/settings/SettingsPanel.tsx`)**: 
    * Add `'payments'` and `'marketplace'` to the `TabId` union.
    * Use the existing `<SettingField>` and `<ToggleSwitch>` components to ensure visual consistency for auto-approve toggles.
*   **Status Bar (`apps/desktop/src/components/StatusBar.tsx`)**: 
    * `WalletBalanceSegment` and `MppSpendSegment` should follow the `CostSegment` pattern. 
    * Ensure you poll using the `useEffect` with a `cancelled` flag pattern currently used in `ResourceUsageSegment` to prevent memory leaks.
*   **Event Mapper (`apps/server/src/services/event-mapper.ts`)**: 
    * Add cases to `mapSystemEvent` or `mapStreamEvent` for the new payment stream events. Ensure you handle unknown sub-types gracefully, as the current `default:` block warns on unhandled types.
*   **Routes (`apps/server/src/app.ts`)**: 
    * Register your new routes (`app.route('/api/wallet', createWalletRouter(db))`) cleanly in the main router block.
