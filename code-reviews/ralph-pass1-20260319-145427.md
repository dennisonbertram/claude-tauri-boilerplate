## HIGH

### 1) HTTP hook `method`/`url` allow CRLF injection (potential request-splitting / header-smuggling downstream)
**Where**
- `src/lib/canvasCompiler.ts` → `buildHookEntry()` for `http`:
  - `entry.method = data.method.slice(0, 20);` (no allowlist, no CRLF filtering)
  - `entry.url = data.url.slice(0, 2000);` (no CRLF filtering)
- `src/lib/canvasGenerator.ts` → `buildActionData()` for `http`:
  - `method = typeof hook.method === 'string' ? hook.method : 'GET';` then `slice(0, 20)` (still no CRLF filtering)

**Exploit sketch**
Import hooks JSON containing:
```json
{
  "hooks": {
    "PreToolUse": [{
      "hooks": [{
        "type": "http",
        "url": "https://example.com/\r\nX-Evil: 1",
        "method": "GET\r\nX-Injected: 1"
      }]
    }]
  }
}
```
This survives generator/compiler (only truncated), and can be persisted via canvas save.

**Impact**
If the runtime that executes these hooks ever constructs raw HTTP requests, logs, or headers using these fields unsafely, this can become request splitting/smuggling, header injection, or log injection. Even if *your* executor uses `fetch` and rejects such values, other environments/libraries may not.

**Fix**
- Enforce strict allowlist for method: `GET|POST|PUT|PATCH|DELETE` (and reject/normalize anything else).
- Reject any `\r`/`\n` and other control chars in `url` and `method`.
- Optionally validate URL scheme (`http:`/`https:`) and disallow whitespace.

## MEDIUM

### 2) Unbounded JSON parsing of profile-provided `hooksJson` / `hooksCanvasJson` → UI freeze / persistent DoS
**Where**
- `HooksTab.tsx`: multiple `useMemo()` blocks repeatedly `JSON.parse(draft.hooksJson)` on edits.
- `HookCanvas.tsx`: `JSON.parse(hooksCanvasJson)` and `generateCanvasFromHooks(hooksJson)` with no size cap on incoming strings (import has 500KB cap, but profile data path does not).

**Impact**
A malicious or corrupted profile can contain multi-megabyte JSON that causes long main-thread stalls (parse + repeated parsing), effectively a persistent UI DoS when opening the tab or toggling views.

**Fix**
- Enforce max length for `draft.hooksJson` / `draft.hooksCanvasJson` before parsing (same or stricter than import cap).
- Parse in a Web Worker for large inputs, or debounce/parse only on “Validate” action for JSON editor.

### 3) Cross-origin drag-and-drop injection + no payload size cap on DnD JSON → unwanted node creation / DoS
**Where**
- `HookCanvas.tsx` `onDrop()` does `JSON.parse(raw)` on `application/reactflow` with no size limit and no origin/authenticity check.

**Exploit sketch**
An attacker website can set `dataTransfer.setData('application/reactflow', '{...}')` in its own `dragstart`; user drags into your app window; app accepts and creates nodes (including dangerous hooks after a confirm prompt).

**Impact**
- Social-engineering vector to plant hook graphs into a profile (persistence via autosave).
- Possible UI DoS if the dropped string is huge (or intentionally pathological) since it’s parsed on the main thread.

**Fix**
- Add a length cap for `raw` (e.g., 50KB).
- Add an internal nonce to DnD payloads generated by your palette (e.g., `source: 'node-palette', nonce` stored in memory) and reject drops without it.
- Consider only accepting drops when `event.dataTransfer.types` matches an expected internal marker.

### 4) Canvas round-trip drops *unknown fields* on otherwise supported hook types (silent data corruption)
**Where**
- `canvasGenerator.buildActionData()` only maps a small subset of fields (`command/url/method/headers/prompt/model/description/timeout`)
- `canvasCompiler.buildHookEntry()` outputs only those same fields
- `HooksTab` warning only covers “unsupported hook types”, not “supported types with extra fields”

**Impact**
A profile with supported hook types but additional properties (e.g., future Claude Code hook extensions like `body`, `auth`, `retries`, `env`, etc.) will be silently lost after a canvas edit/save. This is integrity corruption and can break behavior without the user understanding why.

**Fix**
- Preserve unknown fields via a `raw`/`extra` bag in node data and re-emit them in the compiler.
- Or warn users: “Canvas will drop unknown fields even for supported hook types” and require confirmation before first canvas save.

### 5) Regex matcher can be ReDoS payload (catastrophic backtracking) for downstream executor
**Where**
- `ConditionNodeData.matcher` is a “regex” string (`NodeConfigPanel` explicitly calls it regex).
- Only length-limited (200 in compiler/generator), but no safety validation.

**Impact**
If the runtime uses backtracking regex engines (typical JS/PCRE), a malicious matcher like `(a+)+$` can spike CPU when matched against crafted tool names/logs, causing runtime DoS.

**Fix**
- Use a non-backtracking regex engine (RE2) in the executor, or enforce a safe-regex validator / timeouts.
- Consider offering glob/wildcard matching instead of raw regex.

## LOW

### 6) `isValidCanvasState()` allows `node.data` to be arrays / non-plain objects (type confusion & weird persistence)
**Where**
- `HookCanvas.tsx` `isValidCanvasState`: `if (!n.data || typeof n.data !== 'object') return false;` (arrays pass)

**Impact**
Can create odd canvas states that render incorrectly and/or compile to empty hooks, enabling subtle persistence bugs and hard-to-debug corruption.

**Fix**
Require “plain object”: `!Array.isArray(n.data)` and optionally `Object.getPrototypeOf(n.data) === Object.prototype || Object.getPrototypeOf(n.data) === null`.

---

### 7) No filtering of control characters / bidi controls in strings → UI/log obfuscation risk
**Where**
- `sanitizeNodeData()` truncates but does not strip control chars (e.g., `\u202E` RTL override, ESC `\u001b`, NUL).

**Impact**
Malicious commands/URLs/prompts can be visually disguised in the UI (or later poison logs/terminals elsewhere in the app), increasing social-engineering success.

**Fix**
Strip non-printable controls (except `\t` maybe) and consider removing bidi override characters from user-visible summaries.

---

### 8) Safety prompt logic is inconsistent/brittle in places (false negatives/positives)
**Where**
- `HookCanvas.tsx`: `hooksAlreadyDangerous` uses substring search for `"type":"command"` / `"type":"http"` (format-dependent)

**Impact**
Not a direct security break, but reduces reliability of warnings/confirmations, which are part of your threat mitigation for dangerous hooks.

**Fix**
Parse JSON and walk it (as you already do elsewhere) instead of substring matching.
