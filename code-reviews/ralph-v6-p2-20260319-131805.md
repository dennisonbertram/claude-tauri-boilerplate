## CRITICAL

### 1) **HTTP `headers` (and any nested fields) are silently dropped when loading saved canvas**
**Where:** `HookCanvas.tsx` → `sanitizeNodeData()` + load-from-`hooksCanvasJson` path  
**Why it breaks:** `sanitizeNodeData` *intentionally* “keeps data flat” and skips objects/arrays. But `canvasGenerator.buildActionData()` **does** put `headers` (object) into action node data for `http` hooks. Result:

- If a profile ever has `hooksCanvasJson`, loading it will **remove `headers` from node.data**.
- After *any* structural canvas edit, `compileCanvasToHooks()` runs and outputs hooks JSON **without headers**, permanently corrupting `hooksJson`.

**Repro (pure data-loss):**
1. Start with `hooksJson` that contains an HTTP hook with `headers`.
2. Open Canvas (generator creates nodes with `headers`).
3. Make any structural change (add node, connect, edit action) → `notifyChange()` compiles.
4. Export JSON: headers are gone.

**Fix ideas:**
- Make sanitizer schema-aware: allow `headers` as a plain object (with its own key/value sanitization).
- Or: treat `hooksJson` as source-of-truth and store *only layout/edges* in `hooksCanvasJson` (positions + connectivity), not hook payloads.

---

### 2) **Long prompts get truncated to 10k on canvas reload (hard data-loss)**
**Where:** `HookCanvas.tsx` → `MAX_STRING_LENGTH = 10_000` in `sanitizeNodeData()`  
**Why it breaks:** Compiler/generator allow prompts up to **50,000** (`canvasCompiler.ts` and `canvasGenerator.ts`), but when loading saved canvas state (`hooksCanvasJson`) you truncate *all* strings to 10k. That means:

- A user can import / author a 50k prompt (valid),
- Canvas saves it to `hooksCanvasJson`,
- Next load from `hooksCanvasJson` truncates it to 10k,
- Next compile overwrites `hooksJson` with the truncated prompt.

**Fix ideas:**
- Per-field limits (prompt 50k, command 10k, url 2k, etc.), not one global cap.
- Or again: avoid storing hook payload in canvas JSON; regenerate from hooks JSON.

---

## HIGH

### 1) **HookCanvas does not react to prop changes; `profileId` keying is unsafe when `profileId` is optional**
**Where:** `HookCanvasInner` mount effect has `[]` deps + `initialized.current` guard; `HooksTab` uses `key={profileId}`  
**Impact:** If `profileId` is `undefined` (new profile / draft) and the user switches between two drafts that both have no id, **React reuses the same HookCanvas instance**, and HookCanvas will **not reload** new `hooksJson` / `hooksCanvasJson`. You can end up editing/saving the wrong profile’s hooks.

**Fix ideas:**
- Key by something always-changing per draft (e.g. `profileId ?? draftTempId`).
- Or make HookCanvas respond to `hooksJson`/`hooksCanvasJson` changes (and reset `initialized.current` accordingly).

---

### 2) **Canvas can “rewrite” imported JSON structure (group semantics)**
**Where:** `canvasCompiler.ts`  
**Problem:** For “Trigger → Action directly”, compiler emits **one group per action**:
```ts
groups.push({ hooks: [hookEntry] });
```
But original Claude hooks format supports a single group with multiple hooks (no matcher). If user imports JSON that has:
```json
{ "matcher": null, "hooks": [ {...}, {...} ] }
```
Canvas generator will render both actions connected to the trigger, but compiler will later re-export as multiple groups, potentially changing ordering/evaluation semantics (depending on runtime behavior).

**Fix ideas:**
- Preserve grouping for no-matcher groups: represent “group” explicitly in canvas, or keep a group node.
- Or compile direct trigger actions into a single group when they originated from the same group.

---

### 3) **Non-deterministic hook order / diff noise**
**Where:** `canvasCompiler.ts` adjacency traversal uses raw `edges` iteration order  
If edge insertion order changes (ReactFlow internal behavior, delete+readd, etc.), `compileCanvasToHooks()` can reorder groups/hooks. That creates “random” diffs and can subtly change execution order if the runner preserves array order.

**Fix ideas:**
- Sort connected targets deterministically (e.g., by node.position.y then id).
- Or store an explicit ordering index on edges/nodes.

---

### 4) **Max node limit can be exceeded with rapid drops (stale closure)**
**Where:** `HookCanvas.tsx` `onDrop` checks `if (nodes.length >= MAX_NODE_COUNT)` using captured `nodes`  
If you drop nodes rapidly, React state batching can make `nodes.length` stale, allowing >200 nodes before the guard catches up.

**Fix ideas:**
- Enforce in the functional updater:
```ts
setNodes(nds => nds.length >= MAX ? nds : [...nds, newNode])
```

---

### 5) **Invalid/unsafe edges in saved canvas are not validated against connection rules**
**Where:** `isValidCanvasState()` only checks dangling edges, not type compatibility  
A saved canvas can contain trigger→trigger, condition→condition, etc. You’ll load it, render weird edges, and compiler will silently ignore parts of the graph (partial export, surprising behavior).

**Fix ideas:**
- Validate edges using `connectionRules.isValidConnection()` during load and drop invalid edges.
- Optionally show a “canvas repaired” warning.

---

### 6) **Incomplete action nodes disappear from exported hooks JSON (surprising data loss)**
**Where:** `buildHookEntry()` returns `null` for empty command/url/prompt/description  
Users can create an action node and later export JSON (or any structural change triggers compile) and that node simply won’t exist in the output—without UI feedback. If they switch to JSON view, it looks like their node “vanished”.

**Fix ideas:**
- Keep invalid hooks as disabled placeholders in JSON (if format allows), or
- Block compile/export with validation errors, or
- Mark nodes as invalid and don’t overwrite `hooksJson` until fixed.

---

## MEDIUM

### 1) `crypto.randomUUID()` may crash in some environments
**Where:** `HookCanvas.tsx` `id: ${nodeType}-${crypto.randomUUID()}`  
If `crypto.randomUUID` isn’t available, drag/drop throws and breaks the canvas.

**Fix:** fallback to a simple UUID polyfill or incrementing counter.

---

### 2) `URL.revokeObjectURL(url)` immediately after `a.click()` can break downloads in some browsers
**Where:** `HooksTab.tsx` `handleExport`  
**Fix:** revoke in a `setTimeout(..., 0|1000)`.

---

### 3) Palette icons render as literal entities
**Where:** `NodePalette.tsx` passes `"&#9889;"` / `"&#9881;"` as strings  
These show up literally instead of as symbols (minor UI break).

---

### 4) Selection-only changes can trigger `saveLayout` → `onChange` noise
**Where:** fingerprint ignores `selected`, so selection changes look like “position-only change”  
Result: unnecessary persistence writes (every select/deselect schedules a 2s save).

**Fix:** detect true position change (compare positions) rather than “not structural”.

---

### 5) `isValidCanvasState` allows `data` arrays/odd objects; sanitizer then mutates meaningfully
This increases chances of “valid canvas json” producing unusable nodes (missing required fields) with no user-facing error.

---

### 6) Dangerous-hook detection inconsistently uses parsing vs substring search
**Where:** `HookCanvas.tsx` `hooksAlreadyDangerous = hooksJson.includes('"type":"command"')...`  
This can miss cases with spacing/newlines and cause extra prompts or incorrect assumptions.

**Fix:** parse once and inspect types (like `HooksTab` does).

---

## LOW

### 1) Importing invalid JSON fails silently
**Where:** `HooksTab.tsx` `handleImport` catch block  
User gets no error, thinks import did nothing.

### 2) Trigger `<select value={data.event}>` can be `undefined`
React controlled/uncontrolled warnings; minor but noisy.

Not huge, but on large JSON it’s avoidable (memoization already helps, but still multiple separate parses).
