## 1. `name-generator.ts` (#244)
> **Change:** Replaced food-name random generator with timestamp formatter `'Mar 20, 2:34 PM'`

### Implementation:
```ts
export function generateRandomName(): string {
  return new Date().toLocaleDateString('en-US', {
    month: 'short',
    day: 'numeric',
    hour: 'numeric',
    minute: '2-digit',
  });
}
```
#### Correctness
- The function returns a date/time stamp like `Mar 20, 2:34 PM` as specified.
- This relies on the user’s local system clock and locale.

#### Edge Cases
- Always `new Date()`: no support for custom timestamps or reproducibility.
- No seconds; may have collisions if many are created within the same minute.
- Always in English (`en-US`) — matches stated intent.
- Generates a string, never empty.
- No possibility for failure in normal use.

#### Regressions 
- The API signature remains unchanged (`(): string`), so all call sites expecting a random string should continue to work.
- No more food names—if code elsewhere relies on `generateRandomName()` to output a fun name rather than a timestamp, it’s a regressional break in semantics, not API. Nothing in the code shown relies on food name format, so it's safe.

#### Assessment
- Correct, robust, and matches the new requirements. No major risks.

## 2. `McpTab.tsx` (#247)
> **Change:** MCP server config now includes a form UI (server list + add-server form with name/command/args/env fields) above raw JSON textarea. (Previously only raw JSON?)

### Implementation Overview:
- The UI displays:
    - List of current servers (from decoded JSON), with "Remove" buttons.
    - A form to add a new server (name, command, args, env vars) with validation.
    - Direct JSON editing in the textarea below, with live validity indication.
- All internal server data is merged into/exported as JSON using helper functions for round-trip.

### Correctness

#### Server List (from JSON)
- Parsing is safe (`parseServersFromJson`) with error fallback (`[]`).
- Only the `mcpServers` key is considered, matching action domain.

#### Add Server Form
- Non-empty name/command required.
- Duplicate names are rejected.
- Args: comma-separated string, split into string array for the config.
- Env vars: an array of `{key,value}` pairs, processed into a dict for the final JSON.
- After add, form resets.
- Remove server: by name, per server in list.

#### Raw JSON Handling
- Textarea is always authoritative for JSON (single source of truth).
- Any edit in form merges into JSON, and textarea gets updated.
- Direct edting of JSON is supported; the parsed data for the UI (list, etc.) is always derived from the latest JSON value.
- JSON validation is run on every change, and the validation badge updates accordingly.

#### UX/Edge Cases
- Malformed JSON disables the server list and form cannot update it until repaired (because the parsed server list will be empty).
- If two servers with same name tried, it's blocked by form logic.
- Adding/removing servers via form updates the JSON correctly, and vice versa.
- Arg and env parsing is robust (trims/filters blanks).
- Always at least one env var row in the form (cannot delete the last row).

### Regression/Compatibility
- Users can now configure via form or JSON: strictly an enhancement.
- There’s no loss of prior behavior. Existing `mcpServersJson` configs stay valid.
- Edge: if someone hand-edits the JSON into an invalid structure (e.g., omits `mcpServers` key), the list and form view just shows empty/blank, with no crash.

### Potential UX Issue (Not a bug, but for awareness)
- If user edits both form and raw JSON simultaneously, form always merges into JSON, so JSON textarea is always up to date. Directly editing the JSON can overwrite the underlying server list, so switching between the two rapidly may cause some confusion. But this is the desired relationship based on the dual-edit approach.
