## CRITICAL

### 1) Debounced canvas saves are **canceled on unmount** → easy, silent data loss
**Where:** `HookCanvas.tsx` unmount effect: `return () => { saveCanvas.cancel(); };`  
**Why it breaks:** Any canvas edits within the 500ms debounce window are never propagated to `onChange` if the component unmounts (switching JSON/Canvas tabs, navigating away, closing a modal, profile re-render keyed by `profileId`, etc.). The UI gives no indication changes were dropped.

**Repro:**
1. Go to Canvas view.
2. Drag a node (or edit config).
3. Immediately click “JSON” tab (<500ms).
4. Your last canvas change often won’t appear in JSON; draft won’t include it.

**Impact:** Users lose work without warning; can also produce partial/inconsistent `hooksJson` vs `hooksCanvasJson`.

**Fix:** On unmount, **flush** pending debounced work instead of canceling (or flush when switching view). If cross-profile writes are the worry, detect profile change and only cancel in that case; for normal unmounts, flush.

---

### 2) Canvas view can silently drop hooks when JSON is valid but schema is not what generator expects
**Where:** `HooksTab.tsx` (gating) + `canvasGenerator.ts` (skips non-array groups)  
**Why it breaks:** `hasUnsupportedHooks` only iterates `Object.values(hooksObj)` and **ignores** any event where the value is not an array (`if (!Array.isArray(groups)) continue;`). That means malformed (but valid JSON) hooks structures won’t trigger the “unsupported” warning/confirmation.  
Then `generateCanvasFromHooks()` will skip those entries and may return `null` (empty canvas). Editing canvas afterwards overwrites `hooksJson` with whatever the new canvas compiles, effectively **deleting the original content** with no warning.

**Repro JSON (valid JSON, wrong shape):**
```json
{ "hooks": { "PreToolUse": { "matcher": "Bash", "hooks": [{ "type": "command", "command": "echo hi" }] } } }
```
- `hasUnsupportedHooks` likely returns false (it never inspects that object)
- Canvas becomes empty; next edit overwrites hooks to `{ "hooks": { ...new stuff... } }`

**Impact:** Silent destructive overwrite of user hooks content.

**Fix:** Treat any non-array event value as “unsupported/invalid for canvas” and block or require explicit confirmation (“Canvas cannot represent this JSON; editing will overwrite it.”). Same for groups where `hooks` is missing/non-array if you want to preserve.

---

### 3) If `hooksCanvasJson` and `hooksJson` diverge, layout-only edits can overwrite hooks with **stale** `hooksJson`
**Where:** `HookCanvas.tsx` initialization path when `hooksCanvasJson` loads successfully  
**Why it breaks:** When loading canvas from `hooksCanvasJson`, you set:
```ts
lastCompiledHooksJsonRef.current = hooksJson ?? compileCanvasToHooks(...)
```
If `hooksJson` is non-null but doesn’t correspond to the loaded canvas, then **moving nodes** (a non-structural change) triggers save with:
- `canvasJson` from current nodes/edges (updated positions)
- `hooksJson` reused from `lastCompiledHooksJsonRef` (stale `hooksJson`)

So a simple drag can push an out-of-sync hooks JSON back upstream.

**Impact:** Corrupts draft by reasserting stale hooks content while updating canvas positions, making it harder to detect/repair divergence.

**Fix:** When canvas is loaded from `hooksCanvasJson`, set `lastCompiledHooksJsonRef` to `compileCanvasToHooks(loadedCanvas)` (or mark `structuralDirtyRef=true` once) unless you have proven `hooksJson` matches.

## HIGH

### 1) “Unsupported hooks” confirmation sticks even if hooks JSON changes later
**Where:** `HooksTab.tsx` state `unsupportedConfirmed` resets only on `[profileId]` change.  
If a user confirms once, then later edits/imports new JSON with unsupported fields/types, Canvas view becomes available without requiring a fresh acknowledgement.

**Impact:** Users can accidentally drop unsupported hooks without realizing they need to reconfirm.

**Fix:** Reset confirmation when `draft.hooksJson` changes in a way that reintroduces unsupported content (e.g., `useEffect(() => setUnsupportedConfirmed(false), [hasUnsupportedHooks])` or on any hooksJson edit/import).

---

### 2) Generator truncates large hook sets with no UI warning → destructive on edit/save
**Where:** `canvasGenerator.ts` caps at `MAX_GENERATED_NODES=200`, `MAX_EDGES=400` and truncates.  
There’s only `console.warn`, and after truncation, user edits/saves will permanently lose the omitted hooks.

**Impact:** Large profiles are silently lossy in Canvas.

**Fix:** Surface a blocking banner/modal: “Canvas truncated your hooks; editing will drop the rest.” Require confirmation or disallow canvas edit for oversized graphs.

---

### 3) `crypto.randomUUID()` hard dependency can break drag/drop in some runtimes
**Where:** `HookCanvas.tsx` onDrop: `id: `${nodeType}-${crypto.randomUUID()}``  
Some environments (older WebViews / certain Tauri configurations) may not support `crypto.randomUUID()`.

**Impact:** Dragging nodes can throw, breaking canvas creation.

**Fix:** Provide a fallback (e.g., `crypto.getRandomValues` based uuid, or a monotonic counter + timestamp).

---

### 4) Config panel inputs may become uncontrolled (React warnings / weird editing) due to `undefined` data
**Where:** `NodeConfigPanel.tsx`
- `TriggerConfig`: `<select value={data.event}>` (can be `undefined`)
- `ConditionConfig`: `<input value={data.matcher}>` (can be `undefined`)

Your sanitizers preserve `undefined` as a possible value, and loaded/legacy states may omit fields.

**Impact:** React controlled/uncontrolled warnings; potential UI weirdness when selecting/editing.

**Fix:** Always coerce to strings: `value={data.event ?? ''}`, `value={data.matcher ?? ''}` and ensure defaults on node creation/load.

## MEDIUM

### 1) Action nodes with invalid/missing `hookType` aren’t rejected at canvas-state validation time
**Where:** `isValidCanvasState()` does not validate `action.data.hookType`.  
These actions then compile to `null` and get silently dropped by `compileCanvasToHooks()`.

**Impact:** Silent hook loss; confusing “why didn’t it save?”

**Fix:** Validate `hookType` ∈ supported set in `isValidCanvasState` (or mark unsupported and force confirmation).

---

### 2) Empty/placeholder actions round-trip poorly (generator creates them, compiler drops them)
**Where:** `buildActionData()` creates a command/http/prompt/agent action even if required fields are empty; `buildHookEntry()` requires non-empty and will drop them.

**Impact:** User sees nodes, but saving can delete them from hooks JSON; inconsistent mental model.

**Fix:** Either (a) prevent creating invalid action nodes, or (b) compile placeholders into hooks JSON (if allowed), or (c) visually flag invalid nodes and block save/compile until fixed.

---

### 3) Unknown trigger events aren’t editable/visible correctly in the Trigger select
**Where:** `TriggerConfig` renders only `HOOK_EVENTS` options.  
If hooks JSON contains an unknown event key, generator will create it, but the select won’t show that value as an option.

**Impact:** Users can’t see/preserve the exact unknown value via UI; they may accidentally change it.

**Fix:** Include the current value as an option if it’s not in `HOOK_EVENTS`, or switch to a combobox allowing custom values.

---

### 4) Shared condition nodes across multiple triggers are allowed and can duplicate effects in compiled output
**Where:** Connection rules allow multiple triggers → same condition; compiler traverses per-trigger.  
One condition feeding actions can be compiled into multiple events with identical hook groups.

**Impact:** Surprising behavior: editing one action changes multiple event flows; hard to reason about.

**Fix:** Either disallow multi-inbound to condition nodes, or clearly indicate shared nodes, or duplicate the subtree per trigger when compiling.

---

### 5) No deduplication of identical edges; visual duplicates possible
**Where:** `onConnect` doesn’t prevent same `source`→`target` being added repeatedly (only action inbound is limited).  
Compiler avoids duplicating hooks via `processedActionIds`, but UI can show multiple edges.

**Impact:** Confusing canvas visuals, harder selection/deletion.

**Fix:** Prevent adding an edge if an identical `(source,target)` already exists.

---

### 6) Import errors are silently ignored
**Where:** `HooksTab.handleImport` catch block does nothing.  
Invalid JSON just “does nothing”, looks broken.

**Impact:** User confusion; repeated attempts; potential support burden.

**Fix:** Show an error toast/dialog: “Invalid JSON file.”

---
