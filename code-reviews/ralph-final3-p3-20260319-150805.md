## HIGH

### 1) Canvas → hooks compiler breaks grouping semantics for **trigger → action** paths
**Where:** `src/lib/canvasCompiler.ts` (`compileCanvasToHooks`), “Trigger → Action directly” branch  
**What happens:** For actions connected directly to a trigger, the compiler emits **one HookGroup per action**:

```ts
groups.push({ hooks: [hookEntry] });
```

But:
- The file comment says: “Trigger → Action(s) directly: creates a group WITHOUT a matcher.” (implying potentially multiple hooks in one group).
- `generateCanvasFromHooks` *does* allow a single matcherless group to contain multiple hooks, and it will generate multiple trigger→action edges for that group.
- Roundtrip `hooksJson -> canvas -> hooksJson` will therefore **restructure** matcherless groups: a single group with N hooks becomes N groups with 1 hook each.

**Impact:** Not a mere formatting change—this mutates the JSON structure and may change semantics now or in the future (if group-level properties beyond `matcher` exist / are added). It is also inconsistent with the stated rule in the compiler comment.

**Fix direction:** For direct trigger→action connections, compile them into **one** matcherless group containing all those hook entries (in a deterministic order), instead of one group per action.

## MEDIUM

### 2) Hook ordering is not deterministic / not tied to canvas layout
**Where:** `src/lib/canvasCompiler.ts` adjacency traversal order  
**What happens:** The compiler uses:
- trigger outgoing list: `adjacency.get(trigger.id)` (edge insertion order)
- condition outgoing list: `adjacency.get(condition.id)` (edge insertion order)

No ordering by node `position.y` (or any other stable ordering) is applied. Edge array order can change due to internal ReactFlow behavior, edits, import ordering, etc.

**Impact:** Hook execution order may shift unexpectedly across edits/saves, even if the user only rearranges nodes visually or if edge ordering changes. For hooks, order often matters.

**Fix direction:** Before emitting hooks, sort each node’s outgoing targets by something stable and user-meaningful (commonly `targetNode.position.y`, then id as tiebreaker).

### 3) Loading from `hooksCanvasJson` may preserve a stale `hooksJson` even when canvas is authoritative
**Where:** `src/components/agent-builder/HookCanvas.tsx` (mount load effect)  
On successful canvas JSON load, the code sets:

```ts
lastCompiledHooksJsonRef.current = hooksJson ?? compileCanvasToHooks(...)
```

So if `hooksJson` is present (even if inconsistent with canvas JSON), it becomes the “canonical” hooks text. Later, if the user only performs **layout-only changes** (node dragging), `saveCanvas()` will not recompile (because `structuralDirtyRef` is false) and will keep emitting that stale `hooksJson`.

**Impact:** Canvas state and hooks JSON can diverge silently if the stored pair is inconsistent (e.g., legacy data, backend mismatch). Layout edits can cause saving canvas JSON while leaving hooks JSON incorrect.

**Fix direction (any of):**
- On canvas JSON load, **always** set `lastCompiledHooksJsonRef` by compiling the loaded canvas (and/or mark `structuralDirtyRef.current = true` once) so the next save is consistent.
- Or explicitly define precedence rules and enforce them (canvas vs hooks source of truth).

### 4) Condition nodes with empty matcher compile to matcherless groups; roundtrip drops the condition concept
**Where:** `canvasCompiler.ts` condition branch + `canvasGenerator.ts` matcher truthiness check  
Compiler: emits `matcher` only if `condData.matcher` truthy.  
Generator: creates a condition node only if `(group as HookGroupInput).matcher` is truthy.

**Impact:** A condition node that exists in canvas but has an empty matcher will compile to a matcherless group; regenerating from hooks JSON will not recreate that condition node. This is a structural “data loss” in the canvas representation.

**Fix direction:** Either:
- disallow/flag empty matcher conditions in the UI, or
- persist empty matcher explicitly (if allowed by the spec), and treat empty string as “matcher present” consistently in generator/compiler.

## LOW

### 5) Palette icons passed as HTML entities in strings render literally
**Where:** `NodePalette.tsx`  
Example: `icon="&#9889;"` is a string; JSX won’t decode it, so it likely renders `&#9889;` not ⚡. (Trigger/Condition nodes do it correctly by placing `&#9889;` directly in JSX text.)

**Fix:** Use actual characters (`'⚡'`, `'⚙'`) or pass ReactNode and render it.

---

### 6) Generator doesn’t guard against dangerous event keys (minor consistency/security hardening)
**Where:** `canvasGenerator.ts` iterating `Object.entries(hooks)`  
Compiler blocks `__proto__` etc. Generator does not. This can lead to creating trigger nodes that later cannot compile (compiler will skip), and is inconsistent with the rest of the codebase’s hardening approach.

**Fix:** Skip or sanitize dangerous event keys in `generateCanvasFromHooks` similarly to compiler/HookCanvas.

---

### 7) HookCanvas mount effect intentionally ignores dependencies (fragile if keying/remount assumptions change)
**Where:** `HookCanvas.tsx` load effect with `[]` while referencing props  
Currently mitigated by `key={profileId ?? ...}` and conditional rendering, but if reused without remounting, it will not react to prop changes.

**Fix:** Either document the remount requirement clearly or include dependencies and handle re-init safely.
