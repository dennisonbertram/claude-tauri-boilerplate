## Overall assessment
The React Flow → hooks JSON pipeline is close, but the **HookCanvas state→persistence logic is currently unsound**: it can **revert user edits** and/or **rewrite hooks JSON without explicit user action**. There are also **runtime crash vectors** from insufficient runtime type-guards in the compiler, and several graph invariants are not enforced on load.

---

## CRITICAL

### 1) `layoutSave` runs for *structural* changes and can overwrite freshly compiled hooks JSON (data loss)
**File:** `src/components/agent-builder/HookCanvas.tsx`

You intend:
- `notifyChange` (500ms) for *structural* changes (nodes/edges/data)
- `layoutSave` (2000ms) for *position-only* changes

But due to effect ordering + the way `prevStructuralRef` is updated, this condition is wrong:

```ts
const fingerprint = getHooksFingerprint(nodes, edges);
if (fingerprint === prevStructuralRef.current) {
  layoutSave(nodes, edges, hooksJson);
}
```

Because the “structural change” effect runs first and does:
```ts
prevStructuralRef.current = fingerprint;
notifyChange(nodes, edges);
```

…the subsequent layout effect sees `fingerprint === prevStructuralRef.current` as **true even for structural edits**, so **`layoutSave` is scheduled for essentially every change**.

Worse, `layoutSave` is called with `hooksJson` from props at scheduling time, which can be **stale** relative to the just-compiled hooks JSON. Typical failure sequence:
1) User adds/edits nodes → schedules `notifyChange` (500ms) and also schedules `layoutSave` (2000ms) with *old* `hooksJson`
2) `notifyChange` runs first → parent `onChange(newHooksJson, newCanvasJson)`
3) ~1.5s later `layoutSave` runs → parent `onChange(oldHooksJson, latestCanvasJson)` **reverting hooks JSON**

This is a **reproducible corruption/data-loss** bug.

**Fix direction (one of):**
- Track a `structuralChangedThisCommit` ref set in the structural effect, and in the layout effect do:
  - if `structuralChangedThisCommit` then clear & skip layout save
  - else layout save
- Or compute a separate fingerprint that only includes `{id, position}` and compare that for layout-only changes.
- Or change `onChange` API to allow updating canvas layout without rewriting hooks JSON (recommended: `onChange({hooksJson?:..., hooksCanvasJson?:...})`).

## HIGH

### 2) Opening Canvas view can rewrite `hooksJson` (and drop unsupported hooks) without any user edit
**File:** `src/components/agent-builder/HookCanvas.tsx`

After initial mount, you load from `hooksCanvasJson` or generate from `hooksJson`, which updates `nodes/edges`. That triggers `notifyChange`, which compiles and calls `onChange(...)` even though the user didn’t change anything.

Impact:
- Merely switching JSON → Canvas can:
  - reformat hooks JSON
  - **drop unsupported hook types immediately** (your UI warning says “Editing in Canvas view will drop…”, but *viewing* already triggers it)

**Fix direction:**
- Add an initialization gate: after loading/generating nodes, set `prevStructuralRef` to the loaded fingerprint and **skip the first structural notify**.
- Alternatively, only call `notifyChange` in response to explicit user actions (connect, drop, config edit), not from generic `nodes/edges` effects.

---

### 3) Pending debounced changes are canceled on unmount → last edits can be lost
**File:** `src/components/agent-builder/HookCanvas.tsx`

Cleanup does:
```ts
notifyChange.cancel();
layoutSave.cancel();
```

If the user edits and quickly switches tabs/views (unmount), the debounced commit never reaches the parent, so edits vanish.

**Fix direction:**
- Implement `flush()` in your debounce helper and flush on unmount if dirty, or
- Do not cancel; allow the debounced `onChange` to run after unmount (it’s a prop callback, not setState), or
- Immediately commit on critical interactions (edge add/remove, node add/remove, config changes).

---

### 4) `compileCanvasToHooks` can throw at runtime for malformed condition data (`matcher.slice(...)`)
**File:** `src/lib/canvasCompiler.ts`

This line is unsafe without a runtime type check:
```ts
if (condData.matcher) {
  group.matcher = condData.matcher.slice(0, MAX_MATCHER_LENGTH);
}
```

If `hooksCanvasJson` contains `{ matcher: true }` (or any non-string truthy), compilation throws, which can break the canvas tab.

**Fix direction:**
```ts
if (typeof condData.matcher === 'string' && condData.matcher.trim()) {
  group.matcher = condData.matcher.slice(0, MAX_MATCHER_LENGTH);
}
```

Also consider validating/sanitizing node.data on load, not just on drag/drop.

## MEDIUM

### 5) Loaded canvas graphs are not normalized to enforced invariants (connection rules, single-incoming, edge uniqueness)
**File:** `src/components/agent-builder/HookCanvas.tsx`

`isValidCanvasState` checks basic shape + dangling edges, but **does not** enforce:
- connectionRules (`trigger→condition/action`, `condition→action`, etc.)
- “action has ≤1 incoming edge” invariant (enforced in UI, not enforced on load)
- unique edge ids (React Flow can behave oddly with duplicates)

This can lead to:
- confusing rendering
- non-deterministic compilation (compiler de-dupes some cases but not a full normalization)

**Fix direction:** after parsing saved state, filter edges through `isValidConnection(...)` and enforce the single-incoming constraint, plus ensure edge id uniqueness.

---

### 6) Order/semantics drift: un-matched groups with multiple hooks become multiple groups
**Files:** `canvasGenerator.ts` + `canvasCompiler.ts`

Generator allows a group `{ hooks: [a,b,c] }` (no matcher) and maps to multiple action nodes connected directly to trigger. Compiler always emits **one group per direct Trigger→Action** connection. This is likely semantically equivalent in the current hook schema, but it’s not a stable round-trip and could matter if group-level properties are added later.

---

### 7) `compileCanvasToHooks` doesn’t require `event` to be a string
**File:** `src/lib/canvasCompiler.ts`

`const event = data.event; if (!event) continue;` allows non-string truthy values. Those become keys via string coercion.

**Fix direction:** `if (typeof event !== 'string' || !event.trim()) continue;`

---

### 8) Header key sanitization misses proto-pollution keys
**File:** `src/lib/canvasCompiler.ts`

You sanitize CRLF/length, but don’t exclude `__proto__`/`constructor`/`prototype` in header keys, while assigning into `{}`:
```ts
sanitizedHeaders[k] = v;
```

Add `DANGEROUS_KEYS` check (or use `Object.create(null)` for `sanitizedHeaders`).
