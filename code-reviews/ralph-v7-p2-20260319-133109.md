## CRITICAL

### 1) Debounced “structural save” vs “layout save” race can overwrite newer positions / desync `hooksJson` vs `hooksCanvasJson`
**Where:** `HookCanvas.tsx` (`notifyChange` @500ms, `saveLayout` @2000ms, fingerprint ignores `position`)  
**Impact:** If the user makes a structural change (e.g., edit a field) and then drags nodes before the 500ms compile fires, `notifyChange(nodesSnapshot, edgesSnapshot)` will later serialize **stale positions** into `hooksCanvasJson`, overwriting the more recent drag positions. If the user closes/navigates before `saveLayout` fires, the latest layout can be lost. Even worse, `hooksJson` and `hooksCanvasJson` can become inconsistent for a period (old hooks + new layout or new hooks + old layout) depending on timing.

**Repro:**
1. In canvas, edit an Action prompt/command (structural change; schedules `notifyChange`).
2. Immediately drag a node to a new location (position-only change; schedules `saveLayout`).
3. Wait ~0.5s (compile fires) then close quickly before 2s elapses.
4. Reopen: layout may revert (compile overwrote canvas JSON with old positions).

**Why it happens:** `notifyChange` and `saveLayout` each capture *different snapshots* of nodes/edges at scheduling time, and `notifyChange` serializes positions even though it’s intended for structural changes.

**Fix ideas:**
- Keep a `latestNodesRef/latestEdgesRef` updated on every render/change; in debounced callbacks, **read from refs**, not arguments.
- Or: don’t serialize positions in `notifyChange` at all; only update `hooksJson` there, and let a single unified debounced writer persist canvas layout.
- Or: when any node change occurs (including position-only), cancel pending `notifyChange` and reschedule it with newest state (even if compilation ignores positions, canvas JSON shouldn’t).

### 2) Missing/invalid hook `type` defaults to `"command"` in generator, bypassing “dangerous hook” warnings and can create executable command hooks unexpectedly
**Where:** `canvasGenerator.ts` → `buildActionData()`:
```ts
const type = typeof hook.type === 'string' ? hook.type : 'command';
```
**Impact:** A hooks JSON with missing `type` (but containing `command`) is treated as a **command hook** in Canvas. This is especially problematic because:
- `HooksTab.hasDangerousHooks` and `handleImport` warnings only trigger on `hook.type === "command"/"http"` (or string-contains checks), so **missing `type` can evade warnings**.
- If the user makes *any* structural edit in canvas, `compileCanvasToHooks` will write back explicit `"type":"command"` hooks, potentially turning previously-nonfunctional/ambiguous config into an explicitly dangerous executable hook set.

**Repro:**
1. Paste/import:
   ```json
   { "hooks": { "PreToolUse": [ { "matcher":"Bash", "hooks":[ { "command":"rm -rf ~" } ] } ] } }
   ```
2. No “dangerous hooks” warning (because `type` absent).
3. Switch to Canvas; a command action appears.
4. Make a minor structural edit (e.g., change matcher); hooks JSON is recompiled and now contains explicit `"type":"command"`.

**Fix ideas:**
- Do **not** default missing `type` to `"command"`. Prefer: return `null` (skip) or mark as unsupported and force JSON-view preservation.
- Extend warnings to detect “command-like” hooks even if `type` is missing (e.g., presence of `command`/`url` fields).
- On canvas entry, show a confirmation when dangerous hooks are inferred/normalized.

## HIGH

### 1) `getHooksFingerprint()` stringifies full node `data` (including huge prompts) on every drag → severe UI jank / near-freeze
**Where:** `HookCanvas.tsx`:
```ts
JSON.stringify(nodes.map(n => ({ id, type, data }))) ...
```
**Impact:** With prompts up to 50k (and currently unbounded via config panel), every node movement causes expensive stringify/allocations. Dragging can become unusable with surprisingly few large-text nodes.

**Fix ideas:** Fingerprint only structural identifiers:
- Include `{id,type,hookType,event,matcher}` and **exclude** large text fields (`prompt`, `command`, `description`, `label`) and anything not affecting compilation topology.
- Or maintain a lightweight “dirty structural version” incremented on relevant edits.

---

### 2) Config panel updates bypass `sanitizeNodeData` limits → users can paste megabytes into state
**Where:** `HookCanvas.tsx` `onUpdateNode` merges raw `newData` directly.  
**Impact:** The app’s intended caps (`MAX_PROMPT_LENGTH`, etc.) are not enforced on user edits, only on drag-drop/load. This enables accidental or intentional state bloat and worsens the fingerprint performance issue above. Compiler later truncates, but the UI/state can already be huge.

**Fix ideas:** Apply `sanitizeNodeData(newData)` (and possibly per-node schema validation) inside `onUpdateNode`.

---

### 3) Canvas editing silently drops unsupported *fields* and unknown top-level JSON (not just unsupported hook types)
**Where:** `compileCanvasToHooks()` and `generateCanvasFromHooks()` only understand a narrow schema.  
**Impact:** Even when hook `type` is supported, additional fields (e.g., `body`, `env`, `cwd`, future Claude Code hook fields, or any metadata) will be lost after any structural canvas edit. `HooksTab` warns only about unsupported *types*, not unsupported *properties*.

**Fix ideas:**
- On entering canvas view, diff the parsed JSON against supported schema and warn/confirm if extra keys exist anywhere (top-level, group-level, hook-level).
- Alternatively preserve unknown fields in node `data` and round-trip them (harder but safest).

---

### 4) `hasUnsupportedHooks` can throw on non-array `group.hooks`, returns false, and bypasses the “will drop unsupported hooks” warning
**Where:** `HooksTab.tsx`:
```ts
if (!group?.hooks) continue;
for (const hook of group.hooks) { ... } // can throw if not iterable
```
**Impact:** A valid JSON with malformed shape (e.g., `hooks: {PreToolUse:[{hooks:{...}}]}`) disables warning due to catch→false, then canvas view can overwrite and effectively wipe content on edit.

**Fix ideas:** Use `Array.isArray(group.hooks)` consistently (like you did in `hasDangerousHooks`).

---

### 5) Condition nodes with empty matcher compile into groups with no matcher → round-trip destroys the condition semantics
**Where:** `canvasCompiler.ts`:
- If condition has hooks but `condData.matcher` is empty, group is emitted *without* `matcher`.
**Impact:** A user can build `Trigger → Condition(empty) → Action`. JSON output has no matcher, and on regeneration it becomes `Trigger → Action` (condition node disappears). This is confusing and causes structural churn.

**Fix ideas:**
- Enforce matcher required for condition groups (don’t compile a condition path unless matcher is non-empty).
- Or treat empty-matcher condition as invalid UI state (block connecting actions until matcher set).

## MEDIUM

### 1) Generator creates Trigger nodes for events even if groups/hook lists are empty/invalid; compiler later drops them
**Where:** `canvasGenerator.ts` always pushes a trigger per event key before checking if any valid hooks exist.  
**Impact:** Canvas can show “phantom” triggers that will never persist back to JSON if recompiled (events disappear), confusing users.

**Fix:** Only create a trigger after at least one valid hook/group is discovered, or show explicit “empty event” UI and preserve it intentionally.

---

### 2) Generator creates Action nodes for incomplete hooks (empty `command` / empty `url` / empty `prompt`), but compiler skips them
**Where:** `buildActionData()` returns command/http/prompt nodes even if required content is empty.  
**Impact:** Canvas shows nodes that evaporate from JSON on save, causing “why did my hook disappear?” reports.

**Fix:** In generator, skip hooks missing required fields (mirror compiler validation).

---

### 3) Import dangerous-hook detection via string-contains is brittle (false negatives/positives)
**Where:** `HooksTab.handleImport()`:
```ts
const hasCommandHooks = stringified.includes('"type":"command"');
```
**Impact:** This misses formatting variations (spacing, different ordering, etc.) and can false-positive if those substrings appear inside prompt text. You already parsed JSON—string search is unnecessary.

**Fix:** Traverse parsed object (like `hasDangerousHooks`) and detect semantically.

---

### 4) Edge constraints are enforced on connect, but not validated on load → loaded canvases can compile unpredictably
**Where:** `HookCanvas` load from `hooksCanvasJson` sanitizes but doesn’t enforce invariants (e.g., action single-inbound).  
**Impact:** A saved canvas with multiple inbound edges to an Action can yield non-deterministic grouping depending on edge order and `processedActionIds` behavior.

**Fix:** Normalize/validate edges on load (dedupe edges, enforce “action has ≤1 inbound”, etc.), or surface an error banner.

## LOW

### 1) Palette icons rendered as literal HTML entities
**Where:** `NodePalette.tsx` passes `"&#9889;"` as a string prop; React won’t decode it.  
**Impact:** Cosmetic but looks broken.

**Fix:** Pass the actual character (`"⚡"`, `"⚙"`) or render entity directly in JSX, not as a string.

---

### 2) File import creates an `<input>` element and never removes it
**Where:** `HooksTab.handleImport()`  
**Impact:** Minor DOM leak over many imports.

**Fix:** `input.remove()` after use, or keep a hidden input in JSX.

### 3) Delete key only bound to `"Delete"` (no Backspace)
**Where:** `ReactFlow deleteKeyCode="Delete"`  
**Impact:** UX inconsistency; laptop users often expect Backspace.

**Fix:** Provide both or use ReactFlow’s recommended key handling.
