## Security review: Visual Hook Builder (ReactFlow canvas ↔ hooks JSON)

### 1) CRITICAL — “Dangerous hook” safety gating is bypassable (leads to RCE/SSRF when agent runs)
**Where**
- `src/components/agent-builder/tabs/HooksTab.tsx` (`hasDangerousHooks` warning)
- `src/components/agent-builder/HookCanvas.tsx` (no confirm/warn when *loading* existing hooks into canvas)
- `src/lib/canvasGenerator.ts` (generates command/http action nodes from hooks without any user confirmation)

**Impact**
If profiles/hooks can come from untrusted sources (shared profiles, synced profiles, downloaded templates, etc.), an attacker can inject **command** or **http** hooks that execute on the victim machine when the agent runs. The UI’s warning is not robust and can be bypassed, resulting in **silent inclusion of execution hooks**.

**Exploit sketch**
- Paste hooks JSON that contains command hooks but avoids the exact substrings checked by `hasDangerousHooks`, e.g.:
  - Different whitespace: `"type"\n:\n"command"`
  - Tab/CRLF variants: `"type"\t:\t"command"`
  - Other JSON formatting variants not covered by the 4 `includes(...)` checks
- Victim sees no warning banner (false negative), switches to Canvas, edits/saves profile, later runs agent → command executes.

**Why it happens**
- `hasDangerousHooks` does *string matching on the raw JSON text* with only a few exact patterns.
- When rendering the canvas from `hooksJson` / `hooksCanvasJson`, there is **no “this profile contains execution hooks” confirmation** (only the fragile banner in JSON view).

**Fix**
- Replace substring checks with a real parse + structural walk of `hooks.hooks[*].hooks[*].type`.
- Enforce a hard policy boundary: require explicit, non-bypassable acknowledgement before **saving** or **running** profiles containing `command`/`http`, not just on import/drag.
- Also warn when loading from `hooksCanvasJson` (saved canvas state) since it can already contain dangerous actions.

### 2) HIGH — Untrusted `hooksCanvasJson` can corrupt internal state / potential prototype pollution via IDs & keys
**Where**
- `src/components/agent-builder/HookCanvas.tsx`: `isValidCanvasState` validates `id` only by length/non-empty, does **not** block dangerous identifiers.
- `setNodes(saved.nodes)` / `setEdges(saved.edges)` loads data directly.

**Impact**
A crafted `hooksCanvasJson` can include node/edge IDs like `__proto__`, `constructor`, `prototype` or other special keys. While your code uses `Map` in some places, ReactFlow/XYFlow internals (or future code) may use plain objects keyed by IDs. That opens the door to:
- **prototype pollution** in library internals (best case: crashes/DoS; worst case: unexpected behavior/security issues)
- hard-to-debug canvas corruption (selection, deletion, edge resolution)

**Exploit sketch**
Provide a saved canvas JSON containing a node like:
```json
{"id":"__proto__","type":"trigger","position":{"x":1,"y":1},"data":{"event":"PreToolUse"}}
```
If any internal lookup uses `{[id]: node}`, it can mutate prototype chains or break lookups.

**Fix**
- Extend `isValidCanvasState` to reject dangerous IDs for `node.id`, `edge.id`, `edge.source`, `edge.target`:
  - block `__proto__`, `prototype`, `constructor`, and consider a conservative `^[a-zA-Z0-9_-]{1,100}$` allowlist
- Deep-sanitize `node.data` keys too (see next issues).

### 3) HIGH — Multiple DoS vectors via unbounded string sizes + expensive fingerprinting
**Where**
- `HookCanvas.tsx`
  - No size limit on `hooksJson`/`hooksCanvasJson` props
  - `getHooksFingerprint()` does `JSON.stringify` over **all node data strings** on every structural change
- `canvasGenerator.ts`: stores full `command/prompt/url/description` into node data without caps
- `onUpdateNode` doesn’t enforce `MAX_STRING_LENGTH`

**Impact**
A profile can freeze or crash the UI by supplying huge strings (e.g., multi-megabyte prompts/commands/URLs) because:
- those strings become `node.data`
- structural edits trigger repeated `JSON.stringify` of large payloads (fingerprint + `notifyChange` serialization)

**Exploit sketch**
- Paste a very large hooks JSON (no paste limit) containing a single prompt hook with multi-MB content.
- Switch to Canvas → generator loads it → any edit causes expensive stringify/compile → UI becomes unresponsive.

**Fix**
- Impose strict maximum sizes on:
  - `hooksJson` length accepted for canvas generation
  - per-field sizes at generation time (`buildActionData`) and at edit time (`onUpdateNode`)
  - `hooksCanvasJson` length before parsing
- Make fingerprinting cheap: hash stable small summaries (IDs + small capped previews), not full `data`.

### 4) MEDIUM — `hooksCanvasJson` “valid state” check is shape-only; `node.data` is not constrained (data corruption + hidden payloads)
**Where**
- `isValidCanvasState` only checks `n.data` is an object; it doesn’t enforce:
  - flat key/value schema
  - max string lengths
  - allowed keys per node type
  - absence of nested objects/arrays

**Impact**
- Hidden oversized payloads can be stored in `data` and re-serialized back into `hooksCanvasJson`, amplifying DoS risk.
- Invalid/malformed node data can cause silent drops during compilation (`buildHookEntry` returns null), creating **surprising loss of hooks**.

**Fix**
- Validate `data` by node type (trigger/condition/action) and enforce caps consistently.
- Reuse/extend `sanitizeNodeData` when *loading* saved state, not only on drag-drop.

---

### 5) MEDIUM — Silent truncation & mismatch between editor state and compiled hooks (integrity issue)
**Where**
- `canvasCompiler.ts` truncates:
  - command to 10k, prompt to 50k, url to 2k, matcher to 200, etc.
- UI (`NodeConfigPanel`) allows arbitrarily long inputs; generator stores arbitrarily long values.

**Impact**
Users can edit content that appears saved in the canvas, but the compiled `hooksJson` is silently truncated. On round-trip (JSON → canvas → JSON), this becomes **data loss** and can be abused for “what you see is not what executes” confusion.

**Fix**
- Enforce the same limits at input time and show explicit validation errors when exceeding limits.

---

### 6) LOW — Position/layout changes are not persisted (hooksCanvasJson integrity)
**Where**
- `HookCanvas.tsx`: `getHooksFingerprint` excludes `position`, so dragging nodes does not trigger `notifyChange` → updated positions aren’t saved.

**Impact**
Not a direct security issue, but it is state integrity corruption: users think layout is saved; it isn’t. This can also complicate reviews/audits of hook graphs.

**Fix**
- Persist canvas JSON on position changes without recompiling hooks (separate debounced “save canvas state” from “compile hooks”).

---

### 7) LOW — Drag-and-drop JSON parse has no input size cap (minor DoS)
**Where**
- `HookCanvas.tsx` `onDrop`: `JSON.parse(raw)` on attacker-controlled `dataTransfer` without length limit.

**Impact**
A malicious page/app could entice the user to drag an item into the canvas with a very large `application/reactflow` payload, causing a parse spike.

**Fix**
- Reject `raw` over a small threshold (e.g., 50–200KB) before parsing.
