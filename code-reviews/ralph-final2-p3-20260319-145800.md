## Audit focus
Reviewed canvas state invariants, state transitions, compilation/generation round-trips, connection constraints, and type/validation soundness across:
- `HookCanvas.tsx`
- `canvasCompiler.ts`
- `canvasGenerator.ts`
- `connectionRules.ts`
- node/panel components as they affect data correctness.

---

## HIGH

### 1) Hook execution order is not deterministic / not controllable from the canvas
**Where:** `src/lib/canvasCompiler.ts` (group/hook ordering), `src/components/agent-builder/HookCanvas.tsx` (no ordering metadata)  
**Problem:** `compileCanvasToHooks` derives hook order from the *iteration order of `edges`* via the adjacency map. React Flow edge array order is not a stable semantic ordering, and moving nodes (visual ordering) does not affect adjacency ordering.  
**Impact:** Hooks within a group (and groups within an event) are ordered in Claude hooks JSON. If ordering changes unexpectedly, runtime behavior can change even though the canvas “looks the same”. This breaks a key invariant: “canvas layout should reliably map to hook execution order.”  
**Fix:** Establish a deterministic ordering rule at compile time, e.g.:
- Sort triggers by `position.y` (then `id`) before compiling.
- For each trigger, sort connected conditions/actions by `position.y`.
- For actions under a condition, sort by `position.y`.
This requires `compileCanvasToHooks` to have access to node positions (it does, via `nodes`). You’ll also need to treat position changes as “structural” if order is position-based (see HookCanvas `structuralDirtyRef` logic).

---

### 2) `crypto.randomUUID()` can hard-crash drag/drop node creation in some runtimes
**Where:** `src/components/agent-builder/HookCanvas.tsx` (`onDrop`)  
**Problem:** Node IDs are created with `crypto.randomUUID()`. In some WebViews / older environments this may be missing or not enabled, causing a runtime exception on drop.  
**Impact:** Users cannot add nodes; canvas becomes partially unusable.  
**Fix:** Provide a safe fallback:
```ts
const uuid =
  (globalThis.crypto?.randomUUID?.() ??
   `${Date.now()}-${Math.random().toString(16).slice(2)}`);
id: `${nodeType}-${uuid}`
```

## MEDIUM

### 3) Duplicate Trigger→Condition edges are allowed and can duplicate groups in output JSON
**Where:** `HookCanvas.tsx` (`onConnect`), `canvasCompiler.ts` (iterates adjacency entries directly)  
**Problem:** `onConnect` prevents multiple inbound edges to an **action**, but does not prevent creating duplicate edges between the same `source`/`target` for other node types (notably Trigger→Condition). If duplicates exist, `compileCanvasToHooks` will process the same condition multiple times, potentially emitting duplicated groups.  
**Impact:** Exported hooks JSON can contain repeated groups/hooks, changing behavior (hooks run multiple times).  
**Fix:** In `onConnect`, prevent duplicate `(source,target)` pairs for all connections:
```ts
if (eds.some(e => e.source === params.source && e.target === params.target)) return eds;
```
Also consider de-duping in `pruneInvalidEdges` and/or compiler adjacency build.

---

### 4) HookCanvas initialization ignores subsequent prop changes (relies on parent `key`)
**Where:** `HookCanvas.tsx` init `useEffect([])` with `initialized.current`  
**Problem:** The canvas only loads `hooksJson/hooksCanvasJson` once. In this codepath, `HooksTab` uses `key={profileId}` to force remounting, but `HookCanvas` itself is not robust to prop updates if used elsewhere.  
**Impact:** Stale canvas state if the component is ever reused without remount. This is a state transition hazard.  
**Fix:** Either:
- Document the invariant “must remount on profile change”, or
- Add a controlled “reset when props change” effect keyed on `(hooksCanvasJson, hooksJson)` with appropriate confirmation handling.

---

### 5) CRLF in HTTP URL causes silent hook drop on compile (data loss)
**Where:** `canvasCompiler.ts` (`buildHookEntry` rejects URL containing CRLF) vs `HookCanvas.tsx` sanitization (does not strip CRLF for `url`)  
**Problem:** A user can type/paste a URL with `\r` or `\n`. The canvas will display it, but `compileCanvasToHooks` returns `null` for the hook, so it disappears from exported JSON without an explicit UI error.  
**Impact:** Surprising data loss; user thinks hook exists but it’s omitted at save/export.  
**Fix:** Prefer sanitizing `url` in `sanitizeNodeData` (strip CRLF) and/or show validation error in UI instead of dropping. At minimum, normalize:
```ts
if (key === 'url') result[key] = val.replace(/[\r\n]/g, '').slice(0, MAX_URL_LENGTH);
```

## LOW

### 6) Controlled form fields can receive `undefined` leading to React warnings
**Where:** `NodeConfigPanel.tsx` (`select value={data.event}`, `input value={data.matcher}`)  
**Problem:** If a malformed/partial canvas state exists (allowed by `isValidCanvasState` since it permits `event`/`matcher` undefined), these become controlled inputs with `undefined` value.  
**Impact:** Console noise + inconsistent UI state; not typically fatal.  
**Fix:** Coalesce:
- `value={data.event ?? HOOK_EVENTS[0]}` or `''`
- `value={data.matcher ?? ''}`

---

### 7) “already dangerous” detection uses brittle substring matching
**Where:** `HookCanvas.tsx` (`hooksAlreadyDangerous` uses `includes('"type":"command"')`)  
**Problem:** Formatting differences (`"type": "command"`) bypass the check and can cause extra confirmation prompts.  
**Impact:** UX annoyance; not a correctness break.  
**Fix:** Parse JSON and walk like `hasExecutionHooks` does in `HooksTab`.

---

### 8) Sanitized node data uses normal `{}` instead of null-prototype object
**Where:** `HookCanvas.tsx` (`sanitizeNodeData`)  
**Problem:** You do strip dangerous keys, which is the main protection. Using `Object.create(null)` would further reduce accidental prototype interactions.  
**Impact:** Minor hardening opportunity.  
**Fix:** `const result: Record<string, unknown> = Object.create(null);`

---

## Overall assessment
Core invariants are *mostly* enforced:
- Connection types are validated (`connectionRules`, `onConnect`, `pruneInvalidEdges`).
- Single inbound edge per action is enforced both interactively and on load.
- Node/edge caps exist in generator, validator, and editor.
- Compiler/generator are reasonably type-safe at runtime (skip malformed entries, length limits).

However, **hook ordering semantics** and **runtime ID generation** are significant enough to block approval.
