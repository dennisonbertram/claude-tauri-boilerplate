## CRITICAL

### 1) Prototype pollution via drag-and-drop node `data` (and via `sanitizeNodeData`)
**Where:** `HookCanvas.tsx` ‚Üí `onDrop()` + `sanitizeNodeData()`  
**How to trigger (abnormal but realistic in desktop/webview):**
- Drag *any external* payload (or from another app/window) that sets `application/reactflow` to JSON like:
  ```json
  {"nodeType":"action","data":{"hookType":"prompt","__proto__":"x","prompt":"hi"}}
  ```
- Drop onto canvas.

**What happens:**
- `sanitizeNodeData()` copies keys verbatim and does `result[key] = ...`.
- Assigning `result["__proto__"] = "x"` can mutate the prototype of `result` (JS `__proto__` setter behavior), causing prototype pollution / weird behavior downstream.
- You already wrote `stripDangerousKeys()` but you **don‚Äôt apply it** on drop/new-node creation (only on loading saved canvas JSON).

**Impact:**
- Potential prototype manipulation in runtime objects, undefined behavior, potential security issues depending on how other code uses objects.

**Fix:**
- In `sanitizeNodeData`, reject dangerous keys (same set you already have).
- In `onDrop`, run `stripDangerousKeys()` *before* `sanitizeNodeData()` (or integrate both).
- Also consider sanitizing in `onUpdateNode` (defense-in-depth).

## HIGH

### 2) Node position/layout changes are never persisted unless a ‚Äústructural‚Äù change happens
**Where:** `HookCanvas.tsx` ‚Üí structural fingerprint excludes `position`; `notifyChange()` only runs when fingerprint changes.  
**Normal-user repro:**
1. Switch to Canvas.
2. Drag nodes to rearrange.
3. Close/reopen profile (or switch away/back).
4. Layout snaps back.

**Cause:**
- `getHooksFingerprint()` excludes `position`, so dragging doesn‚Äôt trigger `notifyChange()`.
- But `notifyChange()` is the only place you serialize `hooksCanvasJson` back to the parent.

**Impact:** major UX + data integrity issue for a visual builder (users think layout is saved; it isn‚Äôt).

**Fix options:**
- Persist `hooksCanvasJson` on node drag end (ReactFlow has callbacks like `onNodeDragStop`).
- Or include `position` in a *separate* debounced persistence path for canvas JSON only (not hooks compilation).

### 3) Max node limit can be bypassed by rapid drops (stale closure race)
**Where:** `HookCanvas.tsx` ‚Üí `onDrop` checks `if (nodes.length >= MAX_NODE_COUNT)` using captured `nodes`.  
**Repro:**
- Rapidly drop many nodes without giving React time to re-render between drops.
- Each handler sees the same old `nodes.length` and allows adding; you can exceed 200.

**Impact:** bypass limits ‚Üí performance degradation/freezes, huge canvas state.

**Fix:**
- Enforce inside the functional updater:
  ```ts
  setNodes(nds => nds.length >= MAX ? nds : [...nds, newNode]);
  ```
- Avoid using `nodes.length` from closure for enforcement.

### 4) Serious performance risk: fingerprint JSON.stringify runs on every node position change
**Where:** `HookCanvas.tsx` ‚Üí effect recomputes `getHooksFingerprint(nodes, edges)` on *every* `nodes` change.  
**Normal-user repro:**
- Create many nodes, then drag one around: UI can lag because each mousemove updates positions ‚Üí triggers effect ‚Üí `JSON.stringify` of all node data.

**Impact:** degraded interactivity; can become unusable with bigger graphs or large text fields.

**Fix:**
- Don‚Äôt compute full fingerprint on every `nodes` update.
- Options:
  - Track ‚Äústructural changes‚Äù explicitly (reactflow change types: position vs data/edges).
  - Split persistence: canvas position persistence throttled; hooks compilation only on data/edge changes.
  - Maintain a lightweight hash for structural bits only and update it only when those bits change.

### 5) Unbounded text inputs can DOS the UI (and saved canvas load doesn‚Äôt truncate)
**Where:**

### 6) `profileId` being undefined causes state leakage + unsafe confirmation persistence
**Where:** `HooksTab.tsx`
- `HookCanvas key={profileId}`: if `profileId` is `undefined` for multiple drafts, HookCanvas won‚Äôt remount between them ‚Üí canvas state can carry across drafts.
- `unsupportedConfirmed` resets only on `[profileId]`; if undefined, it may never reset when switching between ‚Äúnew/unsaved‚Äù profiles.

**Impact:**
- Wrong canvas shown/edited for a different draft.
- User can accidentally drop unsupported hooks without seeing the confirmation again.

**Fix:**
- Key HookCanvas with something always-unique per draft (fallback to stable draft identifier, or hash of initial content).
- Reset `unsupportedConfirmed` on draft identity change, not only `profileId`.
