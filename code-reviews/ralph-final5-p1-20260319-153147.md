## HIGH — Unbounded JSON parsing → app freeze/crash (persistent DoS)
**Where**
- `HooksTab.tsx`: `hasDangerousHooks`, `hasUnsupportedHooks`, `jsonValid` all repeatedly `JSON.parse(draft.hooksJson)` with no size cap.
- `HookCanvas.tsx`: mount effect does `JSON.parse(hooksCanvasJson)` and/or `generateCanvasFromHooks(hooksJson)` with no size cap.

**Exploit / vector**
- If `draft.hooksJson` or (more importantly) `draft.hooksCanvasJson` can come from an untrusted profile file/sync source, an attacker can provide a *very large* JSON string (tens/hundreds of MB). Even though you later cap node/edge counts and slice strings, the damage is done at `JSON.parse` time (allocation + blocking parse on UI thread).
- This can become a “profile bricking” issue: opening the profile repeatedly freezes the UI / crashes the Tauri webview.

**Impact**
- Persistent denial-of-service on opening/editing certain profiles; potential memory exhaustion.

**Fix**
- Enforce a hard maximum *string length* for `hooksJson` and `hooksCanvasJson` **before** any `JSON.parse` (both in UI state and wherever the draft/profile is loaded).
- Consider parsing in a worker for large payloads, or reject and require manual cleanup.
- Add a max total size in storage/back-end too (UI-only limits are bypassable if profiles are loaded from disk/network).

## HIGH — Regex ReDoS via `Condition.matcher` (catastrophic backtracking at runtime)
**Where**
- `NodeConfigPanel.tsx`: condition “Matcher (regex)” accepts arbitrary regex strings.
- `canvasCompiler.ts`: matcher is only length-capped (`MAX_MATCHER_LENGTH`), not safety-checked.

**Exploit / vector**
- A shared profile can include a pathological regex (e.g. `^(a+)+$`-style) that causes catastrophic backtracking when matched against tool names (or any longer string the runtime uses).
- Length ≤ 200 does not prevent ReDoS.

**Impact**
- Agent/tool execution pipeline can hang or spike CPU, potentially locking up the app/agent session.

**Fix**
- Use a safe regex engine (RE2) in the runtime, or validate with a “safe-regex” checker and reject/require confirmation for unsafe patterns.
- Consider offering a non-regex matcher mode (exact match / substring / glob) and discourage raw regex.

## MEDIUM — “Dangerous hook” confirmation is inconsistent / can be bypassed by source selection
**Where**
- `HookCanvas.tsx`: confirmation is only shown when loading **saved canvas JSON** (`hooksCanvasJson`) and only if `canvasHasDangerousActions(savedState)` is true.
- Fallback path (`generateCanvasFromHooks(hooksJson)`) loads command/http actions without a confirmation dialog.

**Exploit / vector**
- If attacker supplies an invalid/unsupported `hooksCanvasJson` (fails `isValidCanvasState`) but includes dangerous `hooksJson`, the canvas will still load dangerous command/http hooks via the fallback **without the “Load canvas anyway?” confirmation**.
- Even when the confirm appears and user clicks “Cancel”, the code “falls through” to generate from `hooksJson`, which may contain the same dangerous hooks—effectively nullifying the user’s intent.

**Impact**
- Safety UX can be bypassed; user may believe they refused loading dangerous content but still ends up loading it.

**Fix**
- Centralize dangerous-hook confirmation at the *entry to Canvas view* (or before any canvas state is derived), regardless of whether the source is `hooksCanvasJson` or `hooksJson`.
- If user declines, do not auto-fallback into another dangerous source.

## MEDIUM — CPU DoS via enormous `headers` objects (no entry-count cap)
**Where**
- `HookCanvas.tsx`: `sanitizeNodeData()` iterates `Object.entries(headers)` without a maximum count.
- `canvasGenerator.ts`: `sanitizeHeaders()` iterates all header entries without a maximum count.

**Exploit / vector**
- An attacker can provide a `headers` object with thousands of keys (within overall JSON size limits or via non-import paths) causing expensive iteration during load/sanitize and again on subsequent updates.

**Impact**
- UI stalls during load/sanitize; amplified if repeatedly triggered (autosave + rerenders).

**Fix**
- Add a header entry cap (e.g., 50–200). Short-circuit once reached.
- Also cap total serialized header size.

## MEDIUM — Data corruption / silent dropping of content on round-trip
**Where**
- `compileCanvasToHooks()` drops:
  - orphaned nodes (not reachable from a trigger),
  - actions with unknown `hookType`,
  - incomplete actions (empty command/url/prompt/description),
  - triggers that end up with no groups.
- `generateCanvasFromHooks()` may create trigger nodes even when groups/hooks are malformed, which later compile back to “nothing” (loss on save).

**Exploit / vector**
- A crafted `hooksCanvasJson` can include many “valid enough to render” nodes that will be silently dropped when the user makes any edit that triggers compilation (`structuralDirtyRef = true`).
- This is a profile integrity issue: users can lose hooks without clear attribution.

**Impact**
- Unintended loss of hooks/configuration; potential downgrade attack on profiles.

**Fix**
- Surface validation errors in UI (“these nodes will not compile”) and/or preserve unknown/invalid nodes in a non-executing “passthrough” section.
- Consider a strict mode: refuse to compile while invalid nodes exist.

## LOW — HTTP method validation is too permissive (non-CRLF control chars / spaces)
**Where**
- `canvasCompiler.ts` and `canvasGenerator.ts`: method is only CRLF-stripped and length-limited; not validated as an HTTP token or allowlist.

**Exploit / vector**
- If the runtime uses a permissive HTTP stack or constructs raw requests, weird methods (with spaces/control chars other than CRLF) can cause undefined behavior.

**Impact**
- Potential request malformation / edge-case injection in downstream tooling (depends on runtime implementation).

**Fix**
- Enforce an allowlist (`GET|POST|PUT|PATCH|DELETE`) or strict token regex (`^[A-Z]{1,10}$`).

---

## LOW — Unbounded node positions can create “unusable canvas” / render perf issues
**Where**
- `HookCanvas.tsx` `isValidCanvasState()` only checks `isFinite(pos.x/y)`; no magnitude clamp.

**Exploit / vector**
- A saved canvas can place nodes at extreme coordinates (e.g., `1e308`), potentially breaking minimap/fitView math or causing performance issues.

**Impact**
- UI degradation / “blank canvas” effect (soft DoS).

**Fix**
- Clamp positions to a reasonable range on load (and/or normalize layout when out of bounds).

---

## LOW — Inconsistent field length caps (state bloat)
**Where**
- `HookCanvas.tsx` `sanitizeNodeData()` caps `model` to `MAX_STRING_LENGTH` (10k) while compiler caps to 200.

**Impact**
- Larger-than-needed `hooksCanvasJson`, more memory usage; minor performance hit.

**Fix**
- Align per-field caps with compiler caps (e.g., `model` 200).
