### CRITICAL

1) **Debounce implementation is not stable across renders â†’ pending saves can be canceled on any re-render (data loss / stale JSON)**
- **Where:** `HookCanvas.tsx`  
  ```ts
  const notifyChange = useCallback(
    debounce(() => { ... }, 500),
    [onChange],
  );
  ...
  useEffect(() => () => { notifyChange.cancel(); saveLayout.cancel(); }, [notifyChange, saveLayout]);
  ```
- **What happens:**
  - `debounce(...)` is executed **during render**, so a *new* debounced function is created every render.
  - Because `notifyChange`/`saveLayout` identities change every render, the â€ścancel on unmountâ€ť effect runs its cleanup **on every render**, canceling timers that were scheduled by the previous instance.
  - If a structural edit schedules `notifyChange()` and then *any* re-render occurs within 500ms (selection change, ReactFlow internal updates, panel state, etc.), the pending compilation can be canceled and **never rescheduled** (especially since `pendingStructuralRef` remains `true`, blocking `saveLayout()`).
- **Impact:** hooks JSON/canvas JSON updates can be dropped, leaving the parent `draft.hooksJson` stale relative to the canvas edits. This undermines the core contract of the feature.
- **Fix:**
  - Create debounced functions with **stable identity** via `useMemo` or `useRef`, e.g.:
    ```ts
    const notifyChange = useMemo(() => debounce(() => { ... }, 500), [onChange]);
    const saveLayout = useMemo(() => debounce(() => { ... }, 2000), [onChange]);
    ```
  - The cleanup effect should depend on stable instances, so it only runs on unmount (or when `onChange` truly changes).

2) **Prototype-pollution / runtime exception risk via HTTP headers (`__proto__`)**
- **Where:** `canvasCompiler.ts` in `buildHookEntry()` and `canvasGenerator.ts` in `buildActionData()`
- **Problem A (compiler):**
  ```ts
  const sanitizedHeaders: Record<string, string> = {};
  for (const [k, v] of Object.entries(data.headers)) {
    sanitizedHeaders[k] = v; // k may be "__proto__"
  }
  ```
  Assigning `sanitizedHeaders["__proto__"] = "x"` can trigger the `__proto__` setter; in many engines this can throw (`TypeError: Object prototype may only be an Object or null`) or mutate prototype depending on valueâ€”either way itâ€™s unsafe.
- **Problem B (generator):**
  ```ts
  headers: hook.headers && typeof hook.headers === 'object' ... ? hook.headers : undefined,
  ```
  This passes imported JSON headers **through without sanitizing**. A malicious/odd hooks.json can inject `__proto__` into canvas state, later crashing compilation.
- **Impact:** crafted hooks JSON can crash compilation or create prototype pollution hazards.
- **Fix:**
  - Always construct header maps with `Object.create(null)` and explicitly skip dangerous keys:
    ```ts
    const sanitizedHeaders = Object.create(null) as Record<string,string>;
    if (DANGEROUS_KEYS.has(k)) continue;
    ```
  - Apply the same sanitization in the generator (`buildActionData`) before storing in node data.
  - Consider reusing a single shared sanitizer for headers across canvas load, generator, and compiler.

### MEDIUM

1) **Loaded canvas state validation does not enforce connection invariants**
- **Where:** `HookCanvas.tsx` `isValidCanvasState`
- **Issue:** It checks node existence and basic shapes, but does **not** validate:
  - allowed type-to-type edges (`triggerâ†’condition/action`, `conditionâ†’action`)
  - single-incoming constraint for `action` nodes
- **Impact:** a saved/edited `hooksCanvasJson` can contain illegal graphs, producing surprising compilation (duplicate/omitted hooks depending on traversal/ordering).
- **Fix:** validate edges against `connectionRules`, and enforce at-most-one incoming edge for action nodes during validation/sanitization.

2) **Unsupported hooks warning misses hooks with missing `type`**
- **Where:** `HooksTab.tsx` `hasUnsupportedHooks`
- **Issue:** It only flags `hook.type && !SUPPORTED_TYPES.has(hook.type)`. Hooks that omit `type` but contain other fields (e.g. `command`, `url`) are not marked unsupported, yet canvas editing will drop them (generator requires a supported `type`).
- **Impact:** silent loss of hooks when switching to canvas.
- **Fix:** treat hooks without `type` as unsupported if they contain hook-like fields, or generally if they are not representable by the generator.

3) **Import â€śdangerous hooksâ€ť detection is bypassable**
- **Where:** `HooksTab.tsx` `handleImport`
  ```ts

  stringified.includes('"type":"command"')
  ```
- **Issue:** String search is not a robust structural check (spacing, alternative representations, different casing, etc.).
- **Impact:** user may not be warned on importing dangerous hooks.
- **Fix:** reuse the structural traversal logic similar to `hasDangerousHooks`.

4) **NodePalette icons passed as HTML entities render literally**
- **Where:** `NodePalette.tsx` (`icon="&#9889;"` etc) and `DraggableNode` renders `{icon}`
- **Impact:** UI polish/clarity issue; icons wonâ€™t display as intended.
- **Fix:** pass real unicode characters (e.g. `"\u26A1"`, `"\u2699"`) instead of HTML entities in strings.
